分库分表实战：百万级表的水平拆分策略

发表于2025-12-07|更新于2026-07-19|后端工程
|浏览量:

前言

业务初期，单库单表简单高效。但随着业务增长，当订单表数据量突破千万、单表体积超过 20G 时，问题会集中爆发：查询越来越慢、索引体积膨胀、数据库连接数打满、单机存储逼近上限。加索引、优化 SQL 的收益越来越小，这时候分库分表就成了必然选择。

分库分表不是简单地把一张表拆成多张表，它涉及拆分策略选型、分片键设计、平滑扩容、分布式事务等一系列问题。选对了方案，系统容量和吞吐会明显提升；选错了分片键，后续会陷入跨库查询、数据倾斜的无尽泥潭。

本文就以最经典的订单表拆分为例，从拆分策略、分片键选型、ShardingSphere 落地到平滑扩容，完整讲透水平分库分表的实战全流程。

一、为什么需要分库分表？

单库单表的四道天花板：

瓶颈 说明
性能瓶颈 热点数据与大范围扫描增加，Buffer Pool 命中、IO 与写入竞争逐渐恶化
存储瓶颈 磁盘空间逼近单机上限，大表 DDL 无法在线执行
连接瓶颈 单机连接数有限，业务量大时连接打满
并发瓶颈 单机 CPU、IO 有上限，无法横向扩展

什么时候该考虑分库分表？

观察信号 判断方式
容量增长 单机存储、备份恢复或 DDL 窗口已无法满足目标
读写压力 核心 SQL 优化后，单机 CPU、IO、锁竞争仍接近瓶颈
扩展需求 单库吞吐无法继续垂直扩容，需要把负载分散到多节点
替代方案 索引、归档、缓存、读写分离等成本更低的方案已验证不足

行数和表大小没有通用阈值：500 万行的宽表可能很吃力，数亿行的窄表按主键点查也可能稳定。决策应来自容量模型、压测和运维目标。

分库分表是优化的「最后手段」，不是首选方案。能不分就不分。

二、拆分策略：垂直拆分 vs 水平拆分

2.1 垂直拆分：按业务/字段维度

垂直分库：按业务模块拆分

垂直分表：按字段冷热拆分

2.2 水平拆分：按数据行维度

水平分表：同库内拆成多张表

水平分库分表：既分库又分表

2.3 核心对比

维度 垂直拆分 水平拆分
拆分依据 业务/字段维度 数据行维度
表结构 每张表不同 所有子表相同
解决问题 业务解耦、宽表性能 单表数据量过大
实现复杂度 较低 较高

业界通用做法：先垂直拆库，再水平拆表。

三、分片键怎么选？

分片键（Sharding Key）决定数据路由到哪个库哪张表，是分库分表最核心的决策。

3.1 三大原则

原则 说明
查询高频 优先选业务查询最常用的过滤字段
分布均匀 避免数据倾斜，热点集中在某个分片
避免跨片 让大部分操作落在单个分片上

分片键没有绝对标准，要以核心查询路径为准。用户侧订单列表常选 user_id，订单详情如果主要按 order_no 查询，就要提前设计 order_no → user_id 的路由能力。

3.2 分片算法对比

算法 实现 优点 缺点 适用场景
取模法 key % N 简单、均匀 扩容需全量迁移 数据量稳定
范围法 按时间/ID区间 扩容简单 容易倾斜 日志、流水
哈希法 哈希后取模 更均匀 范围查询慢 用户ID、订单ID
一致性哈希 哈希环+虚拟节点 扩容迁移少 实现复杂 频繁扩容

3.3 非分片键查询怎么办？

订单表按 user_id 分片，但运营要按 order_no 查询，怎么办？

方案 说明
字段冗余 order_no 里嵌入 user_id，解析后路由
映射表 建 order_no → user_id 的映射表，先查映射再路由
宽表/ES 同步到 Elasticsearch，复杂查询走 ES
广播表 适合字典、配置等小表每个库都存一份，避免跨库关联；不适合订单这类大表查询

四、读写分离

分库分表前，可以先考虑读写分离：

组件 说明
主库 处理所有写操作
从库 处理读操作，主从复制同步数据
路由 应用层或中间件决定读写走哪个库

读写分离解决的是读并发问题，不解决单表数据量问题。如果单表已经很大，还是需要分库分表。

优点：简单可靠。缺点：需要停机。

建议 说明

七、常见坑点

问题 解决方案
分布式事务 Seata AT/TCC 模式，或消息队列最终一致性
跨库分页 游标式分页，或走 ES
跨库排序/聚合 全分片查询后内存合并，复杂统计走数仓
数据倾斜 选择分布均匀的分片键，热点用户单独处理
非分片键查询 字段冗余、映射表、ES

八、面试高频问题

Q1：什么时候该分库分表？

没有绝对行数阈值。通常在单表持续增长、索引和 SQL 优化收益明显下降、单机存储/IO/连接数接近瓶颈时再考虑。能不分就不分，分库分表是最后手段。

Q2：垂直拆分和水平拆分的区别？

垂直按业务/字段拆，解决业务解耦和宽表问题；水平按数据行拆，解决单表数据量过大问题。实际通常先垂直后水平。

Q3：分片键怎么选？

三个原则：查询高频（核心查询能单分片路由）、分布均匀（避免倾斜）、避免跨片。用户侧订单列表常选用户 ID，订单详情或运营查询更强的系统要配合 order_no 映射表、宽表或搜索引擎，不能盲目固定一种分片键。

Q4：非分片键查询怎么办？

字段冗余、映射表、同步到 ES；广播表只适合字典、配置等小表。复杂查询不要强行走分库分表。

Q5：分库分表后怎么做分页？

带分片键的分页可以直接路由；不带分片键的深分页需要全分片查询后内存合并，建议用游标式分页或走 ES。

Q6：如何扩容？

成倍扩容（停机迁移，简单可靠）或双写平滑迁移（不停机，复杂）。初期建议预留足够分片数。

写在最后

分库分表从来不是目的，而是解决数据量增长问题的手段。它在提升系统容量的同时，也带来了复杂度的大幅提升，每一个拆分决策都要结合业务场景。

从拆分策略选型、分片键设计，到中间件落地、平滑扩容，每一步都有明确的方法论。理解了这些底层逻辑，你就能根据自己的业务场景，设计出最适合的拆分方案。
