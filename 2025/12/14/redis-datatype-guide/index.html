Redis 数据结构与应用场景：远不止缓存

发表于2025-12-14|更新于2026-07-19|后端工程
|浏览量:

前言

提到 Redis，大多数人的第一反应是「缓存」：把热点数据放进去，减轻数据库压力。但如果你只把 Redis 当一个键值对缓存用，那就太浪费它的能力了。

Redis 真正的核心优势，在于它提供了一整套成熟的内置数据结构，从简单的字符串到有序集合，每一种都对应着一类典型的业务问题。不用自己手写链表、去重集合、排行榜，一行命令就能解决复杂的业务逻辑，这才是 Redis 不可替代的价值。

今天我们就拆解最常用的五大核心数据结构：String、Hash、List、Set、ZSet，讲透每一种的底层实现、核心特性与典型业务场景，看完就能直接对应到自己的项目里。

一、总览：两层结构设计

Redis 的数据结构分为两层：

  • 逻辑结构：我们日常开发直接使用的 API 层，也就是 String、Hash、List、Set、ZSet 五种核心类型
  • 底层编码：每种逻辑结构对应多种底层实现，Redis 会根据元素数量、数据大小自动切换，在空间占用和执行性能之间做最优平衡

理解了底层编码，才能真正看懂不同场景下的性能差异，避免「小数据量很快、数据上来就变慢」的认知偏差。

二、逐个拆解：底层实现与典型场景

2.1 String：最基础也最通用

String 是 Redis 最基础的数据结构，value 不仅支持普通字符串，还支持数字、二进制数据，单值最大 512MB。

底层实现：SDS 简单动态字符串

Redis 没有直接使用 C 语言原生字符串，而是自研了 SDS（Simple Dynamic String），相比原生字符串有三个核心优势：

  • O(1) 获取长度：结构体直接记录字符串长度，无需遍历计算
  • 二进制安全：不以 \0 作为结束标志，可以存储图片、序列化对象等任意二进制数据
  • 空间预分配：修改字符串时预分配冗余空间，减少频繁内存重分配

SDS 的内部结构：

Redis 3.2 之后，SDS 根据字符串长度自动选择 sdshdr8/16/32/64，短字符串只用 3 字节元数据开销，极致节省空间。

根据存储内容，String 会自动切换三种编码：

  • int 编码：值为长整型范围内的数字，直接用整数存储，极致省空间
  • embstr 编码：长度 ≤ 44 字节的短字符串，RedisObject 和 SDS 连续分配，一次内存分配搞定
  • raw 编码：长度超过 44 字节的长字符串，RedisObject 和 SDS 分开分配

为什么是 44 字节？ Redis 的 RedisObject 固定占 16 字节，SDS 最小头部（sdshdr8）占 3 字节，加上 \0 结尾共 19 字节。jemalloc 分配内存按 64 字节对齐，64 - 19 = 45，但实际可用 payload 为 44 字节。embstr 的意义在于只触发一次内存分配，访问效率更高。

典型业务场景

String 是万能结构，但最能发挥价值的是以下场景：

  1. 对象缓存

    把用户信息、商品详情等序列化成 JSON 字符串存储，是最经典的缓存用法。

  2. 原子计数器

    利用 INCR/DECR 原子命令，实现阅读量、点赞数、访问量等计数场景。

  3. 分布式锁

    配合 SET key value NX EX/PX 原子命令和唯一 value，实现轻量级分布式锁。不要把 SETNX 和 EXPIRE 拆成两条命令。

    value 必须是每次加锁生成的唯一 token，释放时再用 Lua 比对 token 后删除；不能直接 DEL，否则旧客户端可能误删新持有者的锁。

  4. 接口限流

    结合时间窗口自增，实现用户级、接口级限流。

    如果把 INCR 与 EXPIRE 拆开，进程在两条命令之间失败会留下永不过期的 key；每次请求都重设 TTL 又会改变固定窗口语义。高精度限流可使用 Lua 实现滑动窗口或令牌桶。

  5. 全局唯一 ID

    用自增命令生成趋势递增的全局 ID，替代数据库自增主键。

2.2 Hash：结构化对象存储首选

Hash 的 value 本身是一个键值对结构，相当于 Java 中的 Map<String, String>，天然适合存储结构化对象。

底层实现：listpack + 哈希表

  • listpack：Redis 7.0+ 中，字段数量和单值大小未超过 hash-max-listpack-entries、hash-max-listpack-value 时使用，连续内存存储，节省空间
  • 哈希表（hashtable）：超过上述任一阈值时自动转为哈希表，保证 O(1) 读写

Redis 6.2 及更早版本使用 ziplist；Redis 7.0+ 已由 listpack 取代，避免了 ziplist 的级联更新问题。默认阈值通常仍是 512 个条目、单值 64 字节，但可以通过配置调整。

核心优势

相比 String 序列化存对象，Hash 的价值在于：

  • 支持字段级读写，无需全量反序列化，更新单个字段性能更高
  • 小对象场景下内存占用远低于 String 序列化

典型业务场景

  1. 对象属性存储

    用户信息、商品参数等结构化数据，支持单个字段更新，灵活度更高。

  2. 购物车

    用户 ID 为 key，商品 ID 为 field，商品数量为 value，增删改查天然对应。

  3. 系统配置管理

    业务参数、功能开关用 Hash 存储，支持单个字段动态修改，无需整体加载。

2.3 List：有序可重复的双向列表

List 是按插入顺序排列的字符串列表，元素可重复，支持两端插入弹出，支持按索引访问，本质是有序序列。

底层实现：快速列表 quicklist

Redis 3.2 之后，List 底层统一为 quicklist，它把多个紧凑顺序块串成双向链表：

  • 列表拆分为多个节点，Redis 7.0+ 的每个节点内部使用 listpack；旧版本使用 ziplist
  • 节点之间用双向指针连接，保证两端插入弹出的高性能
  • 完美平衡了空间占用与插入性能

典型业务场景

  1. 简易消息队列

    LPUSH + RPOP 实现先进先出队列，适合简单的异步解耦场景。

    注意：原生 List 不支持 ACK 确认、消费者组、重复消费，复杂场景建议用 Stream 或专业 MQ。

  2. 最新内容列表

    最新公告、文章列表、动态 Feed，头部插入、按范围读取，天然有序。

  3. 用户时间线

    社交产品的关注动态，发布时写入粉丝时间线列表，刷新直接读取。

  4. 栈结构

    LPUSH + LPOP 实现先进后出的栈。

2.4 Set：无序去重集合

Set 是无序、不可重复的字符串集合，核心能力是去重 + 集合运算（交集、并集、差集）。

底层实现：整数集合 + listpack + 哈希表

  • 整数集合（intset）：所有元素都是整数且数量 ≤ 512（set-max-intset-entries）时，用连续整数数组存储，极致省空间
  • listpack：Redis 7.2+ 可将少量短字符串成员编码为 listpack，默认受 set-max-listpack-entries 和 set-max-listpack-value 控制
  • 哈希表（hashtable）：不满足整数集合条件时自动转换，判断元素是否存在为 O(1)

intset 内部按升序排列，支持 int16/int32/int64 自动升级：当插入一个 int64 的值时，整个集合自动升级为 int64 编码，保证兼容性。

典型业务场景

Set 的不可替代性在于集合运算，这是其他结构很难高效实现的。

  1. 标签系统

    用户标签、内容标签，天然去重，支持快速判断标签是否存在。

  2. 共同关注 / 共同好友

    用交集命令 SINTER 一键算出两个用户的共同集合。

  3. 去重统计

    页面独立访客 UV、活动参与用户数，存入 Set 天然去重。

  4. 抽奖系统

    • SRANDMEMBER：随机读取不删除，适合可重复抽奖
    • SPOP：随机取出并删除，适合抽完即止的场景

  5. 黑白名单

    IP 黑白名单、用户黑名单，存在性判断 O(1)，性能极高。

2.5 ZSet：有序不重复的加权集合

ZSet（Sorted Set）每个元素都附带一个分数（score），元素不重复、分数可重复，自动按分数从小到大排序，支持按分数、排名范围查询。

底层实现：listpack 或跳表 + 哈希表

元素较少且成员较短时，Redis 7.0+ 使用 listpack；超过阈值后会转换为跳表与哈希表的组合，两种结构各司其职：

  • 哈希表：存储元素与分数的映射，O(1) 查询元素对应的分数
  • 跳表（Skip List）：按分数有序存储所有元素，通过多层索引把查找复杂度降到 O(logn)，支持高效的范围查询、排名查询

跳表相比平衡树的优势：

维度 跳表 平衡树（红黑树）
实现复杂度 简单，只需维护多层链表 复杂，需要旋转、染色
范围查询 从起点沿底层链表遍历即可 需要中序遍历
更新范围 插入主要调整搜索路径上的指针 需要维护平衡并可能旋转
内存占用 指针数量可调 每个节点固定 2~3 个指针

为什么不用 B+ 树？ B+ 树更适合磁盘存储（减少 IO），而跳表是纯内存结构，实现更简洁、范围查询更直观。Redis 作为内存数据库，跳表是更自然的选择。

ZSet 的小数据默认阈值是元素数 ≤ 128、成员长度 ≤ 64 字节，对应 Redis 7.0+ 的 zset-max-listpack-entries 和 zset-max-listpack-value；Redis 6.2 及更早版本对应 ziplist 配置。Redis 命令主要串行执行，因此选择跳表的核心并不是“并发加锁更容易”，而是实现、范围遍历和更新成本之间的权衡。

典型业务场景

ZSet 是业务想象力最强的结构，核心价值是「有序 + 权重」。

  1. 各类排行榜

    游戏积分、商品销量、热搜榜、贡献榜，是 ZSet 最经典的场景。

  2. 延迟队列

    时间戳作为分数，任务内容作为元素，定时轮询取出到期任务执行。

  3. 优先级任务队列

    任务优先级作为分数，高优先级任务优先被消费。

  4. 热度加权排序

    综合点击、点赞、评论计算热度分存入 ZSet，自动按热度排序更新。

  5. 滑动窗口限流

    结合时间戳与分数，实现比固定窗口更精准的滑动窗口限流。

三、选型指南：一张表快速决策

业务场景 首选结构 核心理由
普通对象缓存、计数器 String 最简单通用，实现成本最低
结构化对象、需频繁更新单个字段 Hash 比 String 省空间，字段级操作更灵活
有序列表、最新消息、简易队列 List 天然有序，两端操作性能极高
去重、标签、共同好友、集合运算 Set 原生支持交并差集，去重成本为零
排行榜、延迟队列、优先级排序 ZSet 唯一支持按权重有序排列的结构

四、底层编码速查表

数据结构 小数据编码 大数据编码 切换阈值
String int / embstr raw 数字→int，≤44字节→embstr，>44→raw
Hash listpack（7.0+） hashtable 默认 entries>512 或 value>64 字节时转换
List quicklist（7.0+ 节点内为 listpack） quicklist 节点大小由 quicklist 配置控制
Set intset / listpack（7.2+） hashtable 整数集合与小字符串集合分别受对应阈值控制
ZSet listpack（7.0+） skiplist + hashtable 默认 entries>128 或 member>64 字节时转换

可以用 OBJECT ENCODING key 命令查看任意 key 的底层编码。

五、常见误区澄清

1. “存对象用 String 就够了，没必要用 Hash”

小对象、更新少的场景用 String 没问题。但如果经常需要更新单个字段，Hash 无需全量读写反序列化，性能更好，内存占用也更低。

实际对比：

2. “List 可以替代专业消息队列”

只能应对非常简单的解耦场景。它没有 ACK 确认、不支持消费者组、不保证消息可靠性，功能和稳定性远不如 RocketMQ、Kafka。如果需要 Redis 做消息队列，建议使用 Redis 5.0 引入的 Stream，它支持消费者组、消息确认（XACK）、持久化，是更完整的消息队列方案。

3. “数据量小的时候也一样快”

小数据量下 Redis 会使用 listpack、整数集合等省空间编码，部分操作的时间复杂度和大数据量下不同。比如 Hash 在 listpack 编码下查找字段是 O(n) 遍历，而不是 O(1) 哈希查找。设计方案时要考虑数据增长后的性能表现。

4. “Redis 只有这五种数据结构”

这五种是核心基础结构。除此之外 Redis 还有大量高级结构：

结构 用途
Bitmap 签到统计、用户在线状态、布隆过滤器底层
HyperLogLog 基数统计（UV 去重），12KB 内存可统计 2^64 个元素
Geo 地理位置、附近的人、距离计算
Stream 消息队列（支持消费者组）
Bitfield 位域操作，紧凑存储多个小整数

5. “ZSet 只能用于排行榜”

排行榜只是最直观的场景。延迟队列、滑动窗口限流、优先级队列、带权重的任务调度，都是 ZSet 的高效应用场景。只要是「按权重排序 + 快速查找」的需求，ZSet 都是首选。

六、面试高频问题

Q1：Redis 五种数据结构的底层编码分别是什么？

String 有 int/embstr/raw 三种；Redis 7.0+ 中 Hash、ZSet 的小数据编码是 listpack，大数据分别用 hashtable 和 skiplist+hashtable；List 使用 quicklist，节点内部是 listpack；Set 可用 intset，Redis 7.2+ 的小字符串集合还可用 listpack，超阈值后转为 hashtable。

Q2：为什么 ZSet 用跳表而不用红黑树？

跳表实现更简单，范围查询直接沿底层链表遍历，而红黑树需要中序遍历；跳表并发友好，局部修改不需要像红黑树那样做树旋转。

Q3：String 存对象和 Hash 存对象怎么选？

读多写少、整体更新 → String 序列化；需要频繁更新单个字段 → Hash。小 Hash 在 Redis 7.0+ 使用 listpack，Redis 6.2 及更早版本使用 ziplist；超过配置阈值后会转为 hashtable。

Q4：Redis 的 List 可以做消息队列吗？

简单场景可以（LPUSH + RPOP），但没有 ACK、消费者组、可靠性保证。复杂场景建议用 Redis Stream 或专业 MQ（Kafka、RocketMQ）。

Q5：Set 的交集运算时间复杂度是多少？

SINTER 的最坏时间复杂度是 O(N*M)，N 是最小集合的元素数，M 是集合个数。SINTERSTORE 只减少结果传输并保存结果，计算本身仍会占用 Redis 执行线程，不能用来“避免阻塞”。大集合应限制基数、拆分或预计算；若只要交集数量，Redis 7.0+ 可用 SINTERCARD ... LIMIT 提前停止。

Q6：什么是 ziplist 的级联更新？

ziplist 中修改一个节点的长度，可能导致后续节点连续改变前置长度字段。Redis 7.0 用 listpack 替代 Hash、List 节点和 ZSet 中的 ziplist，通过不依赖前一节点长度的布局消除了这类级联更新。

写在最后

Redis 之所以能成为后端开发的标配，绝不只是因为它能做缓存。它把业务开发中最常见的几类数据结构，都做成了高性能、开箱即用的实现，帮我们省去了大量重复造轮子的工作。

理解每种结构的底层实现，不是为了应付面试，而是让你在做技术选型时，清楚地知道什么场景该用什么、为什么这么选、性能边界在哪里。选对了数据结构，往往能让业务实现事半功倍。
