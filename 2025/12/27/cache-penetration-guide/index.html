缓存穿透/击穿/雪崩：一次缓存故障的排查与解决

发表于2025-12-27|更新于2026-07-19|后端工程
|浏览量:

前言

大促预热活动上线当晚，运维告警群突然炸锅：商品详情接口大面积超时，数据库 CPU 飙升至 98%，连接池被打满，大量请求直接报错。第一反应是流量突增，但拉监控一看，Redis 的 QPS 并不高，缓存命中率跌到了 30% 以下——很明显，缓存层出问题了。

很多人做缓存设计时，只想着「把数据放进 Redis 就能提速」，却忽略了缓存失效带来的反向冲击。缓存穿透、击穿、雪崩是三种最经典的缓存故障，很多人经常把三者混为一谈，但它们的触发原因、影响范围、解决方案完全不同。

本文就以这次真实的线上故障为例，先理清三者的本质区别，再从紧急止血到根治优化，完整讲解布隆过滤器、互斥锁、热点 key 治理等核心方案，最后给出一套可复用的缓存稳定性设计原则。

一、先分清：三者根本不是一回事

很多人面试时背得滚瓜烂熟，真遇到故障却分不清是哪一种。我们先用一张表把核心差异说清楚：

问题类型 核心原因 典型特征 影响范围
缓存穿透 请求的 key 在缓存和数据库中都不存在 缓存命中率极低，大量请求直接打数据库 全量无效请求都打库
缓存击穿 单个热点 key 突然过期失效，大量并发同时打向数据库 特定热点时间点出现流量尖刺，单 key 对应数据库 QPS 突增 单个热点数据对应的库表
缓存雪崩 大量 key 同时过期，或 Redis 整体宕机 缓存命中率断崖式下跌，数据库整体压力瞬间拉满 全库全表，极易引发数据库雪崩

简单总结：

  • 穿透是数据根本不存在，缓存挡不住
  • 击穿是单个热点 key 过期，单点突破
  • 雪崩是大面积缓存失效，整体崩盘

我们这次故障比较特殊：三者叠加出现——恶意爬虫刷不存在的商品 ID 引发穿透，热门商品缓存集中过期引发击穿，再加上批量预热时统一设置了 2 小时过期时间，大量 key 同时失效，最终叠加导致数据库直接被打满。

二、排查过程：从现象定位根因

故障排查不能靠猜，我们按「从宏观到微观」的顺序逐步定位。

第一步：看宏观指标，确认缓存失效

先看监控大盘，得到几个关键信息：

  • 接口平均响应从 80ms 涨到 2100ms
  • MySQL CPU 使用率 98%，主要是查询语句
  • Redis 命中率仅 28%，远低于平时的 95%+
  • Redis 本身内存、CPU 正常，没有宕机

结论：Redis 没挂，但大量请求没命中缓存，直接打到了数据库。

第二步：分析请求特征，定位穿透

捞取 Nginx 日志，发现大量请求的商品 ID 是不规则的负数、超长数字，明显是非法参数。这些 ID 在数据库里根本不存在，自然缓存里也没有，每次请求都会穿透到数据库查一遍，查不到又不缓存，相当于缓存完全失效。

这就是典型的缓存穿透。

第三步：看热点分布，定位击穿

按 SQL 执行次数排序，Top 3 的热门商品查询语句执行量突然暴涨。去 Redis 查这几个 key 的过期时间，发现它们几乎在同一时间点过期——刚好是活动上线 2 小时后。

热门商品过期瞬间，几千个请求同时发现缓存失效，一起去查数据库，直接把对应表的连接占满。

这就是典型的缓存击穿。

第四步：看过期策略，定位雪崩

进一步排查发现，运营后台批量预热商品缓存时，统一设置了 expire = 7200秒，所有预热的商品 key 都会在同一时间点过期。到了过期时间点，十几万条商品缓存同时失效，大量普通商品的请求也直接打向数据库，进一步放大了数据库压力。

这就是缓存雪崩的诱因。

三、逐个击破：从紧急止血到根治方案

故障处理的原则永远是：先止损，再根治。我们先做紧急处理把数据库压力降下来，再逐个优化彻底解决问题。

3.1 缓存穿透：拦住不存在的请求

问题本质

用户请求的数据，在缓存和数据库里都不存在。每次请求都会穿过缓存层直达数据库，相当于缓存形同虚设。

常见触发场景：恶意攻击、参数非法、数据被物理删除。

方案一：紧急止血——缓存空值

最快的临时方案：数据库查不到的数据，也往 Redis 里存一个空值（比如空字符串、null 标记），设置较短的过期时间（比如 30 秒）。这样后续同样的非法请求会直接命中缓存的空值，不会再打数据库。

优点：实现简单，见效快。

缺点：会占用额外内存，空值过期前如果数据新增，会出现短暂不一致。如果攻击者用大量不同的不存在 key 轮询，空值缓存会被打满，内存压力增大。

方案二：根治方案——布隆过滤器

要从根源上拦截不存在的 key，最通用的方案就是布隆过滤器（Bloom Filter）。

布隆过滤器原理

它是一种空间效率极高的概率型数据结构，核心是一个位数组 + 多个哈希函数：

  1. 写入时：将 key 经过 k 个哈希函数计算，映射到位数组的多个位置，把对应位置设为 1
  2. 查询时：同样计算 k 个位置，如果所有位置都是 1，说明 key 可能存在；如果有任意一个位置是 0，说明 key 一定不存在

它的特点是：

  • 不存在的一定判断为不存在（100% 准确）→ 刚好适配穿透场景
  • 存在的可能有误判（False Positive）→ 误判率可通过参数控制
关键参数计算

布隆过滤器的性能源于精确的参数设计：

参数 含义 计算公式
m 位数组大小（bit） m = -n×ln(p) / (ln2)²
k 哈希函数个数 k = (m/n) × ln2
n 预期元素数量 业务预估
p 期望误判率 通常 0.01%~1%

例如：100 万个商品 ID，误判率 0.01%，需要约 19.2Mbit（约 2.4MB）的位数组，约 13 个哈希函数。相比把 100 万个 ID 全部存到 Set 中（几十 MB），空间节省了一个数量级。

实战实现

生产环境常用两种方案：

  • Guava BloomFilter：本地内存实现，适合单应用场景，性能极高
  • RedisBloom 模块：通过 Redis 模块实现，分布式场景下共享，推荐使用

  • 原生布隆过滤器不支持删除元素，需要删除场景可用 Counting Bloom Filter（用计数器代替 bit）
  • 数据新增时要同步写入布隆过滤器，保证一致性
  • 重启或持久化时需要确认过滤器数据的持久化策略

方案三：前置拦截

在网关层、接口层做参数合法性校验，比如 ID 范围校验、格式校验、频率限流，先把明显非法的请求拦在最外层，减少无效流量进入系统。

这是最容易被忽视但效果最直接的方案——能进缓存和数据库的请求，都应该先过一遍基本的合法性检查。

3.2 缓存击穿：稳住热点 key 的过期瞬间

问题本质

某个访问量极高的热点 key，在缓存过期的一瞬间，成千上万的并发请求同时发现缓存失效，一起去查数据库，瞬间把数据库打垮。核心特征是「单点热点、时间点集中」。

方案一：互斥锁方案（Mutex Lock）

最常用的根治方案，核心思路是：缓存失效时，只允许一个线程去查数据库重建缓存，其他线程等待重试，避免大量请求同时打库。

双重检查（Double Check）：拿到锁之后再次查缓存，防止在等待锁期间其他线程已经重建了缓存，避免重复查库。

注意点：

  • 锁必须设置超时时间，避免业务异常导致死锁
  • 重试次数和总等待时间都要有限制，避免线程堆积
  • 锁粒度要细，每个 key 对应一把锁
  • 释放锁时要确认是自己加的锁（用 requestId 做归属校验），避免误删

方案二：热点 key 逻辑过期

对于核心热点数据，可以设置为「物理上永不过期」，在 value 中嵌入一个逻辑过期时间字段。读取时判断是否逻辑过期：

  • 未过期 → 直接返回
  • 已过期 → 返回旧数据，同时异步触发缓存更新

优点：正常情况下不会在逻辑过期瞬间让请求同步打库，读延迟稳定。

缺点：有短暂的数据不一致窗口，返回的可能是旧数据。适合对一致性要求不极致、但对稳定性要求极高的核心热点数据。

方案三：热点数据主动刷新

访问缓存时，判断剩余过期时间，如果低于阈值，就触发后台异步线程提前刷新缓存，相当于把过期时间往后顺延，永远不让它真正过期。

这与 Redisson 锁的看门狗都属于“到期前续期”的思路，但两者语义不同：缓存刷新应依据访问热度、刷新失败重试和数据版本控制设计，不能直接把锁续期机制照搬成缓存方案。

3.3 缓存雪崩：避免大面积失效的灾难

问题本质

两种场景会引发雪崩：

  1. 集中过期型：大量 key 在同一时间点过期，导致大面积缓存失效
  2. 故障型：Redis 节点宕机、集群故障，整个缓存层不可用

方案二：多级缓存架构

引入本地缓存作为一级缓存，Redis 作为二级缓存：

  • 热点数据存到本地 Caffeine 缓存，极短时间内的请求直接走本地，不用访问 Redis
  • 本地缓存设置更短的过期时间（如 1~5 分钟），作为兜底
  • 即使 Redis 整体故障，本地缓存也能扛住一部分流量，不至于全部打到数据库

Caffeine 是 Java 生态常用的高性能本地缓存库，采用 W-TinyLFU 等策略改善命中率。Spring Boot 的缓存抽象可以在引入 Caffeine 依赖后自动配置它，但并不是所有项目都默认携带该实现。

Go 语言方案：Go 生态中可以使用 singleflight 模式——同一个 key 的并发请求合并为一次调用，其他请求等待结果共享，效果类似互斥锁但更轻量。

方案三：Redis 高可用架构

针对 Redis 故障引发的雪崩，核心是避免单点故障：

  • 生产环境必须做主从 + 哨兵架构，或者 Redis Cluster 集群
  • 多可用区部署，单节点故障自动故障转移
  • Cluster 模式下数据按槽分片并可配置副本；主节点故障时需要副本完成故障转移。若某些槽没有可用副本，对应 key 仍会不可用，不能只靠“分片”保证高可用

方案四：降级熔断兜底

在服务层接入限流熔断组件（如 Sentinel）：

  • 当数据库 QPS 达到阈值时，自动触发降级，非核心请求直接返回兜底数据
  • 保护核心业务的可用性，避免数据库被打死导致全站不可用
  • 降级策略可以是返回默认值、返回缓存中的旧数据、或直接提示稍后重试

方案五：缓存预热

大促、活动上线前，提前把热点数据批量加载到缓存中，避免冷启动时大量请求同时穿透到数据库。

预热时同样要注意错开过期时间，不要统一设置。可以用延迟队列或分批脚本来分散预热时间。

四、优化效果：数据验证

全部优化上线后，运行一周的监控数据对比非常明显：

指标 故障时 优化后
缓存命中率 28% 98.5%
数据库 CPU 使用率 98% 22%
接口平均响应时间 2100ms 92ms
非法穿透请求 全部打库 布隆过滤器拦截 99.9%
热点过期尖峰 明显突刺 完全抹平

数据库压力下降了 70% 以上，接口稳定性大幅提升，后续几次流量高峰都平稳度过。

五、避坑总结：缓存设计的最佳实践

回顾整个故障和优化过程，很多问题其实在设计阶段就可以避免。整理几条通用的缓存设计原则：

  1. 防御前置
    不要等请求到了缓存层才处理，网关校验、参数校验、布隆过滤器，能前置的拦截尽量前置。

  2. 避免绝对集中
    过期时间一定要加随机值，不要统一设置；批量预热、批量更新都要错开时间，避免尖峰。

  3. 热点特殊处理
    核心热点数据不要和普通数据用同一套过期策略。该永不过期就永不过期，该互斥锁就互斥锁，特殊场景特殊对待。

  4. 多级兜底架构
    不要把所有希望都寄托在 Redis 上。本地缓存、降级兜底、限流熔断，多层防护才能保证极端情况下系统不垮。

  5. 监控不可少
    重点监控缓存命中率、热点 key 排行、Redis 节点状态、数据库 QPS。命中率突降、热点突增都是故障前兆，提前发现就能提前处理。

六、面试高频问题

Q1：缓存穿透、击穿、雪崩的区别？

穿透：数据在缓存和数据库都不存在，请求穿透到库。击穿：单个热点 key 过期，并发请求同时打库。雪崩：大量 key 同时过期或 Redis 宕机，缓存层整体失效。

Q2：布隆过滤器的原理？误判率怎么控制？

位数组 + 多个哈希函数。写入时把对应位设为 1，查询时所有位为 1 则可能存在，有 0 则一定不存在。误判率由位数组大小 m 和哈希函数个数 k 决定，m 越大、k 越合理，误判率越低。

Q3：布隆过滤器不支持删除怎么办？

用 Counting Bloom Filter，用计数器代替 bit 位，删除时把对应计数器减 1。或者使用 Redis 的 Cuckoo Filter（布谷鸟过滤器），支持删除且空间效率接近布隆过滤器。

Q4：互斥锁方案为什么要双重检查？

获取锁的过程中可能有其他线程已经重建了缓存。拿到锁后再次查缓存，避免重复查库，减少数据库压力。

Q5：逻辑过期和物理过期怎么选？

对一致性要求高（如金融、库存）→ 物理过期 + 互斥锁。对可用性要求高、允许短暂不一致（如商品详情、推荐列表）→ 逻辑过期 + 异步更新。

Q6：多级缓存的一致性怎么保证？

本地缓存设置短过期时间可以容忍有限的不一致窗口。Redis Pub/Sub 能通知各节点清除缓存，但消息不持久化、断线期间可能漏收，因此只能改善最终一致性，不能保证强一致。要求更高时可使用带持久化和重试的变更事件、版本校验，或避免在该数据上叠加本地缓存。

写在最后

缓存不是「把数据丢进 Redis」就万事大吉了。它在提升性能的同时，也引入了新的故障模式。穿透、击穿、雪崩，本质都是缓存失效带来的反向冲击。

理解它们的底层差异，不是为了应付面试，而是线上出问题时，你能快速判断是哪一类，知道该先做什么、再做什么，有章法地排查解决，而不是手忙脚乱瞎试。

缓存设计的精髓从来不是堆机器、加容量，而是提前预判风险，做好每一层兜底。
