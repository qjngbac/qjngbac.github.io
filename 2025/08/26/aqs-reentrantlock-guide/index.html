AQS 源码分析：ReentrantLock 背后的秘密

发表于2025-08-26|更新于2026-07-19|Java 深度
|浏览量:

前言

写 Java 并发代码，ReentrantLock 是常用的显式锁：它可配置公平策略，并支持可重入、可中断和超时获取，在需要这些能力时比 synchronized 更灵活。但很多人用了很久，却不知道它的底层建立在 AQS 之上。

面试中 AQS 更是并发模块的必考题：state 变量有什么用？CLH 队列怎么工作？公平锁和非公平锁到底差在哪？这些问题看似是八股，实则是理解整个 JUC 并发工具包的钥匙。

ReentrantLock 本身几乎没有复杂逻辑，它的全部锁能力，都委托给了内部的 AQS 实现类。今天我们就从 ReentrantLock 切入，逐层拆解 AQS 的核心设计：state 同步状态、CLH 等待队列、公平与非公平实现，彻底搞懂显式锁的底层原理。

一、AQS 是什么？JUC 并发的基石

AQS 全称 AbstractQueuedSynchronizer，抽象队列同步器，是 java.util.concurrent 包的核心骨架。

它本质上是一个抽象框架：用一个共享变量维护同步状态，再用一个近似 FIFO 的等待队列管理竞争失败的线程。基于这套框架，JDK 实现了一整套并发工具：

并发工具 AQS 模式 state 含义
ReentrantLock 独占 重入次数
ReentrantReadWriteLock 独占+共享 高16位读锁计数，低16位写锁计数
Semaphore 共享 剩余许可数
CountDownLatch 共享 倒计数数值
CyclicBarrier 间接基于 ReentrantLock/Condition 分代屏障状态

AQS 支持两种工作模式：

  • 独占模式：同一时刻只能有一个线程持有锁，代表是 ReentrantLock
  • 共享模式：同一时刻可以有多个线程持有许可，代表是 Semaphore、CountDownLatch

本文聚焦 ReentrantLock 对应的独占模式，这也是 AQS 最基础、最常考的部分。

二、核心变量 state：锁状态的本质

整个 AQS 的核心，是一个被 volatile 修饰的整型变量：

别看它只是一个 int，它承载了同步器的状态。竞争空闲锁等并发更新通常使用 CAS；而持锁线程重入或释放时，AQS 子类也可能在所有权约束下用普通的 setState 更新，因此不能概括成每次变化都由 CAS 完成。

2.1 state 在 ReentrantLock 中的含义

在 ReentrantLock 的独占模式下：

  • state == 0：当前没有线程持有锁，锁处于空闲状态
  • state > 0：已经有线程持有锁，数值代表该线程的重入次数

比如线程 A 第一次拿到锁，state 从 0 变为 1；同一个线程再次加锁（重入），state 累加为 2；释放锁时依次递减，减到 0 时锁才真正释放，其他线程可以争抢。

2.2 为什么用 volatile + CAS

  • volatile 为 state 的读写提供可见性与有序性保证：后续读取不会长期使用普通缓存变量那样的旧值，但仍应按 Java 内存模型的 happens-before 语义理解，而不是把“立刻”当作物理时间承诺
  • 提供 compareAndSetState() 方法，供子类在存在竞争的状态转换中执行原子比较并更新

在 ReentrantLock 获取空闲锁的路径中，CAS 是决定竞争结果的关键步骤；重入、释放、排队与线程所有者检查则共同构成完整语义。

三、CLH 等待队列：抢不到锁的线程去哪了？

如果抢锁失败，线程总不能一直死循环自旋浪费 CPU，也不能直接丢弃。AQS 的解决方案就是 CLH 等待队列：所有抢锁失败的线程，会被封装成节点加入队列，挂起等待，直到被前面的线程唤醒。

3.1 CLH 队列是什么

CLH 队列是一个虚拟的双向链表（FIFO），它没有完整的链表对象，只通过节点的 prev 和 next 指针串联。队列有两个关键指针：

  • head：等待队列的占位头节点，表示队列推进位置，不代表当前锁拥有者；拥有者由同步器单独记录
  • tail：尾节点，新的等待线程会追加到尾部

在 JDK 8 实现中，waitStatus 记录节点状态，常见值如下：

状态 值 含义
SIGNAL -1 后继节点处于等待状态，当前节点释放时需要唤醒后继

3.3 入队过程：线程怎么排队

当线程抢锁失败，就会调用 addWaiter() 方法入队：

  1. 将当前线程封装成一个 Node 节点
  2. 尝试用 CAS 将尾节点指向新节点，成功则完成入队
  3. 如果 CAS 失败（并发激烈），进入 enq() 自旋循环，直到成功插入队尾

为什么入队要用 CAS？因为多个线程同时抢锁失败，会同时往队尾插入节点。尾节点是共享资源，必须用 CAS 保证同一时刻只有一个线程能成功修改尾指针。

3.4 挂起与唤醒

节点入队后会在获取循环中检查前驱：只有前驱是头节点时才有资格再次调用 tryAcquire。获取失败后，JDK 8 代码会先把前驱状态调整为可通知状态，再在后续循环中调用 LockSupport.park()；它不是一个固定“自旋几次”的算法。

当持有锁的线程释放锁时，会调用 LockSupport.unpark() 唤醒头节点的后继节点；后继线程被唤醒后，会再次尝试抢锁，成功则将自己设为新的头节点。

3.5 为什么要用双向队列

很多人会问：排队用单向链表不行吗？双向的价值在于：

  1. 支持节点取消：线程超时、被中断时，可以快速找到前驱和后继，把自己从队列中摘除
  2. 安全唤醒：唤醒后继前可以通过前驱指针回溯，确认前驱状态有效，避免空唤醒
  3. 容错性强：入队 CAS 失败时，可以通过前驱指针回退重试，不会出现断链

四、公平锁 vs 非公平锁：差的真不只是一行代码

ReentrantLock 有两种模式：默认非公平锁，也可以通过构造函数指定公平锁。它们的差异，本质是 AQS 两个子类 NonfairSync 和 FairSync 的实现差异。

4.3 两种模式怎么选

维度 非公平锁（默认） 公平锁
执行顺序 不保证，可能插队 竞争时倾向最长等待者，不保证严格线程调度顺序
吞吐量 更高，减少线程切换开销 更低，排队唤醒开销大
线程饥饿 可能出现，极端情况下队列尾部线程一直抢不到 基本避免饥饿，但仍受系统调度影响
适用场景 大多数强调吞吐的场景 希望降低长期饥饿概率的场景

为什么 JDK 默认用非公平锁？因为在大多数场景下，吞吐量比绝对公平更重要。线程挂起和唤醒的开销非常大，非公平锁的”插队”能大幅减少上下文切换，整体性能更高。

还有一个常被忽略的边界：即使创建的是公平 ReentrantLock，无参 tryLock() 只要锁当时可用仍可能插队，不遵守公平设置；需要等待且遵循公平策略时，应使用 lock() 或带超时的 tryLock(timeout, unit)。

五、完整流程：一次抢锁与释放的全链路

讲完了核心组件，我们把整个流程串起来，看一个线程调用 lock() 之后，到底经历了什么。

5.1 独占锁获取流程（acquire）

acquire(1) 是独占锁获取的入口，核心逻辑只有四行：

执行步骤拆解：

  1. 尝试抢锁：调用 tryAcquire()，成功则直接返回，方法结束
  2. 封装入队：抢锁失败，调用 addWaiter() 把当前线程封装成节点，加入 CLH 队尾
  3. 队列自旋：进入 acquireQueued() 循环：
    • 如果前驱是头节点，再尝试抢一次锁，成功就把自己设为新头节点，退出循环
    • 不是头节点或抢锁失败，就把前驱状态设为 SIGNAL，然后 park 挂起自己
  4. 中断处理：如果被中断唤醒，记录中断标记，拿到锁后自行补中断

5.2 独占锁释放流程（release）

释放锁的入口是 release(1)：

执行步骤拆解：

  1. 尝试释放：调用 tryRelease()，state 减对应数值
  2. 完全释放判断：如果 state 减到 0，说明锁完全释放，清空持有锁的线程标记
  3. 唤醒后继：找到头节点的下一个有效节点，调用 LockSupport.unpark() 唤醒它
  4. 被唤醒的线程回到 acquireQueued 循环，继续抢锁

5.3 重入的实现细节

重入的逻辑非常简单：

  • 抢锁时，如果发现当前线程就是持有锁的线程，不报错，直接把 state 加 1，返回成功
  • 释放锁时，state 减 1，只有减到 0 时才返回 true，才会去唤醒后继线程

这也解释了为什么 ReentrantLock 加锁几次，就必须解锁几次，多解锁会抛 IllegalMonitorStateException，少解锁会导致锁无法释放。

六、条件变量 Condition：比 wait/notify 更灵活

synchronized 配合 wait()/notify() 可以实现线程间通信，但只能有一个等待队列。ReentrantLock 配合 Condition 可以创建多个条件队列，更灵活。

6.2 Condition 的本质

每个 Condition 对象内部维护一个条件队列（单向链表），与 AQS 的 CLH 等待队列是两套不同的队列：

  • await()：当前线程必须先持有锁，调用后会加入条件队列、释放锁并挂起
  • signal()：把条件队列中的节点转移到 AQS 的 CLH 队列，等待重新竞争锁

七、共享模式简述：Semaphore 和 CountDownLatch

虽然本文聚焦独占模式，但了解共享模式有助于理解 AQS 的全貌。

7.1 Semaphore（信号量）

AQS 实现：state 表示剩余许可数，acquire 尝试用 CAS 扣减所需许可；扣减结果大于等于 0 就获取成功，只有许可不足、结果为负时才进入等待。release 用 CAS 增加许可并传播唤醒共享模式等待者。

7.2 CountDownLatch（倒计时门闩）

AQS 实现：state 表示倒计数数值，countDown 时 CAS 减 1；await 时如果 state 不为 0 则入队等待，减到 0 时唤醒所有等待线程。

八、ReentrantLock vs synchronized

面试常问：有了 synchronized，为什么还需要 ReentrantLock？

对比项 synchronized ReentrantLock
实现层面 JVM 关键字，字节码层面 Java 层面，基于 AQS
锁获取方式 自动获取释放 手动 lock/unlock
可中断 不支持 支持 lockInterruptibly
超时获取 不支持 支持 tryLock(timeout)
公平锁 不支持，只有非公平 支持公平/非公平切换
条件变量 只有 wait/notify 支持多个 Condition
实现优化 JVM 内部实现随 JDK 演进，不能固定概括为“偏向→轻量→重量” AQS 队列与 park/unpark，内部实现同样会随 JDK 演进
性能 JDK 6 后优化很大，差距不大 视竞争场景而定，优势主要在高级能力
释放方式 自动释放（出 synchronized 块） 必须手动 unlock（finally 中）

怎么选？

  • 简单的同步场景 → 用 synchronized，简洁不易出错
  • 需要高级功能（公平锁、可中断、超时、多条件） → 用 ReentrantLock

九、常见误区澄清

  1. “AQS 就是 CLH 队列”

    不准确。AQS 是一整套同步框架，state 状态变量 + CLH 等待队列 + 独占/共享模式 + 条件队列，合在一起才是完整的 AQS。

  2. “公平锁一定比非公平锁慢很多”

    低并发、锁持有时间短的场景下差异很小；只有在高竞争、锁持有时间长的场景下，非公平锁的吞吐量优势才会明显。

  3. “CLH 队列就是原始的 CLH 自旋锁”

    AQS 使用的是变种 CLH 队列。原始 CLH 是单向链表、自旋等待；AQS 改成了双向链表，且线程是挂起等待而非自旋，更适合长时间等待的场景。

  4. “头节点就是持有锁的线程”

    严格来说不是。AQS 的头节点是一个”占位节点”，线程拿到锁后会把自己的节点设为新的 head，并清空 thread 字段，作为占位标记。

  5. “ReentrantLock 的 tryLock 会加入队列”

    无参 tryLock() 不会。它是非阻塞的，获取不到锁直接返回 false；但带超时的 tryLock(timeout, unit) 在等待期间会进入 AQS 获取流程，并支持中断。

十、面试高频问题汇总

Q1：AQS 的 state 变量有什么用？

state 是同步状态变量，volatile 修饰保证可见性。在 ReentrantLock 中，0 表示锁空闲，大于 0 表示锁被持有且数值为重入次数。

Q2：线程抢锁失败后去了哪里？

被封装成 Node 节点加入 CLH 双向链表的队尾，然后调用 LockSupport.park() 挂起等待。

Q3：公平锁和非公平锁的核心区别？

公平锁在抢锁前会调用 hasQueuedPredecessors() 检查队列中是否有等待线程，有人排队就不抢；非公平锁不管队列，直接 CAS 抢。

Q4：ReentrantLock 的可重入是怎么实现的？

抢锁时判断当前线程是否就是持有锁的线程，是的话 state 加 1；释放时 state 减 1，减到 0 才真正释放锁。

Q5：AQS 为什么用 CLH 队列而不是自旋？

自旋会持续消耗 CPU，适合锁持有时间极短的场景。AQS 用 park/unpark 挂起唤醒，避免 CPU 空转，适合锁持有时间不确定的场景。

写在最后

AQS 之所以能成为 JUC 的基石，核心就在于它的设计足够通用：用一个 state 变量抽象同步状态，用一个 CLH 队列统一管理等待线程，把”抢锁/释放”的具体逻辑留给子类实现。

ReentrantLock 只是它最经典的一个实现。当你理解了独占模式下的 state、CLH 队列、公平非公平差异，再去看 Semaphore、CountDownLatch 的共享模式，就会发现核心逻辑高度一致，只是 state 的语义和抢锁策略不同。

并发编程的进阶，从来不是背更多的 API，而是看懂工具背后的统一设计思想。看懂了 AQS，你才算真正推开了 Java 并发的大门。
