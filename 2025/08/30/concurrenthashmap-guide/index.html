ConcurrentHashMap 源码：它为什么是线程安全的

发表于2025-08-30|更新于2026-07-19|Java 深度

前言

在并发场景下，HashMap 会出现元素覆盖、丢失、计数不准等问题；JDK 7 的并发扩容还可能形成链表环。Hashtable 虽然线程安全，却主要依赖整表级 synchronized，并发写入容易形成明显竞争。

为了兼顾「线程安全」与「并发性能」，JDK 推出了 ConcurrentHashMap。它也是面试中的高频考点：分段锁是什么？JDK 1.8 为什么改用 CAS + synchronized？并发下 size() 怎么统计才准确？这些问题的背后，是一整套精细的并发控制思想。

今天我们就从 JDK 1.7 到 1.8 的演进入手，拆解 ConcurrentHashMap 的线程安全实现，把分段锁、CAS 桶锁、分散计数、并发扩容这些核心机制一次性讲透。

一、演进之路：从粗粒度锁到细粒度锁

ConcurrentHashMap 的核心设计目标很明确：在保证线程安全的前提下，尽可能降低锁的粒度，减少锁竞争，提升并发吞吐量。

版本 核心实现 锁粒度 并发度 底层结构
JDK 1.7 分段锁（Segment） 锁整个分段 默认 16 个分段 分段数组 + 链表
JDK 1.8 CAS + synchronized 非空桶锁住首节点 写竞争大致分散到不同桶 数组 + 链表 + 红黑树

四、get 操作：全程无锁

为什么 get 不需要加锁？

因为 Node 的 val 和 next 指针都用 volatile 修饰，table 引用本身是 volatile，桶位读写又通过 volatile 语义的 tabAt / CAS 完成，保证了可见性。读操作全程无锁，性能很高。

五、并发扩容：多线程一起搬数据

JDK 1.8 的一大亮点：扩容不再由单线程完成，所有插入线程都可以参与。

核心思想是让遇到 ForwardingNode 的更新线程协助迁移，缩短单个触发线程独自搬完整张表的停顿。参与线程数仍受 CPU、数组大小和并发写入量限制，并不是数据越多就一定线性加速。

八、常见误区澄清

1. “JDK 1.8 还有分段锁吗？”

没有了。Segment 类依然存在，但只是为了兼容旧版本序列化，没有实际锁功能。

2. “所有操作都是线程安全的吗？”

单个操作（put、get、remove）是线程安全的，但复合操作不保证原子性：

3. “为什么 key 和 value 不能为 null？”

并发场景下的二义性问题：get 返回 null，无法区分「key 不存在」还是「value 本身就是 null」，且并发环境下无法通过二次 containsKey 消除歧义。

4. “get 操作需要加锁吗？”

不需要。Node 的 val 和 next 都用 volatile 修饰，保证可见性，全程无锁。

5. “ConcurrentHashMap 能保证复合操作的原子性吗？”

不能。putIfAbsent、computeIfAbsent、replace 等方法本身是原子的，但多个方法的组合不是。

十、设计思想总结

回顾演进，优化方向非常清晰：不断缩小锁的粒度，尽可能用无锁操作替代加锁操作。

核心思想：让锁竞争尽可能少发生，让发生的锁竞争影响范围尽可能小。

十一、面试高频问题

Q1：ConcurrentHashMap 1.7 和 1.8 的区别？

1.7 用分段锁（Segment），默认 16 个段，段数会受 concurrencyLevel 影响；1.8 用 CAS + synchronized 锁桶头节点，锁粒度更细，并发度更高，且支持多线程协助扩容。

Q2：JDK 1.8 的 put 流程是什么？

计算 hash → 桶为空用 CAS 无锁插入 → 桶不为空用 synchronized 锁头节点 → 链表/红黑树插入 → 计数 + 扩容判断。发现正在扩容则协助扩容。

Q3：为什么用 synchronized 不用 ReentrantLock？

三个原因：1) 锁粒度足够细，竞争概率低；2) synchronized 已有大量 JVM 优化，低竞争下性能好；3) 减少内存开销。

Q4：get 为什么不需要加锁？

Node 的 val 和 next 用 volatile 修饰，table 引用和桶位访问也具备 volatile 可见性，读操作全程无锁。

Q5：size() 怎么保证准确性？

JDK 1.8 用分散计数（LongAdder 思想），baseCount + CounterCell 数组，避免写热点。没有并发修改时统计结果准确；如果统计过程中仍有并发更新，返回的是弱一致结果，不适合当作强一致控制条件。

Q6：ConcurrentHashMap 能用 null key 吗？

不能。并发场景下 get 返回 null 无法区分「key 不存在」和「value 为 null」。

Q7：复合操作怎么保证线程安全？

使用 ConcurrentHashMap 提供的原子方法：putIfAbsent、computeIfAbsent、merge、replace 等，不要自己组合 if + put。

写在最后

ConcurrentHashMap 是 Java 并发编程的经典之作，它不只是一个线程安全的容器，更是一本并发优化的教科书。看懂它的源码，你学到的不只是一个类的原理，更是一套「用空间换时间、用细粒度替代粗粒度、用协作替代阻塞」的工程化思维。

理解了这些，面试时你就不只是能说出「CAS + synchronized」这个答案，更能讲清楚背后的设计权衡和演进逻辑。

前言刚改完的代码本地跑得好好的，一上测试环境就抛出 java.lang.ClassNotFoundException；Maven 依赖明明已经引入，打包也没报错，翻遍 lib 目录也能找到对应 jar 包，可 JVM 就是说找不到类。 很多人遇到这类问题，第一反应都是”缺包了””依赖没打进去”。这当然是最常见原因，但并不是全部。真实线上场景里，类明明存在却依然加载失败，往往和类加载器隔离、双亲委派模型、线程上下文类加载器、容器自定义加载规则有关。 类加载是 Java 最核心的基础机制之一，也是面试高频考点，更是我们排查依赖冲突、容器隔离、插件化、热部署问题的底层依据。本文就从 ClassNotFoundException 切入，把类加载流程、双亲委派模型、自定义类加载器，以及打破双亲委派的典型场景一次讲透。 一、前置认知：一个类从源码到执行要走几步Java 源码会先被编译成 .class 字节码文件。JVM 运行时不会直接执行 .java 文件，而是通过类加载机制把描述类的数据从 Class 文件、jar 包、网络或其他来源加载到内存中，再完成校验、转换、解析和初始化，最终形成...
