ArrayList vs LinkedList：别再被面试官骗了

发表于2025-07-10|更新于2026-07-19|Java 深度
|浏览量:

前言

面试 Java 开发，集合框架几乎是必考题，而 ArrayList 和 LinkedList 的对比更是高频中的高频。很多人背得滚瓜烂熟：

  • ArrayList 底层是数组，查询快、增删慢
  • LinkedList 底层是链表，增删快、查询慢

可真正写业务代码的时候，90% 的场景大家都会默认用 ArrayList，几乎没人主动用 LinkedList。既然 LinkedList 增删这么厉害，为什么实际开发中没人用？

答案并不是一句「链表增删快」就能概括。这个结论只描述了已经定位到节点后的指针修改成本；真实代码还要考虑按索引定位、对象分配、CPU 缓存和数组批量拷贝。综合这些因素，ArrayList 在多数业务场景中通常更合适。

今天我们就从底层结构、时间复杂度、真实性能三个维度彻底讲透，打破刻板印象，告诉你什么时候该用谁。

一、底层结构：从根上就不一样

两者的所有性能差异，本质都来自底层数据结构的不同。

1.1 ArrayList：动态数组，连续内存

ArrayList 的本质是一个可动态扩容的 Object 数组，数组里保存的是元素引用，这些引用在内存中连续存放。

特点：

  • 数组引用连续，支持通过下标直接定位元素，随机访问效率极高
  • 容量不足时会自动扩容，默认扩容为原容量的 1.5 倍
  • 数组槽位连续，CPU 缓存命中率高，顺序读写性能极强
  • 中间插入、删除元素时，需要移动后续所有元素

1.2 LinkedList：双向链表，分散内存

LinkedList 的本质是双向链表，每个元素都是一个独立的 Node 节点。

特点：

  • 理论上，在已知节点的前后插入、删除，只需要修改指针
  • 没有扩容问题，但元素数量仍受内存和 int size 上限约束
  • 每个节点都要额外存储两个指针，内存开销大
  • 内存不连续，CPU 缓存命中率低，遍历性能差

四、时间复杂度：理论和现实不是一回事

4.1 随机访问 get(int index)

集合 时间复杂度 真实表现
ArrayList O(1) 直接通过数组下标定位，极快
LinkedList O(n) 需要从头/尾遍历到指定位置

10 万元素的 LinkedList，随机 get 第 5 万个元素，要比 ArrayList 慢几千倍。

4.2 尾部添加 add(E e)

集合 理论复杂度 真实表现
ArrayList 均摊 O(1) 直接赋值，极快；扩容是低频操作
LinkedList O(1) 每次都要创建 Node 对象 + 修改指针

实测结论：ArrayList 尾部添加更快。 原因：

  • ArrayList 只是数组赋值，操作极简，数组槽位连续，缓存友好
  • LinkedList 每次要创建 Node 对象，内存分散，缓存失效

4.3 头部添加 add(0, E e)

集合 时间复杂度 真实表现
ArrayList O(n) 所有元素都要向后移动
LinkedList O(1) 直接修改头指针

这是 LinkedList 明显占优的场景之一。若业务确实频繁操作两端，通常还应优先比较专门的双端队列实现 ArrayDeque。

4.4 中间插入 add(int index, E e)

这是最反直觉的场景。

实测结论：按索引做中间插入时，ArrayList 通常比 LinkedList 更快。 原因：

  • System.arraycopy() 是 JVM 底层的 native 方法，经过高度优化，批量内存拷贝速度极快
  • LinkedList 的遍历是逐节点循环，内存分散、缓存失效

如果已经持有目标位置的 ListIterator，LinkedList 在当前位置插入/删除才是 O(1)；但按索引操作时，前面的定位成本无法省掉。

4.5 删除操作

场景 更快的 原因
尾部删除 ArrayList 直接置空
头部删除 LinkedList 修改头指针
中间删除 ArrayList System.arraycopy 优化

五、实测对比：数据不会说谎

JMH 测试的绝对耗时会随 JDK、硬件、元素类型、容量预估和基准写法变化，不能脱离环境背固定数字。更稳定的典型结论如下：

操作场景 通常更有优势 关键原因
尾部追加 ArrayList 数组写入连续，扩容成本可均摊
头部增删 LinkedList / ArrayDeque 不需要搬移整个数组；队列场景通常优先 ArrayDeque
按索引在中间插入 ArrayList 常常更快 LinkedList 先要遍历定位，数组搬移则由高度优化的批量复制完成
随机访问 ArrayList 下标访问 O(1)，LinkedList 需要逐节点遍历
顺序遍历 ArrayList 引用连续，缓存局部性更好
已持有 ListIterator 后插入/删除 LinkedList 已省去定位成本，只需修改相邻指针

做自己的基准时，应使用 JMH 预热、消费返回值以避免死代码消除，并把“定位位置”和“修改结构”是否计入测试说清楚。

结论非常清晰：

  1. 除两端操作或已经持有迭代器位置等场景外，ArrayList 在多数常规操作中更有优势
  2. 随机访问是两者差距最明显的场景之一
  3. 如果是按迭代器逐个删除当前位置，LinkedList 才有 O(1) 断链优势；如果只是整体清空，ArrayList 的连续数组更占优

六、CPU 缓存的影响

为什么理论上相同的 O(n) 操作，ArrayList 实际性能远好于 LinkedList？关键在于 CPU 缓存。

CPU 缓存的工作原理：

  • CPU 读取一个内存地址时，会把相邻的一块内存（缓存行，通常 64 字节）一起加载到缓存
  • ArrayList 的数组引用连续存放，遍历时后续槽位大概率已经在缓存中（缓存命中）
  • LinkedList 的元素分散在堆内存各处，遍历时频繁缓存未命中，需要从主存重新加载

这就是为什么 ArrayList 的顺序遍历、批量拷贝性能远超 LinkedList 的根本原因。

八、常见误区澄清

1. “LinkedList 增删比 ArrayList 快”

只在「已知节点位置、只修改指针」的理想情况下成立。真实业务中，基于索引的增删都需要先找到位置，查找开销已经盖过了指针修改的收益。

2. “LinkedList 更省内存”

通常恰恰相反。ArrayList 的主要额外开销是数组对象及未使用槽位；LinkedList 则要为每个元素单独创建 Node，保存对象头、元素引用和前后指针。具体倍数取决于 JVM 位数、压缩指针、对象对齐和 ArrayList 的空余容量，不能写成固定值。

3. “做队列只能用 LinkedList”

ArrayDeque 作为基于数组的双端队列，两端增删的性能远超 LinkedList，内存占用也更低。

4. “数据量小的时候差不多”

即使数据量只有几百条，ArrayList 的综合性能依然更好。

九、到底该怎么选？

优先选 ArrayList 的场景（90% 的业务场景）

  • 以随机访问、遍历为主
  • 增删操作主要发生在列表尾部
  • 不知道选什么的时候，选 ArrayList 大概率不会错

可以考虑 LinkedList 的场景

  • 需要频繁在列表头部或两端进行增删操作
  • 已经通过 ListIterator 定位，并要在当前位置频繁插入或删除
  • 需要双端队列语义时可以使用，但通常更推荐 ArrayDeque

一个现实的建议

LinkedList 并非无用，只是它的优势条件比“增删很多”更严格：最好是两端操作，或者调用方已经持有节点附近的迭代器位置。普通业务列表很少同时满足这些条件。

默认用 ArrayList，遇到性能瓶颈再针对性优化，是性价比最高的选择。

十、面试高频问题

Q1：ArrayList 和 LinkedList 的区别？

ArrayList 底层是动态数组，数组引用连续，随机访问 O(1)，尾部添加均摊 O(1)；LinkedList 底层是双向链表，内存分散，随机访问 O(n)，头部添加 O(1)。实际开发中 ArrayList 在绝大多数场景下性能更好。

Q2：ArrayList 的扩容机制？

默认构造不会立即分配数组，第一次添加元素时容量变为 10；之后每次扩容约为原容量的 1.5 倍，通过 Arrays.copyOf() 拷贝数据到新数组。

Q3：为什么 ArrayList 比 LinkedList 快？

三个原因：1) 数组下标直接定位，O(1) 随机访问；2) 数组槽位连续，CPU 缓存命中率高；3) System.arraycopy() 是 native 方法，批量拷贝效率极高。

Q4：什么场景用 LinkedList？

频繁在列表头部增删的场景，或者需要实现队列/栈（但 ArrayDeque 更优）。实际开发中极少使用。

Q5：ArrayList 和 Vector 的区别？

Vector 线程安全（synchronized），ArrayList 非线程安全。Vector 已经不推荐使用，需要线程安全时用 Collections.synchronizedList() 或 CopyOnWriteArrayList。

Q6：如何优化 ArrayList 的性能？

  1. 预估容量，指定初始大小，避免频繁扩容；2) 尾部操作为主；3) 大量删除时优先批量处理；4) ArrayList 用普通 for、for-each 都可以，LinkedList 要避免 get(i) 循环。

写在最后

为什么面试题总爱强调 LinkedList 增删快？因为它符合数据结构课本上的理论结论，适合出考题。但理论复杂度只描述了增长趋势，不代表实际运行速度。真实工程里，CPU 缓存、内存连续性、native 优化、对象创建开销，这些都会对最终性能产生巨大影响。

学习技术不是死记硬背结论，而是理解底层原理，知道结论的前提和边界。别再被「链表增删快」的八股文骗了，放到真实场景里，ArrayList 才是那个更靠谱的选择。

前言提到 Redis，大多数人的第一反应是「缓存」：把热点数据放进去，减轻数据库压力。但如果你只把 Redis 当一个键值对缓存用，那就太浪费它的能力了。 Redis 真正的核心优势，在于它提供了一整套成熟的内置数据结构，从简单的字符串到有序集合，每一种都对应着一类典型的业务问题。不用自己手写链表、去重集合、排行榜，一行命令就能解决复杂的业务逻辑，这才是 Redis 不可替代的价值。 今天我们就拆解最常用的五大核心数据结构：String、Hash、List、Set、ZSet，讲透每一种的底层实现、核心特性与典型业务场景，看完就能直接对应到自己的项目里。 一、总览：两层结构设计Redis 的数据结构分为两层： 逻辑结构：我们日常开发直接使用的 API 层，也就是 String、Hash、List、Set、ZSet 五种核心类型 底层编码：每种逻辑结构对应多种底层实现，Redis 会根据元素数量、数据大小自动切换，在空间占用和执行性能之间做最优平衡 理解了底层编码，才能真正看懂不同场景下的性能差异，避免「小数据量很快、数据上来就变慢」的认知偏差。 二、逐个拆解：底层实...
