前言

写 Java 代码的人，几乎没有一天不用 HashMap：存键值对、做数据缓存、接口参数传递，随手就能 new 一个。可一旦被问到底层原理，很多人只能说出「数组加链表」，再追问下去就含糊不清：

  • 红黑树到底什么时候才会触发转换？
  • 扩容时数据是怎么重新迁移的？
  • 为什么容量必须是 2 的整数次幂？
  • JDK 1.8 修复了哪些旧问题，为什么仍然不能并发使用？

HashMap 是 Java 集合框架的重中之重，也是面试必考题。它不只是一个简单的工具类，更浓缩了非常多经典的工程设计思想：空间换时间、位运算性能优化、冲突平衡策略、数据结构动态切换。

今天我们从底层数据结构出发，一步步拆解 put 插入、resize 扩容的完整流程，把哈希冲突、树化退化、扩容迁移这些核心机制讲透。

一、底层结构：数组+链表+红黑树的三位一体

JDK 1.8 是 HashMap 一次重要的版本迭代，它将原本的「数组+链表」结构升级为「数组+链表+红黑树」的混合结构，解决了长链表查询效率低下的问题。

这段代码做了两件事：

为什么要这么做？

因为数组容量通常不大，用 (n-1) & hash 寻址时，本质上只用到了哈希值的低位，高位完全不参与计算，很容易导致哈希冲突。通过高 16 位异或低 16 位，让高位特征也混合到低位中，大幅减少冲突概率。

2.2 寻址公式：用位运算代替取模

为什么采用位与而不是直接写 hash % n？除了位运算成本低，更重要的是当容量为 2 的幂时，它能直接提取低位并始终得到非负下标；现代 JIT 也可能优化部分取模表达式，因此不应把原因只归结为一条“位运算一定快得多”。

六、红黑树转换：链表与树的平衡策略

6.1 树化的两个必要条件

同时满足才会真正树化：

  1. 链表长度 ≥ 8
  2. 数组容量 ≥ 64

如果数组还很小，优先扩容而不是树化，因为小数组下扩容缓解冲突的效果更好。

6.2 退化条件

扩容时，红黑树拆分后的元素数量 ≤ 6，退化为链表。

为什么退化阈值是 6 而不是 8？为了避免频繁来回转换，留 2 的差值形成缓冲带。

6.3 为什么阈值是 8

OpenJDK 源码注释给出的泊松分布近似值中，桶内节点数达到 8 的概率约为 0.00000006，也就是约一亿分之六。这个估算依赖哈希分布良好，并不是对任意自定义 hashCode() 的保证。

正常使用几乎不可能遇到这么长的链表，8 是一种兜底保护。

6.4 为什么用红黑树不用 AVL 树

对比项 红黑树 AVL 树

插入删除 旋转次数少，维护成本低 旋转次数多

适用场景 插入删除频繁 查询密集

HashMap 插入、删除、查询都很频繁，红黑树综合表现更优。

十、常见误区与高频面试点

1. HashMap 是线程安全的吗？

不是。JDK 1.8 避免了 JDK 1.7 扩容迁移中常见的链表成环问题，但这不等于线程安全，并发场景下依然会有问题：

  • size 计数不准确
  • 元素覆盖丢失
  • 并发 put 可能导致数据丢失

多线程场景下，应该使用 ConcurrentHashMap。

2. 为什么负载因子默认是 0.75

时间和空间的平衡：

负载因子 效果
太高（1.0） 空间利用率高，但冲突多，链表长，查询慢
太低（0.5） 冲突少，查询快，但空间浪费，频繁扩容
0.75 JDK 默认采用的时间与空间折中值

3. 为什么 String、Integer 适合做 key

因为它们是不可变类，且正确重写了 hashCode() 和 equals()。

十一、面试高频问题

Q1：HashMap 的底层数据结构是什么？

JDK 1.8 之后是数组 + 链表 + 红黑树。数组是主体，链表解决哈希冲突，链表长度 ≥ 8 且数组容量 ≥ 64 时转为红黑树。

Q2：为什么容量必须是 2 的整数次幂？

为了用位运算 (n-1) & hash 代替取模，位运算效率更高。只有容量是 2 的幂时，n-1 的二进制是连续的 1，才能把哈希值均匀映射到 0 ~ n-1 的下标范围内。

Q3：put 的完整流程是什么？

计算 hash → 判断数组是否初始化（未初始化则 resize）→ 计算下标判断桶是否为空（空则直接放入）→ 桶不为空则判断是否是相同 key（是则覆盖）→ 判断是红黑树还是链表 → 链表尾插法插入 → 链表长度 ≥ 8 尝试树化 → size 超过阈值触发 resize。

Q4：扩容时数据是怎么迁移的？

利用高低位拆分：hash & oldCap 为 0 放原位置，不为 0 放原位置 + 旧容量。不需要重新计算 hash，效率很高。

Q5：HashMap 和 ConcurrentHashMap 的区别？

HashMap 线程不安全，ConcurrentHashMap 线程安全。JDK 1.8 的 ConcurrentHashMap 使用 CAS + synchronized 实现细粒度锁，性能远优于 Hashtable 的全表锁。

Q6：为什么树化阈值是 8？

基于 OpenJDK 源码注释中的泊松分布近似，正常哈希分布下桶内节点数达到 8 的概率约为一亿分之六，是一种极端情况的兜底保护。

写在最后

HashMap 之所以能成为 Java 中最经典的数据结构之一，不在于它用了多么高深的技术，而在于每一处设计都充满了工程化的权衡：

  • 用扰动函数降低冲突概率
  • 用 2 的幂容量换取位运算的高性能
  • 用链表 + 红黑树的动态切换，兼顾普通场景和极端兜底
  • 用高低位拆分法，让扩容效率最大化

看懂源码不是为了死记硬背面试题，而是理解这些权衡背后的思路。当你遇到 HashMap 性能问题、并发异常时，能快速定位根因，知道它在什么场景下会出问题，该怎么优化，这才是读源码真正的价值。
