前言

后台管理系统的订单列表页，一直是慢查询的重灾区。初期数据量小的时候一切正常，随着订单量积累到百万级，翻到后面几十页时，页面加载要转圈好几秒，接口直接超时。查日志发现核心查询 SQL 单条耗时达到 5 秒，已经严重影响运营使用。

SQL 优化不是靠瞎蒙索引，而是一套标准化的排查流程：先定位慢 SQL，再用 EXPLAIN 拆解执行计划，判断索引是否生效、扫描了多少行、有没有额外排序。找到瓶颈后针对性优化，往往能带来数量级的性能提升。

本文就以这条订单列表查询 SQL 为例，从 5 秒优化到 50 毫秒，完整覆盖 EXPLAIN 执行计划分析、索引失效场景、深分页优化 三大核心内容。

type：访问类型

性能从好到差：

type 说明 性能

rows：预计扫描行数

这是优化器估算需要检查的行数，不等同于精确实测值。数值很大通常值得关注，但还要结合 EXPLAIN ANALYZE 的实际行数和耗时判断。

五、覆盖索引：性能优化的利器

5.1 什么是覆盖索引

当查询的所有字段都在索引中时，MySQL 不需要回表查询数据行，直接从索引中获取数据，这就是覆盖索引。

InnoDB 二级索引叶子记录会附带主键，因此这里的 id 也能被覆盖；如果 id 不是聚簇主键，就不能直接套用这个结论。

九、面试高频问题

Q1：EXPLAIN 的 type 字段有哪些？

system > const > eq_ref > ref > range > index > ALL，从左到右性能递减。大表 OLTP 查询通常至少要争取达到 range 级别，具体还要结合 rows、Extra 和实际耗时判断。

Q2：什么是覆盖索引？

查询的所有字段都在索引中，不需要回表查询数据行。EXPLAIN 中 Extra 显示 Using index。

Q3：深分页怎么优化？

两种方案：1) 延迟关联法——子查询只查主键走覆盖索引，再回表；2) 书签式分页——用上一页最后一条的 ID 作为起点，性能最优但不支持跳页。

Q4：联合索引的最左前缀原则是什么？

查询条件要尽量从索引最左边的字段开始连续匹配，跳过左边的字段，通常无法充分利用该联合索引。索引(a,b,c)，查 a 走索引，查 b 不走这个联合索引，查 a+b 走，查 a+c 通常只充分利用 a。

Q5：哪些情况会导致索引失效？

索引列用函数/运算、隐式类型转换、左模糊匹配、不满足最左前缀、OR 分支缺少可用索引等都可能让索引无法有效缩小范围。!=、NOT 和低选择性条件并非必然“失效”，但优化器可能判断全表扫描成本更低。

Q6：索引设计的原则？

优先覆盖核心查询：通常等值条件在范围条件之前，同时考虑排序、常用左前缀和覆盖需求；避免无效或重复索引，并以实际执行计划与读写成本验证，没有固定的索引数量答案。

写在最后

SQL 优化不是玄学，也不是靠堆索引就能解决的。本质是理解 MySQL 的执行逻辑：索引怎么工作、排序怎么实现、分页怎么执行。知道了底层原理，就能预判什么样的写法快、什么样的写法慢。

很多人觉得优化就是加索引，但索引只是基础。真正的优化，是从 SQL 写法、索引设计、分页方案到业务交互的全链路思考。同样的业务需求，换一种写法，性能可能天差地别。
