MySQL 索引原理：B+ 树为什么这么快

发表于2025-11-12|更新于2026-07-19|后端工程
|浏览量:

前言

做后端开发，几乎每天都在和索引打交道：慢查询加个索引，性能立刻提升好几倍。可很多人只知道「索引能提速」，却说不清背后的原理：

  • 为什么数据库选 B+ 树做索引，而不用二叉树、哈希表？
  • 聚簇索引和非聚簇索引到底差在哪？什么是回表？
  • 联合索引为什么要遵循最左前缀原则？
  • 索引下推到底「推」了什么，为什么能提速？

索引从来不是什么玄学，它的所有特性、所有优化规则，都根植于底层的数据结构设计。看懂了 B+ 树，你就能理解所有索引规则的来龙去脉，写 SQL、建索引时就能预判性能，而不是靠瞎试。

今天我们从数据结构的底层逻辑出发，把 B+ 树结构、聚簇/非聚簇索引、联合索引最左前缀、索引下推一次性讲透。

一、索引的本质：用排序换查找效率

1.1 没有索引的代价

没有可用索引时，查询通常只能走全表扫描：读取并检查候选记录，代价随数据页数量增长。是否慢到不可接受还取决于表大小、缓存命中率、返回比例和存储性能。

百万级表上的频繁全表扫描往往会带来明显 IO 和 CPU 压力，但小表或需要返回大部分数据时，全表扫描也可能是优化器的合理选择。

1.2 索引的核心思想

对本文讨论的 InnoDB B+ 树索引而言，索引是一种按键值组织、用于快速定位记录的数据结构。并非所有索引都一定有序，例如哈希索引就不支持范围扫描。

通过额外的存储空间和维护成本，换取查询效率的数量级提升。就像书的目录：翻目录虽然多了一步，但能直接定位到页码。

二、为什么是 B+ 树？数据结构的权衡

2.1 其他结构为什么被淘汰

数据结构 查询复杂度 为什么不适合做索引
二叉搜索树 O(logn)~O(n) 有序插入时退化成链表
平衡二叉树(AVL/红黑树) O(logn) 树太高，百万数据约 20 层，IO 太多
哈希表 O(1) 不支持范围查询、不支持排序
B 树 O(logn) 非叶子节点也存记录，扇出通常低于 B+ 树，范围遍历也不如叶子链表直接

2.3 B+ 树的结构设计

B+ 树在 B 树基础上做了两处关键改进：

  1. 非叶子节点只存索引键和指针，不存数据

    同样大小的磁盘页（16KB），可以放下更多索引键，树的高度极低。

  2. 所有数据都存在叶子节点，叶子节点用双向链表串联

    所有查询最终落到叶子节点，耗时稳定；范围查询顺着链表读即可。

2.4 B+ 树到底有多快？

InnoDB 页默认 16KB。内部节点能容纳多少“键 + 子页指针”，以及叶子页能容纳多少行，都会受主键宽度、行大小、页头和页填充率影响，因此只能做数量级估算：

千万甚至亿级数据，B+ 树高度通常也只有 3~4 层。一次查询只需要访问少量索引页，而且根节点和上层节点往往常驻内存。

三、聚簇索引 vs 非聚簇索引

3.1 聚簇索引：索引即数据

InnoDB 的主键索引就是聚簇索引。B+ 树的叶子节点直接存储完整的行数据。

特点：

  • 一张表只能有一个聚簇索引（数据只能按一种顺序排列）
  • 查询找到索引就找到数据，不需要回表
  • 叶子节点有序链表，范围查询高效

如果表没有主键，InnoDB 会选择第一个所有列都非空的唯一索引作为聚簇索引；如果仍没有合适索引，才会生成 6 字节隐藏 DB_ROW_ID。

3.2 非聚簇索引（二级索引）：索引存主键

普通字段的索引是二级索引，叶子记录保存二级索引键 + 聚簇索引主键值，而不是完整行数据。

回表：通过二级索引找到主键，再去聚簇索引查完整数据。

3.4 核心对比

维度 聚簇索引（主键） 二级索引（非聚簇）
叶子节点内容 完整行数据 二级索引键 + 主键值
每张表数量 只能一个 可以多个
是否需要回表 不需要 一般需要
查询效率 最高 稍低
物理排序 决定数据存储顺序 不影响

4.2 最左前缀原则

查询条件要尽量从索引最左侧字段开始连续匹配。 跳过中间字段后，后续字段通常不能再用于缩小索引扫描范围，但如果字段本身还在索引里，MySQL 仍可能借助索引下推做过滤。

以 idx(a,b,c) 为例：

查询条件 索引生效情况
WHERE a = 1 ✅ a 生效
WHERE a = 1 AND b = 2 ✅ a、b 生效
WHERE a = 1 AND b = 2 AND c = 3 ✅ a、b、c 全部生效
WHERE b = 2 ❌ 通常无法有效利用该联合索引，跳过了 a
WHERE b = 2 AND c = 3 ❌ 通常无法有效利用该联合索引
WHERE a = 1 AND c = 3 ⚠️ a 用于定位，c 可能通过 ICP 过滤，但不能像连续前缀那样缩小扫描范围
WHERE a > 1 AND b = 2 ⚠️ a 用于范围扫描，b 通常不能继续缩小扫描范围

六、索引设计原则

6.1 什么时候该建索引

场景 是否建索引
WHERE 条件频繁使用的字段 ✅ 建
ORDER BY 排序字段 ✅ 建
GROUP BY 分组字段 ✅ 建
JOIN 连接字段 ✅ 建
区分度低的字段（如性别） ❌ 单独建收益低
很少使用的字段 ❌ 不建
数据量很小的表 ❌ 不建

6.2 索引失效的常见场景

场景 错误写法 正确写法
索引列用函数 WHERE DATE(create_time) = '2026-01-01' WHERE create_time >= '2026-01-01' AND create_time < '2026-01-02'
隐式类型转换 WHERE order_no = 123456 WHERE order_no = '123456'
左模糊匹配 LIKE '%keyword' LIKE 'keyword%'
OR 连接非索引字段 WHERE a=1 OR b=2 给 b 也加索引

6.3 索引数量控制

  • 单表索引数量没有适用于所有业务的固定上限，应按核心查询、索引大小和写入压力取舍
  • 索引越多，写入越慢（每次 INSERT/UPDATE/DELETE 都要维护索引）
  • 索引越多，优化器选择越困难

七、面试高频问题

Q1：为什么 MySQL 用 B+ 树而不是红黑树？

红黑树是二叉结构，百万数据高度约为几十层量级；即使热点上层可以缓存，扇出仍很低。B+ 树让一个磁盘页容纳大量键和子页指针，常见大表也只有少量层级，范围查询还可以沿叶子页链表顺序读取。

Q2：聚簇索引和非聚簇索引的区别？

聚簇索引叶子节点存完整行数据，一张表只能有一个；InnoDB 二级索引叶子记录保存二级键和主键值，查询未被索引覆盖时需要回表。

Q3：什么是回表？怎么避免？

二级索引查到主键后，再去聚簇索引查完整数据。避免方式：使用覆盖索引，只查索引中包含的字段。

Q4：联合索引的最左前缀原则是什么？

查询条件要尽量从索引最左侧字段开始连续匹配。索引(a,b,c)，查 b 通常不能有效使用这个联合索引，查 a+b 走索引，查 a+c 通常主要用 a 定位，c 可能通过索引下推过滤。

Q5：什么是索引下推？

MySQL 5.6 的优化，把索引中包含的过滤条件下推到存储引擎层执行，减少回表次数。EXPLAIN 中 Extra 显示 Using index condition。

Q6：索引越多越好吗？

不是。索引占用空间并增加写入维护成本，冗余索引还会增加优化器评估负担。应根据真实查询和索引使用情况取舍，而不是套用固定数量。

写在最后

B+ 树之所以能成为 MySQL 索引的终极答案，本质是围绕「磁盘 IO 是最大瓶颈」这个核心矛盾做的极致优化：

  • 用多路矮胖树结构，把 IO 次数压到最低
  • 用有序叶子链表，天然支持排序和范围查询
  • 用聚簇 + 二级索引的分层设计，平衡空间和查询效率
  • 用最左前缀、索引下推等优化，在现有结构上进一步榨取性能

理解了这些底层逻辑，你就不用死记硬背各种索引规则。遇到慢查询时，能自己预判会不会走索引、为什么慢、该怎么优化，这才是底层知识真正的价值。
