MySQL 事务与锁：MVCC 是怎么实现的

发表于2025-11-22|更新于2026-07-19|后端工程
|浏览量:

前言

在并发场景下操作数据库，我们总会遇到各种反直觉的数据问题：刚插入的数据查不到、同一个事务两次查询同一条数据结果不一样、范围更新时莫名多出不存在的行。这些问题的本质，都和事务隔离性与锁机制息息相关。

而 MVCC（多版本并发控制） 就是 InnoDB 解决读写冲突、提升并发性能的核心武器。它让读操作不用加锁、写操作不阻塞读，在保证隔离性的同时，大幅提升了数据库的并发能力。

很多人听过 MVCC 的名字，却搞不清它到底怎么实现的：undo log 起什么作用？Read View 是怎么判断可见性的？行锁、间隙锁、临键锁又和 MVCC 是什么关系？

今天我们就从 ACID 四大特性讲起，逐层深入，把隔离级别、undo 版本链、Read View 可见性、三类行锁一次性讲透。

一、事务的基石：ACID 四大特性

事务是一组原子性的 SQL 操作，要么全部执行成功，要么全部失败回滚。

特性 含义 底层实现
原子性（Atomicity） 要么全成功，要么全回滚 undo log
一致性（Consistency） 事务前后满足业务规则与数据库约束 数据库约束、应用规则以及 A/I/D 共同保障
隔离性（Isolation） 并发事务互不干扰 MVCC + 锁
持久性（Durability） 提交后数据永久保存 redo log

二、隔离级别：并发问题的平衡点

2.1 三类并发异常

异常 含义 示例
脏读 读到其他事务未提交的数据 事务 A 修改数据未提交，事务 B 读到了；事务 A 回滚
不可重复读 同一事务两次读同一行，结果不同 事务 A 读 id=1，事务 B 修改 id=1 并提交，事务 A 再读结果变了
幻读 同一事务两次范围查询，行数不同 事务 A 查 age>20 有 5 条，事务 B 插入一条 age=25 并提交，事务 A 再查变成 6 条

2.2 四个隔离级别

隔离级别 避免脏读 避免不可重复读 避免幻读 并发性能
读未提交 ❌ ❌ ❌ 最高
读已提交（RC） ✅ ❌ ❌ 较高
可重复读（RR） ✅ ✅ 部分解决 一般
串行化 ✅ ✅ ✅ 最低

InnoDB 默认是 可重复读（RR），普通快照读依赖 MVCC 保持一致视图；当前读、范围更新等需要配合临键锁来避免幻读。

三、MVCC 核心原理

MVCC 的核心思想：写操作产生新版本，读操作读历史版本，读写互不阻塞。

实现依赖三大组件：行隐藏字段、undo log 版本链、Read View。

3.1 行记录的隐藏字段

InnoDB 聚簇索引记录包含用于多版本控制的系统字段，其中 DB_TRX_ID 和 DB_ROLL_PTR 始终参与 MVCC；DB_ROW_ID 只有在 InnoDB 需要自动生成聚簇索引时才作为隐藏行 ID 使用：

隐藏字段 长度 作用
DB_TRX_ID 6 字节 最后修改该行的事务 ID
DB_ROLL_PTR 7 字节 回滚指针，指向上一个版本
DB_ROW_ID 6 字节 没有合适聚簇索引时生成的隐藏行 ID

每次修改数据时，生成 undo log 保存修改前的状态。所有历史版本通过 DB_ROLL_PTR 串成版本链：

六、常见误区澄清

1. “MVCC 完全解决了幻读”

不准确。MVCC 只解决快照读的幻读；当前读需要配合临键锁。

2. “所有隔离级别都使用 MVCC”

不准确。MVCC 主要服务于 RC 和 RR 下的普通快照读；读未提交会读到未提交版本，串行化更偏向用锁来保证隔离，不能简单套用 RC/RR 的 Read View 规则。

3. “RC 级别也有间隙锁”

不准确。RC 级别下普通搜索和索引扫描通常不使用间隙锁，但外键检查、唯一键冲突检查等场景仍可能使用间隙锁。它无法像 RR 的临键锁那样普遍防止幻读。

4. “undo log 是旧数据的副本”

不完全准确。undo 记录保存了回滚操作和重建旧版本所需的信息：insert undo 主要服务事务回滚，update undo 还要支持一致性读；删除在 InnoDB 内部先做 delete-mark，之后再由 purge 清理。它不是简单保存整行副本，也不能只概括成固定的“反向 SQL”。

七、面试高频问题

Q1：MVCC 的实现原理？

三个核心组件：行隐藏字段（DB_TRX_ID、DB_ROLL_PTR）、undo log 版本链、Read View。写操作生成新版本，读操作通过 Read View 判断可见性，读历史版本。

Q2：RC 和 RR 的区别？

核心区别是 Read View 生成时机：RC 每次快照读生成新的，RR 只在第一次快照读时生成之后复用。所以 RC 能看到其他事务最新提交，RR 看到的是一致性快照。

Q3：什么是幻读？InnoDB 怎么解决？

同一事务两次范围查询行数不同。InnoDB 通过 MVCC（快照读）+ 临键锁（当前读）组合解决。

Q4：行锁、间隙锁、临键锁的区别？

行锁锁单行记录，间隙锁锁区间不包含记录，临键锁是行锁+间隙锁的组合（左开右闭）。

Q5：什么是死锁？怎么避免？

两个事务互相等待对方释放锁。避免：固定加锁顺序、减小事务粒度、使用低隔离级别、合理建索引。

Q6：快照读和当前读的区别？

快照读是普通 SELECT，走 MVCC 不加锁；当前读是 SELECT FOR UPDATE / UPDATE / DELETE，读最新版本并加锁。

写在最后

回头看 InnoDB 的隔离性设计，思路非常清晰：

  • 用 undo log 保存历史版本，为多版本提供基础
  • 用 Read View 做可见性判断，解决读写冲突
  • 用行锁、间隙锁、临键锁三级锁机制，解决写写冲突和幻读

MVCC 本质是用空间换时间：多存几个版本，换来读写不阻塞的高并发能力。理解了这些底层原理，遇到死锁、幻读、更新丢失等问题时，能快速定位根因。
