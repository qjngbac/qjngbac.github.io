Spring Bean 生命周期：从创建到销毁的全过程

发表于2025-10-04|更新于2026-07-19|Java 深度
|浏览量:

前言

做 Spring 开发的人，每天都在和 Bean 打交道：加个 @Service 就把类交给容器管理，写个 @Autowired 就能注入依赖。可一旦被问到「Bean 从创建到销毁经历了哪些步骤」「三级缓存是怎么解决循环依赖的」，很多人就只能说出个大概，说不清楚完整流程。

Bean 的生命周期是 Spring 最核心的基础机制，也是面试的高频考点。它不只是用来背的八股，更是我们排查 Bean 创建失败、依赖注入异常、循环依赖报错的底层依据。从一个普通的 Java 对象，到一个能被容器管理的 Spring Bean，中间经历了非常多的步骤和扩展点。

今天我们就把整个生命周期拆解开，从实例化、属性注入、初始化到销毁，完整走一遍全流程，再把三级缓存与循环依赖的底层逻辑讲透。

一、总览：Bean 生命周期的四大阶段

Spring Bean 的完整生命周期，可以概括为四个核心大阶段：

  1. 实例化（Instantiation）：选择构造器或工厂方法并创建对象；构造器本身可能已经写入字段和构造器依赖，并非一定是“空对象”
  2. 属性注入（Populate）：为对象的属性赋值，完成依赖注入
  3. 初始化（Initialization）：执行各种初始化回调，完成对象的功能增强
  4. 销毁（Destruction）：容器关闭时，执行销毁回调，释放资源

这四个阶段是主干，中间还穿插着大量的扩展点：Aware 接口回调、BeanPostProcessor 前后置处理、各种初始化销毁接口等。Spring 强大的扩展能力，正是通过这些插入在生命周期各个节点的钩子实现的。

注意：实例化 ≠ 初始化。 实例化阶段完成构造器或工厂方法调用；后续属性填充会处理 setter、字段等依赖，初始化阶段再执行生命周期回调。构造器已经设置的状态不会等到属性填充阶段才出现。

二、第一阶段：实例化——Bean 的诞生

2.1 前置：BeanDefinition 的加载

在真正创建 Bean 之前，Spring 第一步要做的是读取所有的 Bean 定义信息。

无论是 XML 配置、@Bean 注解还是 @Component 扫描，Spring 都会把这些配置统一解析成 BeanDefinition 对象。它记录了 Bean 的全限定类名、作用域、是否懒加载、依赖关系等所有元信息。

所有 Bean 的创建，都是以 BeanDefinition 为蓝图进行的。

2.2 实例化：创建对象空壳

实例化阶段的核心目标，是选择构造器或工厂方法并创建 Java 对象。此时对象仍是 Spring 生命周期意义上的“半成品”：构造器依赖和构造器逻辑已经生效，但 setter/字段依赖尚未全部填充，也没有执行初始化回调。

关键执行点：

  • 先判断该 Bean 是否已经被创建，单例 Bean 只会创建一次
  • 选择合适的构造函数（默认无参构造，也支持有参构造注入）
  • 通过反射实例化对象

这里有一个非常重要的设计：对允许早期暴露、且当前正在创建的单例 Bean，Spring 会在合适时机把用于获取早期引用的 ObjectFactory 放进三级缓存。并非每个 Bean、每个作用域都会走这一步，后面详细讲。

三、第二阶段：属性注入——Bean 的填装

实例化完成后，就进入 populateBean 阶段，也就是我们常说的依赖注入。

3.1 核心工作

这个阶段会遍历 Bean 的所有属性：

  • 处理 @Autowired、@Resource、@Inject 等注入注解
  • 处理 @Value 配置注入
  • 如果依赖的其他 Bean 还没创建，会递归触发目标 Bean 的完整创建流程

比如 A 依赖 B，创建 A 的属性注入阶段，发现 B 还没创建，就会暂停 A 的创建，先去完整走一遍 B 的生命周期，拿到 B 的实例后，再注入给 A。

3.2 循环依赖的触发点

正是在这个阶段，循环依赖的问题会暴露出来：A 创建到一半，需要注入 B；B 创建到一半，又需要注入 A。如果没有特殊处理，就会陷入无限递归，最终栈溢出。

Spring 之所以能解决大部分循环依赖，核心就是在实例化之后、属性注入之前，先通过三级缓存暴露一个可获取早期引用的工厂。

这里还要区分 Spring Framework 能力 与 Spring Boot 默认策略：Spring Framework 可以处理部分单例 setter/字段循环依赖，但 Spring Boot 自 2.6 起默认 spring.main.allow-circular-references=false，新项目遇到这类依赖通常会直接启动失败。三级缓存解释的是容器内部如何处理早期引用，不代表应在业务设计中依赖循环引用。

四、第三阶段：初始化——Bean 的成熟

属性注入完成后，Bean 还不算真正”可用”，接下来会进入完整的初始化流程。这一阶段是 Spring 扩展点最密集的地方，AOP、事务代理、各种增强逻辑都在这里完成。

4.1 第一步：Aware 接口回调

如果 Bean 实现了对应的 Aware 接口，Spring 会注入相应容器资源。BeanNameAware、BeanClassLoaderAware、BeanFactoryAware 由 BeanFactory 直接回调；EnvironmentAware、ApplicationContextAware、ResourceLoaderAware 等通常由 ApplicationContextAwareProcessor 在初始化前处理，因此不要把表格顺序理解成所有 Aware 接口的统一固定调用链：

接口 注入内容 说明
BeanNameAware Bean 的名称 获取自己在容器中的名字
BeanClassLoaderAware 类加载器 获取加载自己的 ClassLoader
BeanFactoryAware BeanFactory 实例 获取 Bean 工厂
EnvironmentAware Environment 获取环境配置
ApplicationContextAware ApplicationContext 获取应用上下文
ResourceLoaderAware ResourceLoader 获取资源加载器

简单说：Aware 接口就是让 Bean 能拿到 Spring 容器的底层资源。

接下来，所有注册的 BeanPostProcessor 会依次执行 postProcessBeforeInitialization 方法。

这是 Spring 最重要的扩展点之一：可以在 Bean 初始化之前，对 Bean 进行自定义修改、包装、属性增强。

常见的 BeanPostProcessor 实现：

4.4 第四步：BeanPostProcessor 后置处理

初始化方法执行完后，会再次经过所有 BeanPostProcessor，执行 postProcessAfterInitialization 方法。

这是大多数 AOP 动态代理生成的地方：如果 Bean 需要被代理（比如加了 @Transactional 注解、切面注解），这里会把原始对象包装成代理对象，最终放入容器的是代理对象，而不是原始对象。遇到循环依赖时，Spring 也可能通过三级缓存里的 ObjectFactory 提前暴露代理引用。

五、第四阶段：销毁——Bean 的落幕

Bean 创建完成后，就会一直存活在容器中，直到应用关闭、容器销毁。此时 Bean 会进入销毁阶段，执行资源释放逻辑。

  • 应用正常关闭，调用 context.close()
  • JVM 正常退出前，注册了关闭钩子的容器会自动触发销毁
  • Spring Boot 应用关闭时自动触发

七、灵魂考点：三级缓存与循环依赖

讲完了完整生命周期，我们来拆解面试必问的三级缓存。这是 Spring 生命周期设计中最精妙的部分之一。

7.1 什么是循环依赖

简单说就是两个 Bean 互相依赖：A 依赖 B，B 又依赖 A。

  • A 创建到属性注入阶段，发现需要 B，于是去创建 B
  • B 创建到属性注入阶段，发现需要 A，于是又去创建 A
  • 没有特殊处理的话，就会无限递归，最终栈溢出

7.2 三级缓存分别是什么

Spring 在 DefaultSingletonBeanRegistry 中维护了三个 Map，也就是我们常说的三级缓存：

缓存层级 对应 Map 存储内容 作用
一级缓存 singletonObjects 完整的成品 Bean 已走完生命周期，可直接使用
二级缓存 earlySingletonObjects 半成品 Bean 已实例化但未初始化，解决循环依赖
三级缓存 singletonFactories ObjectFactory 工厂 生成早期引用，支持代理场景

7.4 为什么需要三级？两级不行吗？

这是面试的高频追问。答案是：为了支持 AOP 代理场景。

如果 Bean 需要被 AOP 增强，最终放入容器的是代理对象，而不是原始对象。那循环依赖注入给其他 Bean 的，也应该是代理对象，而不是原始对象。

三级缓存的 ObjectFactory 就是干这个的：

需要获取早期引用时，通过工厂生成对应的代理对象，保证注入给其他 Bean 的是正确的引用。

如果只有两级缓存，就不得不更早、也更粗暴地决定是否创建代理对象；三级缓存把这个动作延迟到真正需要早期引用时，既能处理循环依赖，又能兼容 AOP 的代理创建流程。

7.5 哪些循环依赖解决不了

不是所有循环依赖 Spring 都能搞定，以下场景依然会报错：

场景 原因
构造器注入的循环依赖 实例化阶段就需要依赖对象，连对象都创建不出来
prototype 作用域的循环依赖 原型 Bean 每次获取都新建，没有缓存机制
某些代理创建时机导致的循环依赖 例如 @Async 等场景可能无法在早期引用阶段得到合适代理

构造器循环依赖的处理思路：

优先做法是重新梳理职责：抽取 A、B 都依赖的第三个服务，或通过领域事件解除双向依赖。把构造器注入改成字段注入只是在部分容器配置下把问题推迟到属性阶段，还会隐藏必需依赖；@Lazy、ObjectProvider 可以作为确有必要时的局部手段，但不应代替结构调整。

八、Bean 作用域

Spring 支持多种 Bean 作用域，不同作用域的生命周期管理方式不同：

作用域 说明 生命周期管理
singleton 默认，单例 容器管理完整生命周期
prototype 原型，每次获取新建 容器只管创建，不管销毁
request 每次 HTTP 请求新建 Web 环境，请求结束销毁
session 每个会话新建 Web 环境，会话结束销毁
application 每个 ServletContext 新建 Web 环境，应用关闭销毁

如果一个 singleton 直接注入 prototype，依赖只在 singleton 创建时解析一次，之后不会自动换成新实例。需要“每次使用都创建”时，可注入 ObjectProvider<PrototypeService> 并在每次调用时执行 getObject()，或使用合适的 scoped proxy。

九、FactoryBean vs BeanFactory

这是两个名字很像但完全不同的东西：

对比项 BeanFactory FactoryBean
本质 Spring 容器的根接口 一种特殊的 Bean
作用 管理所有 Bean 的创建和获取 自定义复杂 Bean 的创建逻辑
使用方式 context.getBean("name") 实现 FactoryBean 接口

十一、常见误区澄清

  1. “BeanPostProcessor 只在初始化前后执行”

    准确说，是每个 Bean 的初始化前后都会执行。容器中有多少个 Bean，就会执行多少次。

  2. “三级缓存就是专门为了解决循环依赖设计的”

    不准确。三级缓存主要服务于单例 Bean 的早期暴露和循环依赖处理，其中 ObjectFactory 的价值在于延迟生成早期引用，兼容 AOP 代理等场景。

  3. “初始化就是给属性赋值”

    不对。属性赋值是第二阶段 populateBean 做的事；初始化阶段是执行自定义的初始化逻辑，两者是完全独立的步骤。

  4. “所有 Bean 的生命周期都由 Spring 管理”

    不准确。singleton 由容器管理完整生命周期，request/session 等受注册 Scope 管理并在作用域结束时销毁；prototype 通常只由容器负责创建，后续销毁要由调用方处理。

  5. “@PostConstruct 是 Spring 的注解”

    不是，它来自 JSR-250 规范。Spring 5 / Boot 2 常见包名是 javax.annotation.PostConstruct，Spring 6 / Boot 3 迁移为 jakarta.annotation.PostConstruct，Spring 负责识别并调用它。

十二、面试高频问题

Q1：Spring Bean 的生命周期是什么？

实例化 → 属性注入 → Aware 接口回调 → BeanPostProcessor 前置处理 → @PostConstruct → InitializingBean → 自定义 initMethod → BeanPostProcessor 后置处理（AOP 代理）→ 使用 → @PreDestroy → DisposableBean → 自定义 destroyMethod。

Q2：三级缓存分别存什么？

一级缓存存完整 Bean，二级缓存存半成品 Bean（已实例化未初始化），三级缓存存 ObjectFactory 工厂对象。

Q3：为什么需要三级缓存而不是两级？

为了支持 AOP 代理场景。三级缓存的 ObjectFactory 可以在需要时动态创建代理对象，保证循环依赖注入的是代理对象而非原始对象。

Q4：哪些循环依赖解决不了？

构造器注入的循环依赖（对象都创建不出来）、prototype 作用域的循环依赖（没有缓存机制）、某些代理创建时机导致的循环依赖（如 @Async 等场景需要结合具体代理方式排查）。

Q5：BeanPostProcessor 和 BeanFactoryPostProcessor 的区别？

BeanPostProcessor 作用于 Bean 级别，在每个 Bean 初始化前后执行；BeanFactoryPostProcessor 作用于容器级别，在 BeanDefinition 加载后、Bean 实例化前执行，可以修改 BeanDefinition。

Q6：@PostConstruct、afterPropertiesSet、initMethod 的执行顺序？

@PostConstruct → InitializingBean.afterPropertiesSet() → 自定义 initMethod。

写在最后

Spring Bean 的生命周期，看似步骤繁多、概念零散，实则逻辑非常清晰：从创建对象到填充依赖，再到初始化增强，最后销毁释放资源。每一个阶段都预留了扩展点，这也是 Spring 能成为最流行框架的重要原因。

理解生命周期，不是为了死记硬背面试题。当你遇到 Bean 注入为 null、循环依赖报错、AOP 不生效这些问题时，只有清楚每一步发生了什么，才能快速定位到是哪个阶段出了问题，而不是靠瞎猜和试错。
