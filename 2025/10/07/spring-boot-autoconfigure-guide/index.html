Spring Boot 自动装配原理：@SpringBootApplication 背后做了什么

发表于2025-10-07|更新于2026-07-19|Java 深度
|浏览量:

前言

做 Spring Boot 开发的人，每天都会写 @SpringBootApplication 注解。点一下启动类，服务就能跑起来；引入一个 mybatis-spring-boot-starter，不用写 SqlSessionFactory 配置，直接就能注入 Mapper 使用。

我们常说 Spring Boot “开箱即用”，这份便利的背后，核心就是自动装配机制。但很多人对它的认知停留在”面试要背的八股”层面，既说不清楚 @EnableAutoConfiguration 到底干了什么，也讲不明白自动配置候选类是怎么被加载的，遇到 starter 不生效的问题只能瞎试。

自动装配不是什么黑魔法，它是一套设计精巧的组件扩展机制。今天我们就从最熟悉的启动注解切入，逐层拆解 @SpringBootApplication 的内部构成，把自动配置候选加载机制、条件注解筛选逻辑、完整装配流程一次性讲透。

一、@SpringBootApplication：原来是个组合注解

很多人以为它是一个全新的魔法注解，其实点开源码就会发现，它本质是一个组合注解，封装了三个核心注解：

三个注解各司其职，共同完成启动类的全部能力。

1.1 @SpringBootConfiguration：配置类标记

它以 Spring 的 @Configuration 为基础，标记启动类也是配置类，并让 Spring Boot 测试等基础设施能够定位主配置。其 proxyBeanMethods 属性还会别名映射到 @Configuration 的同名属性，因此不只是一个没有附加语义的名称包装。

1.2 @ComponentScan：包扫描

这是 Spring 框架原生就有的能力，负责扫描启动类所在包及其子包下所有标注了 @Component、@Service、@Repository、@Controller 等注解的类，把它们注册到 Spring 容器中。

一个高频误区：自动装配就是包扫描。
完全不是。默认情况下，@ComponentScan 从启动类所在包向下扫描；显式配置 basePackages 后也可以扫描第三方包，所以并非技术上“扫不到 jar”。自动装配的关键价值是第三方库无需要求用户把其内部包加入组件扫描，而是通过明确的候选清单和条件注解按需生效。

1.3 @EnableAutoConfiguration：自动装配的开关

这才是 Spring Boot 的灵魂，也是整个自动装配机制的入口。所有”开箱即用”的能力，都由这个注解开启。

二、自动装配入口：@EnableAutoConfiguration

2.2 核心：AutoConfigurationImportSelector

3.2 SpringFactoriesLoader：SPI 加载器

Spring 提供了 SpringFactoriesLoader 工具类，它会扫描所有 classpath 下的 jar 包，读取每个 jar 中 META-INF/spring.factories 文件，根据指定的 key 加载对应的实现类。

需要注意的是：SpringFactoriesLoader 仍然是 Spring 的通用 SPI 加载工具；但 Spring Boot 2.7+/3.x 的自动配置候选类主要来自 AutoConfiguration.imports 文件，不再把 spring.factories 作为新的自动配置注册入口。

四、按需生效：条件注解的筛选逻辑

读到这里你可能会有疑问：Spring Boot 内置了上百个自动配置类，难道全部都会加载吗？如果我没引入 Redis 依赖，RedisAutoConfiguration 也会执行吗？

当然不会。所有自动配置类都不是无条件生效的，它们会通过条件注解进行第二轮筛选，只有满足条件的配置类才会真正执行。

4.1 根注解：@Conditional

条件注解的源头是 Spring 4.0 引入的 @Conditional，它可以指定一个条件判断类，只有条件返回 true 时，对应的 Bean 才会被注册。

4.2 Spring Boot 派生的条件注解

Spring Boot 在 @Conditional 基础上，派生出了一系列开箱即用的条件注解：

注解 作用 典型场景
@ConditionalOnClass 类路径下存在指定的类时才生效 检测是否引入了对应依赖
@ConditionalOnMissingClass 类路径下不存在指定的类时才生效 排除特定依赖时的配置
@ConditionalOnBean 容器中存在指定 Bean 时才生效 依赖其他 Bean 的配置

@ConditionalOnWebApplication 当前是 Web 应用时才生效 Web 相关配置
@ConditionalOnNotWebApplication 当前不是 Web 应用时才生效 非 Web 环境配置
@ConditionalOnResource 类路径下存在指定资源文件时才生效 检测配置文件是否存在
@ConditionalOnExpression SpEL 表达式为 true 时才生效 复杂条件判断

九、常见误区澄清

  1. “自动装配就是包扫描”

    不对。@ComponentScan 按配置的包路径扫描组件，默认根包通常位于项目启动类处，也可以显式包含第三方包；自动装配则通过候选文件和条件注解导入配置类，两者是不同入口。

  2. “spring.factories 只能用来做自动配置”

    不对。它是通用的 SPI 扩展机制，除了自动配置，还可以扩展 ApplicationContextInitializer、ApplicationListener、FailureAnalyzer 等多种组件。

  3. “@ConditionalOnMissingBean 只是为了去重”

    不只是去重，它用于自动配置的“退避”：在评估时没有匹配 Bean 才提供默认实现。自定义自动配置时还要注意配置类处理顺序，避免条件在目标 Bean 注册之前被过早评估。

  4. “引入 starter 就一定会生效”

    不一定。starter 只是把自动配置类引入了 classpath，最终是否生效还要看条件注解是否满足。

  5. “Spring Boot 3.x 还是用 spring.factories”

    不是。3.x 已经改为 META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports，旧格式不再支持自动配置。

十、面试高频问题

Q1：@SpringBootApplication 由哪几个注解组成？

由 @SpringBootConfiguration（配置类标记）、@EnableAutoConfiguration（自动装配开关）、@ComponentScan（包扫描）三个注解组合而成。

Q2：自动装配的原理是什么？

@EnableAutoConfiguration 通过 @Import 引入 AutoConfigurationImportSelector，它先读取自动配置候选类，再通过条件注解筛选，最终将满足条件的配置类注册到容器中。Spring Boot 2.6 及之前主要来自 META-INF/spring.factories，Spring Boot 2.7+/3.x 主要来自 AutoConfiguration.imports。

Q3：spring.factories 的作用是什么？

它是 Spring 的通用 SPI 扩展载体之一，以 Properties 格式存储接口与实现类的映射关系。Spring Boot 2.6 及之前也用它注册自动配置；Spring Boot 2.7+/3.x 的自动配置注册改用 AutoConfiguration.imports。

Q4：条件注解是怎么工作的？

每个条件注解对应一个 Condition 实现类，在 Bean 注册前检查条件是否满足（类是否存在、Bean 是否存在、配置是否匹配等），不满足则跳过注册。

Q5：如何自定义一个 Starter？

  1. 创建 autoconfigure 模块，编写自动配置类和配置属性类
  2. 在 META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports 中注册配置类
  3. 创建 starter 模块，引入 autoconfigure 模块依赖
  4. 使用者引入 starter 依赖，配置属性即可使用

Q6：Spring Boot 2.x 和 3.x 自动装配的区别？

2.6 及之前主要使用 META-INF/spring.factories 文件，2.7 引入并兼容 META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports，3.x 自动配置注册应使用 imports 文件，格式更简洁，加载更高效。

写在最后

Spring Boot 的自动装配，看似神奇，实则是非常经典的架构设计思想：约定大于配置 + SPI 扩展机制 + 条件化装配。

它把通用的、重复的配置封装成 starter，让业务开发者不用关心底层组件的搭建，专注于业务逻辑；同时又保留了足够的扩展能力，需要自定义时可以随时覆盖默认实现。

理解了自动装配原理，你不仅能从容应对面试，更能在遇到 starter 不生效、Bean 冲突、配置异常时快速定位根因，甚至能自己封装业务 starter，提升团队的开发效率。
