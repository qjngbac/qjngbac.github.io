分布式事务实战：Seata 的 AT 模式原理

发表于2026-03-16|更新于2026-07-19|后端工程
|浏览量:

前言

微服务架构下，单体应用被拆成多个服务，数据库也随之按业务拆分。原本一个本地事务就能搞定的下单流程，现在变成了「订单服务建单 + 库存服务扣库 + 账户服务扣钱」跨三个服务、三个数据库的操作。单个服务的本地事务只能保证自己的数据正确，只要有一个环节失败，就会出现订单建了但库存没扣、钱扣了但订单没生成的数据不一致问题。

这就是分布式事务要解决的核心问题：跨服务、跨数据源场景下，如何保证多个操作要么全部成功、要么全部失败。

业界分布式事务方案有很多：TCC、Saga、可靠消息最终一致等等。Seata AT 模式通过代理 JDBC 数据源、自动记录数据镜像和生成补偿操作，把业务侵入降得很低；业务代码通常只需标记全局事务，但数据库表、SQL 类型、数据源代理和调用链透传仍需满足框架约束。

本文就从架构角色、核心原理到完整实战，一次性讲透 Seata AT 模式的底层逻辑与落地方法。

一、Seata 核心架构：三个角色 + 两个层级

Seata 把分布式事务拆分为「全局事务」和「分支事务」，并通过三个核心角色协同完成整个事务的协调与执行。

1.1 两个事务层级

  • 全局事务：贯穿整个业务链路的大事务，比如一次完整的下单流程就是一个全局事务。它由多个分支事务组成，最终要么全部提交，要么全部回滚。
  • 分支事务：每个服务内部的本地事务，是全局事务的一个组成单元。每个分支事务只负责自己数据库的操作，服从全局事务的统一调度。

全局事务有唯一的 XID（全局事务 ID），在整个调用链路中透传，所有分支事务都通过 XID 归属到同一个全局事务下。

1.2 三大核心角色

角色 全称 职责 部署位置
TC Transaction Coordinator 维护全局事务和分支事务的状态，指挥所有 RM 提交或回滚 独立部署的 Seata Server
TM Transaction Manager 全局事务的发起者，负责开启、提交或回滚全局事务 业务发起方（如订单服务）
RM Resource Manager 管理分支事务资源，执行本地事务，生成回滚日志 每个业务服务内嵌

用一次下单流程串起来就是：

  1. 订单服务（TM）发起全局事务，向 TC 申请一个全局事务 ID（XID）
  2. XID 沿着调用链传递到库存服务、账户服务（通过 Feign/Servlet 自动透传）
  3. 每个服务的 RM 把自己的本地事务注册为分支事务，向 TC 报备
  4. 所有分支执行完毕后，TM 向 TC 发起全局提交/回滚
  5. TC 指挥所有 RM 统一提交或回滚分支事务

1.3 Seata 的四种事务模式

Seata 不只有 AT 模式，它支持四种事务模式，适用于不同场景：

模式 原理 侵入性 一致性 适用场景
AT 自动记录数据镜像并执行补偿 低侵入 默认读隔离较弱，最终收敛 支持本地 ACID 事务且通过 JDBC 访问的关系型数据库
TCC 手写 Try/Confirm/Cancel 强侵入 最终一致 非关系型数据库、自定义补偿逻辑
Saga 长事务 + 补偿服务 中等 最终一致 长流程业务、跨多服务编排
XA 数据库原生 XA 协议 低侵入 强一致性更强 数据源支持 XA、可接受资源长期占用的场景

AT 模式是使用最广泛的，也是本文的重点。

二、AT 模式核心原理：自动补偿，低侵入

AT（Auto Transaction）模式是 Seata 最具特色的实现。它的核心设计思想是：自动生成回滚日志，全局回滚时自动反向补偿，业务代码通常不需要手写补偿逻辑。

2.1 整体机制

AT 模式基于 JDBC 数据源代理做拦截，在业务 SQL 执行的同时，自动生成数据的前后镜像，写入 undo_log 表。补偿过程对业务代码较透明，但仍需要全局事务注解、XID 透传、代理数据源、主键和受支持的 SQL。

它的执行流程分为两个阶段：

  • 一阶段：执行业务本地事务，同时生成 undo 回滚日志，向 TC 注册分支，申请全局锁
  • 二阶段提交：异步删除 undo 日志，释放全局锁，速度极快
  • 二阶段回滚：根据 undo 日志反向执行 SQL，补偿数据，释放全局锁

2.2 一阶段：业务执行 + 自动生成回滚日志

一阶段是 AT 模式最精妙的部分，业务执行和回滚日志生成在同一个本地事务内完成。

完整步骤：

  1. 拦截 SQL：RM 通过 DataSourceProxy 代理数据源，拦截业务的 UPDATE/INSERT/DELETE 语句
  2. 查询前镜像：根据业务 SQL 的 WHERE 条件，自动查询出修改前的数据行，生成「前镜像（Before Image）」
  3. 执行业务 SQL：正常执行业务的增删改操作
  4. 查询后镜像：根据主键查询修改后的数据行，生成「后镜像（After Image）」
  5. 生成 undo log：把前后镜像、业务 SQL、表名、主键信息组装成回滚日志，和业务数据在同一个本地事务中写入数据库
  6. 注册分支 + 申请锁：向 TC 注册分支事务，并申请对应数据行的全局锁
  7. 提交本地事务：业务数据和 undo_log 在同一个本地事务中一起提交

关键点：一阶段结束时，本地事务已经提交，数据已经落库。这意味着 AT 模式是最终一致性方案，中间存在短暂的数据不一致窗口。

2.3 二阶段提交：异步清理

如果所有分支都执行成功，TM 发起全局提交：

  1. TC 收到提交请求，标记全局事务为提交状态
  2. 异步通知所有 RM 删除对应的 undo_log 日志，释放全局锁
  3. 提交过程非常快，因为只需要删除日志，不需要操作业务数据

2.4 二阶段回滚：自动反向补偿

如果任意一个分支执行失败，TM 发起全局回滚：

  1. TC 收到回滚请求，指挥所有 RM 执行回滚
  2. RM 根据 XID 和分支 ID 找到对应的 undo_log
  3. 数据校验（脏写检测）：用后镜像和当前数据库数据对比，如果数据不一致，说明在全局事务执行期间被其他事务修改了，触发告警，需要人工介入处理
  4. 生成反向 SQL：根据前镜像自动生成反向语句
    • 前镜像有数据，后镜像有数据 → 生成 UPDATE，用前镜像覆盖当前数据
    • 前镜像无数据，后镜像有数据 → 生成 DELETE，删除当前数据
    • 前镜像有数据，后镜像无数据 → 生成 INSERT，把前镜像数据插回去
  5. 执行反向 SQL，完成数据回滚
  6. 删除 undo_log，释放全局锁，向 TC 上报回滚结果

正常情况下回滚由框架自动完成，业务代码不需要手写反向 SQL，这就是 AT 模式最大的优势。若当前数据与后镜像不一致，Seata 会按脏数据处理策略报告回滚失败或交给自定义 FailureHandler 处理，不能简单理解为一定能够自动恢复。

2.5 隔离性如何保证？

分布式事务下，隔离性是个难题。AT 模式通过两把锁来平衡并发与安全：

  • 本地锁：数据库本身的行锁，一阶段本地事务提交前持有
  • 全局锁：TC 维护的逻辑锁，一阶段结束后持有，直到二阶段提交/回滚才释放

在本地数据库隔离级别为读已提交或更高时，AT 普通 SELECT 仍可能读到“分支已本地提交、全局事务尚未结束”的数据，因此全局事务层面默认近似读未提交。若需要全局读已提交，可使用 Seata 代理支持的 SELECT ... FOR UPDATE：查询会尝试获取全局锁，等待相关全局事务结束后再返回。

2.6 写隔离与脏写防护

AT 模式通过全局锁实现写隔离：

  • 事务 A 修改 row_x，一阶段提交后持有 row_x 的全局锁
  • 事务 B 也想修改 row_x，它会先在本地事务中持有数据库行锁，再反复尝试申请全局锁；超时仍拿不到时会回滚本地事务
  • 事务 A 回滚时，数据被正确恢复；事务 B 拿到锁后才开始执行

如果写操作绕过 Seata 数据源代理，它不会参与全局锁竞争，可能在 AT 一阶段提交后改写同一行，导致后续回滚镜像校验失败。所以需要纳入分布式事务保护的共享数据写入都应经过 Seata 代理；无法接入的写入应通过业务版本号、幂等或其他并发控制补足。

四、常见坑点与最佳实践

1. 表必须有主键

AT 模式通过主键定位数据行、生成前后镜像。没有主键的表无法生成回滚日志，会直接报错。

2. 一致性级别是最终一致

一阶段本地事务已经提交，普通查询可能读到全局事务尚未最终确认的数据；二阶段提交异步清理，回滚则执行补偿。这个窗口取决于链路耗时、锁冲突和故障恢复，不能笼统保证为毫秒级，不适合要求严格隔离的核心账务场景。

3. 尽量避免长事务

全局锁持有时间越长，并发冲突概率越高。长事务会严重降低系统吞吐量，还容易导致锁超时。尽量把全局事务内的操作做精简。

4. TC 必须高可用

TC 是整个分布式事务的协调中心，单点故障会导致所有事务无法提交/回滚。生产环境必须部署 TC 集群，配合注册中心实现高可用。

5. 复杂 SQL 的限制

AT 模式对复杂 SQL 的支持有限：

  • 多表 JOIN 的 UPDATE 语句，可能无法正确生成前后镜像
  • 子查询中的 UPDATE，镜像生成可能不完整
  • 建议 AT 模式下使用简单的单表 CRUD 操作，复杂场景用 TCC 模式

6. undo_log 的清理

正常提交时，undo_log 会在二阶段异步批量删除；回滚完成后也会删除对应记录。需要监控长期残留，但不能脱离 XID、分支状态和保留策略直接手工清表，否则可能破坏仍在恢复中的事务。

五、AT vs TCC：怎么选？

维度 AT 模式 TCC 模式
业务侵入 低侵入，依赖数据源代理和 SQL 支持 强侵入，手写 Try/Confirm/Cancel
开发成本 低 高
一致性 最终一致 最终一致
性能 有镜像查询、undo_log 和全局锁开销 可按业务控制资源预留，性能取决于实现
灵活性 低，只能操作关系型数据库 高，可操作任意资源
适用场景 JDBC 关系型数据库，绝大多数场景 非关系型数据库、自定义补偿、高性能要求

选型建议：

  • 常规 JDBC 单表事务可优先评估 AT：开发成本较低，但要先确认数据库、SQL、热点行竞争和隔离要求
  • 特殊场景选 TCC：操作 Redis/MQ 等非数据库资源、需要自定义补偿逻辑、对性能要求极高

六、面试高频问题

Q1：Seata 的三个角色是什么？

TC（事务协调者）：独立部署，维护全局事务状态，指挥提交/回滚。TM（事务管理者）：业务发起方，开启/提交/回滚全局事务。RM（资源管理者）：每个服务内嵌，执行本地事务，注册分支，执行提交/回滚。

Q2：AT 模式的一阶段做了什么？

代理数据源拦截 SQL → 查询前镜像 → 执行业务 SQL → 查询后镜像 → 生成 undo_log（与业务数据在同一本地事务中提交）→ 注册分支事务 + 申请全局锁 → 提交本地事务。

Q3：AT 模式回滚时怎么保证数据正确？

根据 undo_log 中的前镜像生成反向 SQL。回滚前用后镜像和当前数据做校验；不一致表示存在事务外修改或脏写，回滚会按配置失败重试、记录异常或交给自定义处理器，不能直接覆盖当前数据。

Q4：AT 模式的一致性级别是什么？

AT 一阶段会提交本地事务，普通查询可能在全局事务结束前看到中间状态；全局提交异步清理，回滚通过补偿恢复，因此通常按最终收敛理解。需要全局读已提交时，可结合 SELECT ... FOR UPDATE 与全局锁。

Q5：AT 和 TCC 怎么选？

AT 业务侵入较低，适合受支持的 JDBC 关系型数据库与 SQL 场景。TCC 侵入性强但资源控制更灵活，适合非数据库资源、自定义预留和补偿场景；应按隔离、热点竞争和开发成本选择。

Q6：什么是全局锁？和本地锁的区别？

本地锁是数据库行锁，一阶段提交前持有。全局锁是 TC 维护的逻辑锁，一阶段结束后持有直到二阶段完成。全局锁保证多个全局事务不会同时修改同一行数据，防止脏写。

写在最后

Seata AT 模式之所以流行，核心在于它降低了分布式事务的开发成本。它把协调与常见补偿逻辑封装到框架里，业务侧主要关注事务边界和 SQL，但仍要正确配置数据源代理、XID 透传、undo_log 与 TC，才能形成完整链路。

当然，没有万能的方案。AT 模式也有它的边界：最终一致、依赖关系型数据库、不适合超高频热点数据。理解它的原理和适用场景，才能在合适的地方用合适的方案，而不是为了用而用。
