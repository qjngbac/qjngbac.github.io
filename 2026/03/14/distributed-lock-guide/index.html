Redis 分布式锁：从 Redisson 到 RedLock

发表于2026-03-14|更新于2026-07-19|后端工程
|浏览量:

前言

在分布式系统中，多个服务实例共同操作同一个共享资源时，本地锁已经完全失效。比如库存扣减、订单幂等、定时任务防重执行，这些场景都需要一把全局的「锁」来保证互斥性，这就是分布式锁。

实现分布式锁的方案有很多：数据库、ZooKeeper、Redis 都可以做。其中 Redis 凭借高性能、低延迟、实现简单的特点，成为绝大多数业务场景的首选。但很多人对 Redis 分布式锁的认知停留在「SETNX 加锁、DEL 解锁」的初级版本，既说不清楚 Redisson 的看门狗机制，也讲不明白 RedLock 要解决什么问题。

本文就从最基础的手写锁开始，一步步演进到工业级的 Redisson 实现，再到多集群下的 RedLock 算法，最后对比 Redis 与 ZooKeeper 两种分布式锁的选型差异，把分布式锁的底层逻辑一次性讲透。

一、为什么需要分布式锁？

在单体应用中，我们用 synchronized、ReentrantLock 就能解决线程安全问题。但当服务部署了多个实例，多个 JVM 进程各自独立，本地锁就完全失效了——A 实例加的锁，B 实例根本感知不到，并发操作共享资源时依然会出现数据错乱。

分布式锁要解决的，就是跨进程、跨机器的资源互斥访问问题。一把合格的分布式锁，至少要满足三个核心特性：

  1. 互斥性：同一时刻只能有一个客户端持有锁
  2. 防死锁：客户端宕机后，锁最终能自动释放，不会永远占用
  3. 解铃还须系铃人：只能由加锁的客户端自己释放锁，不能被其他客户端误删

二、初代实现：从 SETNX 到手写分布式锁

很多人入门分布式锁，都是从 SETNX 命令开始的。我们一步步演进，看看手写锁会遇到哪些坑。

2.1 最原始版本：SETNX + DEL

最朴素的想法：用一个 key 代表锁，key 存在就代表锁被占用。

致命问题：如果业务执行到一半服务宕机了，锁永远不会被删除，形成死锁。

2.2 修复死锁：加上过期时间

既然怕服务宕机，那就给锁加个过期时间，到点自动释放。

新的问题：这是两条命令，不是原子操作。如果 SETNX 执行完、EXPIRE 还没执行时服务宕机，依然会死锁。

2.3 原子性修复：SET 复合命令

Redis 2.6.12 之后，SET 命令支持 NX 和 EX 参数组合，一条命令原子完成「加锁 + 设置过期时间」。

注意：原生 SETNX 命令不支持设置过期时间，已被 SET key value NX EX 组合替代，生产环境不要再用 SETNX。

这是目前手写分布式锁的标准写法，解决了原子性和死锁问题，但依然有坑。

2.4 误删问题：谁加的锁谁才能删

考虑这个场景：

  1. A 加锁成功，设置过期 30 秒
  2. A 业务执行太慢，30 秒还没跑完，锁自动过期释放了
  3. B 趁机抢到了锁
  4. 此时 A 业务执行完，调用 DEL 把 B 的锁给删掉了

锁被误删，互斥性直接被打破。

2.5 手写锁的三大遗留问题

到这里，一个基础可用的分布式锁就写完了，但它离生产可用还有很远，存在三个核心问题：

  1. 不可重入：同一个线程不能重复加同一把锁，嵌套调用场景会直接死锁
  2. 锁超时释放风险：业务执行时间超过过期时间，锁会被提前释放，导致并发问题
  3. 无等待重试机制：抢不到锁直接失败，没有排队、重试能力

这些问题，正是 Redisson 要解决的。

三、工业级实现：Redisson 分布式锁

Redisson 是 Java 生态最成熟的 Redis 客户端，它封装的分布式锁，解决了手写锁几乎所有的痛点，是生产环境的标准选型。

3.2 核心灵魂：看门狗自动续期机制

这是 Redisson 最经典的设计，专门解决「业务没跑完、锁却过期了」的痛点。

工作原理

当调用无参的 lock() 方法、不手动指定 leaseTime 时，看门狗自动启动：

  1. 初始设置：加锁成功后，默认设置锁过期时间为 30 秒（lockWatchdogTimeout 默认 30s，可全局配置）
  2. 定时续期：启动一个后台定时任务，每隔 10 秒（过期时间的 1/3）检查一次：如果当前线程还持有锁，就重新把过期时间续期到 30 秒
  3. 停止续期：业务执行完调用 unlock() 时，取消定时任务；如果服务宕机，定时任务随之消失，锁 30 秒后自动释放

正常情况下可以概括为：客户端存活并能按时续期，锁就继续有效；客户端退出后，锁最终过期。但长时间 Stop-The-World、进程调度暂停或网络分区都可能让续期超过超时窗口，锁仍可能提前失效，因此关键资源写入还要有版本号、唯一约束或 fencing token 兜底。

注意：如果手动指定了 leaseTime 过期时间（如 lock.lock(10, TimeUnit.SECONDS)），看门狗不会启动，锁到期自动释放。适合业务执行时长可控的场景。

3.4 锁等待与可中断

Redisson 还支持两个实用特性：

  • 锁等待（tryLock 带超时）：获取不到锁时，最多等待指定时间，超时返回 false，避免无限阻塞
  • 可中断锁（lockInterruptibly）：等待锁的过程中可以响应线程中断，适合需要取消的场景

3.5 更多锁类型

除了最常用的普通可重入锁，Redisson 还提供了丰富的锁实现，覆盖不同业务场景：

锁类型 说明 适用场景
公平锁 按请求顺序排队获取锁，避免线程饥饿 需要公平性的业务
读写锁 读共享、写互斥 读多写少场景，性能大幅提升
联锁（MultiLock） 同时加多把锁，全部成功才算加锁成功 跨多个资源的操作
红锁（RedLock） 多个独立 Redis 主节点上的多数派加锁方案 想降低单主故障风险、且能接受复杂度的场景

四、集群级可靠：RedLock 红锁算法

Redisson 单实例锁已经能满足很多业务场景，但它依然有一个根本性缺陷：锁状态依赖单个 Redis 主节点。

4.1 单节点锁的致命问题

如果 Redis 采用主从架构，会存在一个经典的竞态问题：

  1. 客户端 A 在主节点上加锁成功
  2. 主节点宕机，此时锁数据还没同步到从节点
  3. 从节点被提升为新主节点
  4. 客户端 B 在新主节点上也能加锁成功
  5. 两个客户端同时持有同一把锁，互斥性被打破

这就是单实例锁的天花板：主从切换时可能丢失锁，极端情况下出现双锁。

4.2 RedLock 设计思想

为了解决这个问题，Redis 作者 antirez 提出了 RedLock（红锁）算法。

它的核心思路是：部署 N 个完全独立的 Redis 主节点（互相之间不同步），只要超过半数节点加锁成功，就认为获取锁成功。

常见建议是部署 5 个独立节点，这样在多数派可用时仍能完成加锁。但它并不等价于强一致锁，仍然要面对时钟、网络和客户端停顿等问题。

4.3 完整加锁流程

  1. 记录开始时间：客户端记录当前时间戳 T1
  2. 尽快申请多节点锁：向所有 N 个 Redis 节点发起加锁请求，使用相同的锁 key、唯一随机值和过期时间。每个节点都设置很短的请求超时（几十毫秒），避免节点故障时长时间阻塞
  3. 统计成功数：统计成功加锁的节点数量。如果成功数 ≥ N/2 + 1（5 节点就是 ≥3），且总耗时小于锁的有效时间，就认为加锁成功
  4. 锁有效时间：实际有效时间 = 原始 TTL - 加锁总耗时
  5. 加锁失败处理：如果不满足成功条件，就向所有节点发起解锁请求，无论该节点有没有加锁成功

4.4 释放锁流程

向所有 N 个 Redis 节点发送解锁请求，通过 Lua 脚本校验 value 匹配后删除。即使某个节点加锁失败，也要尝试解锁，避免残留。

4.5 争议与局限性

RedLock 自提出以来就伴随着争议，最著名的是 Martin Kleppmann 提出的质疑，核心问题包括：

  • 时钟漂移问题：服务器时钟跳变可能导致锁提前过期
  • GC 停顿问题：客户端长时间 GC 停顿可能导致锁过期后才执行业务操作
  • 网络分区问题：极端网络分区下可能出现多个客户端同时持有锁
  • 实现复杂度高：需要部署多组独立 Redis 节点，运维成本高

Fencing Token 方案：Kleppmann 建议用 fencing token（单调递增的令牌号）来解决锁过期后的并发问题。每次获取锁时附带一个递增的 token，写入资源时校验 token 是否是最新，过期锁的旧 token 会被拒绝。这个方案不依赖时钟同步，但需要资源端配合实现校验逻辑。

工程实践中，很多“防重复执行、降低并发冲突”的业务用 Redis + Redisson 已经足够；但如果锁失效会造成资金、库存等不可接受的错误，不能只依赖 Redis 锁本身，应该在资源写入侧增加 fencing token、版本号、唯一约束或幂等校验。RedLock 可以降低单主故障风险，但不是强一致锁的银弹。

五、横向对比：Redis vs ZooKeeper 分布式锁

除了 Redis，ZooKeeper 也是分布式锁的经典实现方案。两者代表了两种不同的设计思路：性能优先 vs 可靠性优先。

5.1 ZooKeeper 锁实现原理

ZK 锁基于临时顺序节点实现：

  1. 客户端在锁目录下创建临时顺序节点（如 /lock/order_000001）
  2. 判断自己的节点是不是序号最小的，是则获取锁成功
  3. 不是则监听前一个节点的删除事件，阻塞等待
  4. 前一个节点被删除（锁释放），收到通知后再次判断自己是否最小
  5. 客户端断开连接时，临时节点自动删除，锁自动释放

这个机制天然支持公平锁，而且通过 Watcher 通知机制避免了轮询浪费。

5.2 全方位对比

对比维度 Redis 分布式锁 ZooKeeper 分布式锁
核心思想 基于 key 互斥，过期时间兜底 基于临时顺序节点，会话机制兜底
性能 极高，毫秒级响应 中等，十毫秒级响应
可靠性 主从异步切换可能丢锁，需资源端兜底 写入有一致性协议保障，但会话过期后旧客户端仍需 fencing 防止继续写
死锁风险 依赖过期时间兜底 会话过期自动释放，但暂停客户端恢复后的旧操作仍需防护
公平锁 默认非公平，需额外实现 天然支持公平锁
等待唤醒 取决于客户端；Redisson 使用 Pub/Sub 通知配合定时器 Watcher 通知前驱节点变化
可重入性 Redisson 原生支持 需自行实现
部署运维 简单，多数团队已有 Redis 复杂，需独立维护 ZK 集群

5.3 选型建议

  • 绝大多数业务场景选 Redis：性能好、实现简单、团队普遍已有 Redis 基础设施，Redisson 封装完善，开箱即用
  • 协调与选举场景优先评估 ZooKeeper/etcd：它们的一致性模型更适合协调，但锁服务本身仍不能阻止已过期客户端继续操作资源，应配合 fencing token 或资源端版本校验
  • 不要为了用而用：如果业务本身并发不高、数据一致性要求也没那么极致，数据库乐观锁、唯一索引可能更简单

六、最佳实践与避坑指南

1. 锁粒度要小

锁的范围越小越好，只锁住真正需要互斥的核心代码，不要把无关逻辑放进锁里，减少锁持有时间。

2. 一定要设置过期时间

手写 Redis 锁必须设置过期时间。使用 Redisson 无参 lock() 时会由看门狗维护默认过期时间；如果手动指定 leaseTime，就要确保业务能在该时间内完成，避免锁提前释放。

3. 释放锁必须放 finally 里

无论业务正常执行还是抛出异常，都要保证锁能被释放。这是最基础也最容易踩的坑。

4. 优先用 Redisson，不要手写锁

手写锁看似简单，实则坑点密布。Redisson 经过工业级验证，解决了可重入、续期、原子性等几乎所有问题，不要重复造轮子。

5. 不要过度追求绝对可靠

分布式系统里没有 100% 绝对可靠的锁。根据业务的容错能力选择合适的方案，大多数普通业务用 Redisson 锁已经足够。真正强一致的资源修改，必须在资源端再做版本校验、唯一约束或 fencing token 兜底。

七、面试高频问题

Q1：Redis 分布式锁怎么实现？

SET key value NX EX 30 原子加锁，value 用 UUID 做唯一标识，Lua 脚本校验 value 后删除。生产环境推荐 Redisson，支持可重入、看门狗续期、锁等待。

Q2：Redisson 看门狗机制原理？

调用无参 lock() 时自动启动后台定时任务，每隔锁超时时间的 1/3（默认 10 秒）续期到 30 秒。业务完成 unlock() 后停止；服务宕机后定时任务随之消失，锁自动过期。

Q3：为什么要用 Lua 脚本释放锁？

判断 value 和删除必须原子执行。如果不是原子的，判断完 value 匹配后、删除前锁可能已过期被其他客户端获取，就会误删别人的锁。

Q4：RedLock 解决了什么问题？有什么争议？

它试图降低单 Redis 主节点故障、主从切换导致锁丢失的风险。争议：时钟漂移、GC 停顿、网络分区仍可能导致安全性问题。Kleppmann 建议用 fencing token 方案在资源端兜底。

Q5：Redis 和 ZooKeeper 分布式锁怎么选？

Redis 性能高、实现简单，适合绝大多数普通业务场景。ZK/etcd 可靠性更高、天然适合协调类场景，适合对一致性要求极高的场景（分布式协调、选举）。大多数业务选 Redis + Redisson 即可，关键写操作再用幂等或版本校验兜底。

Q6：分布式锁的可重入怎么实现？

Redisson 用 Hash 结构存储：key 是锁名，field 是客户端 ID + 线程 ID，value 是重入次数。每次加锁 +1，解锁 -1，减到 0 删除锁。

写在最后

从最简单的 SETNX，到完善的 Redisson，再到多节点的 RedLock，Redis 分布式锁的演进过程，本质是不断在「性能」和「可靠性」之间做权衡。

手写锁简单但脆弱，适合理解原理；Redisson 平衡了功能与性能，是多数业务的生产首选；RedLock 试图降低单 Redis 主节点故障风险，但复杂度和争议也更高。没有绝对最优的方案，只有最适合业务的选择。

理解了这些底层原理，遇到锁冲突、死锁、误删等问题时，你就能快速定位根因，知道风险点在哪里、该怎么规避，而不只是会调用 API。
