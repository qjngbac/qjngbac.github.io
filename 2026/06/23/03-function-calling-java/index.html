Function Calling 实战：让大模型安全调用 Java 业务能力

发表于2026-06-23|更新于2026-07-19|AI工程
|浏览量:

前言

普通聊天模型只能根据输入生成文字。如果用户问“文档 42 的最新版本是谁创建的”，模型并不知道你的数据库内容；如果用户说“帮我把文档设为只读”，模型也没有权限直接修改业务数据。

Function Calling（也叫 Tool Calling）提供了一座桥梁：应用把可用工具的名称、描述和参数 Schema 告诉模型；模型在需要时返回“调用哪个工具、传什么参数”；真正执行函数的仍然是你的 Java 服务；执行结果再交给模型生成最终回答。

它不是让模型直接访问数据库，更不是绕过后端权限。正确的理解是：模型负责选择与生成参数，应用负责校验、授权、执行、审计和返回结果。

一、Function Calling 的完整流程

关键点：模型返回的是请求，不是执行结果。无论模型声称“已删除成功”还是“已退款”，只有后端真实执行并返回成功，业务动作才算发生。

六、写操作必须增加安全层

6.1 参数 Schema 不是权限校验

Schema 只能约束类型和结构，不能判断：

  • 当前用户是否能删除文档
  • 订单是否可退款
  • 金额是否超过审批阈值
  • 操作是否重复执行

所有规则必须在 Java 服务中再次验证。

6.2 高风险操作二次确认

推荐分成“计划”和“执行”两个工具：

确认 token 应由后端签发并短期有效，不能由模型自己编造。

确认 token 至少要绑定用户、操作类型、参数哈希、目标资源、过期时间和一次性随机数。执行时重新校验权限与资源状态，防止用户确认后参数被替换，或旧 token 被重放。

账本要保存参数哈希和最终结果。相同键、相同参数返回原结果；相同键、不同参数直接拒绝。数据库写入与账本状态最好放在同一事务，或通过事务消息等机制保证一致性。

  • 未分页的大列表
  • 大段二进制或 Base64
  • 内部异常堆栈

工具失败时返回结构化错误，而不是让模型猜：

工具结果同样是不可信输入：网页正文、文档内容和第三方错误消息都可能包含注入指令。返回给模型前应裁剪、脱敏并标记为数据；对用户只暴露稳定错误码和可理解信息，不要把异常堆栈或内部地址交给模型复述。

八、Function Calling 与 Agent 的关系

Function Calling 是底层机制：模型选择并请求工具。

Agent 是更完整的执行系统，通常还包含：

  • 多步循环
  • 状态和记忆
  • 工具选择
  • 任务规划
  • 错误恢复
  • 最大步骤
  • 人工确认
  • 追踪与评估

只有一次工具调用的“查订单机器人”不一定需要 Agent 框架；当流程存在多步决策、状态持久化和中断恢复时，才需要 LangGraph 等编排能力。

工具列表也可以按当前用户、租户和流程阶段动态过滤。让模型根本看不到无权使用的工具，比在它选择之后才拒绝更能降低误调用和提示注入的攻击面；后端权限校验仍然不能省略。

  • 高风险确认取消率

十、面试高频问题

Q1：Function Calling 是模型直接调用接口吗？

不是。模型返回工具调用请求，宿主应用校验后执行真实函数，再把结果交回模型。

Q2：JSON Schema 能防止越权吗？

不能。Schema 只约束参数结构，鉴权和业务权限必须由后端服务执行。

Q3：如何防止工具重复执行？

设置有限重试，写操作使用幂等键和状态机，保存工具调用 ID 与执行结果，对高风险动作增加确认。

Q4：多个工具能否并行？

互不依赖且不会产生冲突的只读工具可以并行；存在数据依赖或写入风险的工具应串行。

Q5：Function Calling 和 Agent 有什么区别？

Function Calling 是工具调用协议；Agent 在其上增加循环、状态、规划、记忆、错误恢复和人工介入等能力。

写在最后

Function Calling 最有价值的地方，不是“模型会调用接口”，而是让自然语言与结构化业务能力连接起来。但模型只能提出建议，真正的执行权必须始终掌握在后端系统中。

把工具设计得足够小、参数足够明确、权限足够严格，再配合幂等、审计和人工确认，才能从演示性质的工具调用走向可用的业务系统。
