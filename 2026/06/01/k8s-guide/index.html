K8s 核心概念：Pod/Service/Deployment

发表于2026-06-01|更新于2026-07-19|后端工程
|浏览量:

前言

Docker 解决了容器的构建与运行问题，但当业务从几个容器扩展到几十上百个容器时，单机管理就不够用了：实例异常了怎么自动恢复？流量上来了怎么横向扩容？多机集群怎么统一调度？怎样在满足条件时实现尽量不中断服务的滚动发布？

这些问题，正是 Kubernetes（简称 K8s）要解决的。作为容器编排领域的事实标准，K8s 提供了一整套容器调度、服务发现、滚动发布、自动扩缩容的能力，而支撑这一切的，是三个最核心的基础资源：Pod、Service、Deployment。

本文就从「为什么需要」出发，逐层拆解三大核心资源的设计思想、工作原理与典型用法，同时覆盖 Pod 生命周期、服务发现、滚动更新、配置管理等关键知识点，帮你建立起 K8s 的核心认知框架。

一、Pod：K8s 最小的调度单元

很多刚接触 K8s 的人会问：已经有容器了，为什么还要抽象出一个 Pod？

1.1 Pod 是什么

Pod 是 K8s 中最小的可部署计算单元，也是调度器直接调度的对象。它不是替代容器，而是封装一个或多个需要共同运行的容器。

  • 一个 Pod 可以包含 1 个主容器 + 多个辅助容器（Sidecar）
  • 同一个 Pod 内的容器共享网络命名空间，并可以挂载同一个声明过的 Volume 来共享文件
  • 它们可以直接通过 localhost 互相访问，共享同一个 IP 和端口空间

简单理解：Pod 就像一台「逻辑主机」，里面运行着紧密耦合的进程（容器）。它们共享网络上下文，但文件系统默认仍彼此隔离，只有通过共同挂载的 Volume 才共享文件。

最佳实践：绝大多数场景下，一个 Pod 只放一个主容器。只有当多个容器高度耦合，需要共同调度并共享网络或 Volume 时（比如代理 Sidecar），才放在同一个 Pod 里；同一 Pod 内的容器仍可独立重启。

1.2 Pod 的生命周期

Pod 从创建到销毁，有明确的状态流转：

阶段 含义
Pending Pod 已被集群接受，但一个或多个容器尚未准备完成，可能正在等待调度或拉取镜像
Running Pod 已绑定到节点，所有容器均已创建，至少一个容器仍在运行、启动或重启中；不代表应用已就绪
Succeeded 成功结束，所有容器正常执行完毕并退出
Failed 失败，所有容器已终止，且至少有一个异常退出
Unknown 未知，无法获取 Pod 状态，通常是节点失联

Init 容器：主容器启动前的准备

Pod 支持 Init 容器，常规 Init 容器会在应用容器启动之前按顺序执行，常用于：

  • 初始化配置、拉取配置文件
  • 等待依赖服务就绪（比如等数据库启动）
  • 做权限、环境初始化

常规 Init 容器必须依次成功完成，应用容器才会启动。Kubernetes 原生 Sidecar 是定义在 initContainers 中且容器级 restartPolicy: Always 的特殊形式，它启动后可以持续运行，不需要退出才进入后续启动步骤。

容器探针：健康检查

K8s 通过探针来判断容器的健康状态，这是实现自动恢复、流量摘除的核心：

探针 作用 失败后果
LivenessProbe 判断容器是否陷入无法自行恢复的故障 kubelet 终止该容器，再按 restartPolicy 决定是否重启容器
ReadinessProbe 判断容器是否准备好接收流量 在匹配 Service 的 EndpointSlice 中将该端点标记为未就绪
StartupProbe 判断应用是否已经完成启动 成功前暂停 LivenessProbe 和 ReadinessProbe；连续失败达到阈值后重启容器

探针支持四类检测方式：HTTP、TCP、gRPC 和在容器内执行命令（exec）。

生产环境通常应配置 ReadinessProbe，确保未就绪实例不接收流量。LivenessProbe 只应检查“重启能够恢复”的故障，错误的存活探针可能在高负载时引发反复重启；慢启动应用应配合 StartupProbe，避免过早触发存活检查。

1.3 资源请求与限制

给 Pod 中的容器设置 CPU、内存的 request 和 limit，是集群稳定性的重要基础：

  • requests：调度器据此判断节点是否有足够容量，也是 CPU 竞争时分配权重等机制的重要依据；它不等于业务性能保证
  • limits：CPU 超限通常会被节流；进程内存用量超过限制时可能触发 OOM 终止

不设置资源限制，一个容器可能吃光节点所有资源，导致同节点其他 Pod 被影响。

二、Service：稳定的服务访问入口

Pod 是临时的：宕机会重建、扩缩容会增减、滚动更新会替换。Pod 的 IP 一直在变，客户端该访问谁？

Service 就是解决这个问题的：它为一组 Pod 提供统一、稳定的访问入口，同时负责负载均衡。

2.2 四种 Service 类型

类型 作用 适用场景
ClusterIP 分配集群内部虚拟 IP，只能集群内访问 服务间内部调用，默认类型
NodePort 在每个节点开放固定端口，通过 节点IP:端口 访问 测试环境、临时对外暴露
LoadBalancer 通过支持它的云提供商或实现创建外部负载均衡入口，不保证一定是公网 IP 需要四层外部入口的场景
ExternalName 通过 DNS CNAME 返回外部域名，不创建代理或后端转发 用集群内名称引用外部域名

三、Deployment：应用的发布与自愈

直接创建的 Pod 仍可按 restartPolicy 由 kubelet 重启其中的容器，但 Pod 被删除或所在节点失效后，没有上层控制器替它创建新 Pod，也不便于扩容和滚动更新。Deployment 通过 ReplicaSet 管理无状态 Pod，负责副本数量、滚动更新、版本回滚和故障替换。

3.1 Deployment 的三层结构

我们日常只和 Deployment 打交道，ReplicaSet 由 Deployment 自动创建和管理。

3.2 自愈能力

你声明要 3 个副本，Deployment 和 ReplicaSet 控制器就会持续让实际状态向 3 个副本收敛：

  • Pod 被删除或进入终止状态 → ReplicaSet 创建一个具有新 UID 的替代 Pod
  • 节点失联 → 控制面在完成节点故障判定和驱逐处理后创建替代 Pod，由调度器选择合适节点
  • 手动删除受 Deployment 管理的 Pod → 控制器发现副本不足后创建新 Pod
  • LivenessProbe 失败 → kubelet 通常先在原 Pod 内重启容器；ReadinessProbe 失败只会停止向该 Pod 转发 Service 流量

这就是 K8s 的自愈能力：永远向声明的期望状态靠拢。

3.3 滚动更新：尽量避免服务中断

滚动更新会逐步替换旧版本，并不天然保证零停机。要避免请求中断，还需要足够副本和集群容量、正确的 ReadinessProbe、合适的 maxUnavailable/maxSurge，以及应用正确处理终止信号和在宽限期内完成请求。

  1. 创建新版本的 ReplicaSet
  2. 逐步增加新版本 Pod，逐步减少旧版本 Pod
  3. 新版本 ReadinessProbe 通过后才加入 Service
  4. 旧版本 Pod 进入终止流程后不再作为就绪端点接收常规流量，并在终止宽限期内退出
  5. 直到所有旧 Pod 被替换
  • maxUnavailable 越小，可用性越高，但更新越慢
  • maxSurge 越大，更新越快，但瞬时资源占用越多

四、配置管理：ConfigMap 与 Secret

应用配置不应该写死在镜像里，否则换环境就要重新打包。K8s 把配置和镜像解耦。

4.1 ConfigMap：普通配置

两种使用方式：

  • 环境变量注入：envFrom: [{configMapRef: {name: app-config}}]，Pod 启动后不会自动更新，通常需要重建 Pod
  • 挂载为配置文件：kubelet 会以最终一致的方式更新挂载内容，但应用需要自行重新读取；使用 subPath 挂载时不会收到自动更新

4.2 Secret：敏感配置

Secret 专门表达密码、Token、证书等敏感数据，并支持不同 Secret 类型。清单中的 stringData 会在 API 接收时转换为 data 中的 Base64 内容；Base64 只是传输编码，不构成安全保护。

默认情况下，Secret 可能以未加密形式存储在 etcd 中。生产集群应启用静态数据加密、通过 RBAC 实施最小权限、限制 Secret 只暴露给必要容器，并按需要对接外部密钥管理系统。示例值不能直接用于生产，也不应把真实密钥提交到 Git。

五、Namespace：资源组织边界

Namespace 用于在同一个集群内划分资源名称范围，便于按团队或环境实施配额和权限管理：

常见用法是按环境（dev/staging/prod）或团队组织资源。但 Namespace 本身不会提供网络隔离：在遵循标准网络模型且没有额外策略时，跨 Namespace 的 Pod 和 Service 默认可以通信。需要网络隔离时应配置 NetworkPolicy，并确认所用 CNI 插件支持其执行；还应结合 RBAC、ResourceQuota 等机制形成完整边界。

七、面试高频问题

Q1：Pod 和容器的关系？

Pod 是 K8s 最小的可部署计算单元，也是调度对象，可以包含一个或多个容器。同 Pod 内容器共享网络命名空间，通过 localhost 互访；只有挂载共同声明的 Volume 时才共享文件。大多数场景是一个主容器一个 Pod。

Q2：Service 的作用是什么？

为一组 Pod 提供稳定的访问入口和负载均衡。通过 Label Selector 关联 Pod，Pod IP 变化时自动更新后端列表。集群内通过 DNS 服务名访问。

Q3：Deployment 如何滚动更新，避免停机需要哪些条件？

Deployment 通过新旧 ReplicaSet 逐步替换 Pod，并用 maxSurge 和 maxUnavailable 控制过程。避免停机还依赖足够副本与容量、正确的 ReadinessProbe 和优雅终止；滚动更新本身不能无条件保证零停机。

Q4：LivenessProbe 和 ReadinessProbe 的区别？

LivenessProbe 检测容器是否陷入需要重启才能恢复的故障，失败后 kubelet 终止容器并按重启策略处理。ReadinessProbe 检测容器是否就绪，失败时相应 Service 的 EndpointSlice 会把该端点标记为未就绪，不再接收常规流量。

Q5：ConfigMap 和 Secret 的区别？

ConfigMap 用于非敏感配置，Secret 专门表达密码、Token 等敏感数据。Secret 的 Base64 表示不是加密，仍需启用 etcd 静态加密、配置最小权限 RBAC，并考虑外部密钥管理方案。

Q6：K8s 的自愈是怎么实现的？

Deployment 声明期望状态，ReplicaSet 控制器持续协调实际 Pod 数量。Pod 被删除或节点故障完成判定后，控制器创建替代 Pod，再由调度器选择可用节点；容器级故障则通常先由 kubelet 依据重启策略处理。

写在最后

Pod、Service、Deployment 是 K8s 的三块基石：

  • Pod 解决了「容器怎么分组、怎么调度」的问题
  • Service 解决了「Pod 动态变化怎么稳定访问」的问题
  • Deployment 解决了「应用怎么发布、怎么扩容、怎么自愈」的问题

K8s 的设计哲学是声明式 API：你只需要告诉系统期望的状态是什么，系统会自动向这个状态靠拢。不用关心具体怎么实现、在哪台机器跑，这就是容器编排的价值所在。

理解了这三个核心概念，再去看 StatefulSet（有状态服务）、DaemonSet（每节点一个 Pod）、Job/CronJob（批处理任务）等进阶资源，都会非常自然。
