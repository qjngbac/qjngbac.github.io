大模型 API 工程化实战：从 curl 到流式调用、重试与成本治理

发表于2026-06-18|更新于2026-07-19|AI工程
|浏览量:

前言

很多人第一次接入大模型，只需要几行代码：传入一段文字，等待模型返回答案。但一旦把它放进真实后端系统，问题会立刻变多：请求超时怎么办？限流后是否重试？流式响应中途断开怎么处理？如何统计 Token 和费用？怎样在多个模型之间切换？日志里又该记录什么，哪些内容不能记录？

因此，大模型 API 接入不能只停留在“能调用”。真正的工程目标是把不稳定、成本可变、响应时间较长的外部模型服务，封装成一个可超时、可重试、可观测、可降级、可替换的基础能力。

本文以 OpenAI Responses API 的通用调用方式为例，从 curl 开始，逐步搭建同步调用、流式输出、异常处理、统一网关与成本治理方案。示例中的模型名统一从环境变量读取，实际使用时应替换为账号当前可用的模型。

一、一次大模型请求到底包含什么

大模型 API 本质上仍然是 HTTP API。一次请求通常包含四部分：

部分 作用 常见内容
URL 指定服务和接口 /v1/responses
Header 身份认证与请求格式 Authorization、Content-Type
Body 模型、输入和生成参数 model、input、tools
Response 文本、工具调用、用量和状态 output、usage、status

API Key 应放在服务器环境变量或密钥管理系统中，不能写进前端、仓库或博客示例的真实配置里。

响应也不只是一个字符串。应用至少要区分 completed、failed、incomplete、cancelled 等状态，并检查 incomplete_details、错误对象和最终用量。SDK 的 output_text 是聚合文本的便利属性，不能替代对工具调用、拒绝、引用和其他输出项的解析。

三、流式输出：为什么通常使用 SSE

同步调用要等完整答案生成后再返回。长回答可能需要数秒甚至更久，用户会感觉页面“卡住”。流式输出会把生成中的增量事件逐段返回，前端可以边接收边展示。

Responses API 的 HTTP 流式模式基于 Server-Sent Events（SSE）。它是服务器到客户端的单向事件流，适合文字持续输出。需要双向、长连接和频繁交互时，再考虑 WebSocket。

3.1 Python 读取增量事件

流式接口不是把字符串随意切开，而是返回不同类型的语义事件。生产代码至少应区分：

  • 响应创建
  • 文本增量
  • 工具参数增量
  • 响应完成
  • 错误事件

3.2 后端转发 SSE 时的注意点

  1. 设置正确的 Content-Type: text/event-stream
  2. 及时刷新缓冲区，避免反向代理攒满后一次性返回
  3. 客户端断开时取消上游请求，避免继续产生费用
  4. 给整个流设置最大持续时间和最大输出限制
  5. 记录“首段响应时间”和“总完成时间”两个指标
  6. 对已经展示的部分内容考虑安全审核与中断策略

流式输出改善的是感知延迟，不一定减少模型总生成时间。首 Token 更快出现，不代表完整响应更快完成。

四、超时、重试与幂等性

4.1 哪些错误可以重试

场景 是否建议重试 说明
连接超时、临时网络错误 是 使用指数退避和随机抖动
429 限流 通常是 尊重服务端返回的等待信息
5xx 服务端错误 通常是 限制次数，防止雪崩
400 参数错误 否 修改请求内容
401/403 鉴权错误 否 检查 Key、权限和项目配置
内容安全拒绝 否 应走业务降级，而不是原样重试

4.2 生成请求是否幂等

大模型生成具有随机性，即使输入相同，再次调用也可能得到不同答案。因此，重试会带来两个风险：

  • 第一次请求其实已成功，但客户端没有收到结果，第二次又产生一次费用
  • 对外部工具的写操作被重复执行

模型纯文本生成可接受有限重试；涉及退款、发消息、修改数据等工具调用时，必须由业务系统提供幂等键、状态机和重复请求去重，不能依赖模型保证“只调用一次”。

4.3 并发控制、熔断与降级

模型服务通常同时受请求数和 Token 吞吐限制。工程上应把“并发数”“每分钟请求”“每分钟 Token”分开治理：

  • 收到 429 时优先读取服务端限流头或等待提示，而不是固定睡眠
  • 队列已满时快速失败，避免把超时请求继续送往上游
  • 熔断器只针对临时上游故障，不应掩盖参数错误和权限错误
  • 降级到其他模型前检查上下文长度、工具、结构化输出和多模态能力是否兼容
  • 流式请求断开后尽快取消上游任务，但仍要接受“取消与完成并发发生”的情况

限流、重试、熔断都应共享同一份总时间和调用次数预算。各层独立重试会产生重试放大。

六、Token、成本与缓存

大模型费用通常与输入和输出 Token 有关，但不同模型、缓存输入、工具和多模态内容的定价可能不同，应以调用时的官方价格页为准，不要把价格写死在代码里。

建议为每次请求记录：

Token 与费用应优先依据响应中的真实 usage 统计。流式场景通常要等最终完成事件才能获得完整用量；客户端中途断开时，仍要通过供应商账单或异步记录补齐成本。estimated_cost 应绑定一份有生效时间的价格快照，不能把价格常量永久写死。

成本控制常见手段：

  1. 限制输入文档长度和输出长度
  2. RAG 只传递真正相关的片段
  3. 简单任务使用较小模型
  4. 相同且允许复用的请求使用业务缓存
  5. 长固定前缀按供应商能力利用 Prompt Caching
  6. 对用户、租户和接口设置日配额
  7. 异步批处理不要求即时返回的任务

缓存键不能只用用户问题，还应包含模型、Prompt 版本、知识库版本、检索结果版本和参数，否则知识更新后可能返回旧答案。

缓存前还要回答三个问题：该答案是否允许跨用户复用、模型输出是否包含时效信息、请求是否产生工具副作用。写操作、个性化回答和权限相关 RAG 结果通常不应使用普通响应缓存。

七、日志、安全与可观测性

7.1 不要把完整 Prompt 全部写进普通日志

Prompt 可能包含：

  • 用户个人信息
  • 公司内部文档
  • API Token
  • 数据库结果
  • 系统提示词

生产环境应默认记录摘要、长度、哈希和结构化元数据。需要调试完整内容时，应使用受控环境、访问审计、脱敏和有限保留期。

7.2 核心监控指标

指标 作用
成功率 判断服务整体可用性
P50/P95/P99 延迟 发现长尾问题
首 Token 延迟 衡量流式体验
429/5xx 比例 判断限流与上游故障
Token/请求 发现输入膨胀
成本/用户或租户 做预算治理
降级比例 判断主模型是否稳定

还应让业务 request_id、供应商返回的请求 ID、客户端请求 ID 和 Trace ID 贯通，便于从一次用户请求追踪到检索、模型和工具执行。请求 ID 用于追踪，不自动提供业务幂等语义。

7.3 建立可回放但不泄密的失败样本

仅保存错误码通常不足以复现问题，完整保存 Prompt 又可能泄露数据。可以保存经过脱敏的结构化快照：Prompt 模板版本、各消息长度与哈希、检索 chunk_id、工具名、参数字段名、模型参数、错误类型和响应状态。需要查看原文时，通过受控调试存储按权限和保留期访问。

八、面试高频问题

Q1：为什么大模型调用要设置超时？

模型请求通常比普通数据库查询慢，且依赖外部网络。没有连接、读取和总业务超时，线程或连接可能长期占用，最终拖垮整个服务。

Q2：流式输出为什么常用 SSE，而不是 WebSocket？

文本生成主要是服务器持续向客户端推送，SSE 基于 HTTP、实现简单、天然支持事件流。需要双向低延迟交互时才更适合 WebSocket。

Q3：429 应该怎么处理？

识别限流错误，使用有限次数的指数退避与随机抖动，遵守服务端等待提示，并配合客户端并发控制和队列。不能无上限立即重试。

Q4：如何实现多模型切换？

通过内部 ModelClient 或网关接口屏蔽供应商差异，路由策略根据任务、成本、延迟和健康状态选择实现，同时记录真实调用模型和降级原因。

Q5：如何控制模型成本？

控制上下文和输出长度、优化 RAG、区分大小模型、缓存可复用结果、限制配额，并记录每次调用的 Token 和估算费用。

写在最后

调用大模型并不难，难的是把它变成可靠的后端能力。一个真正可上线的模型接入层，至少要具备超时、有限重试、流式中断、降级、成本统计、敏感信息保护和链路追踪。

先把 API 调用这一层打牢，后面学习 Prompt、Function Calling、Agent 和 RAG 时，才能把精力放在业务逻辑，而不是反复处理同一批网络与稳定性问题。
