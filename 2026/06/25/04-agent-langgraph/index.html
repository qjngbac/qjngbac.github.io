AI Agent 工程化实战：从工具调用到 LangGraph 状态工作流

发表于2026-06-25|更新于2026-07-19|AI工程
|浏览量:

前言

Function Calling 解决了“模型如何请求工具”，但一个完整任务往往不止一次调用：先判断是否需要检索，再查询知识库；检索结果不足时改写问题；涉及写操作时暂停等待用户确认；工具失败后决定重试、降级还是结束。

这时仅靠一个 while 循环会越来越难维护。AI Agent 的工程价值，就是把模型、工具、状态和执行循环组织起来。LangChain 提供较高层的 Agent 接口，LangGraph 则提供低层、状态化、可持久化的图工作流，适合需要明确节点、分支、恢复和人工介入的场景。

本文不追求“万能自主 Agent”，而是以一个后端知识助手为例，讲清 Workflow 与 Agent、ReAct、工具调用、状态管理、LangGraph 图结构、持久化和安全边界。

一、Workflow 和 Agent 的区别

维度 Workflow Agent
执行路径 代码预先定义 模型在运行时参与决定
可预测性 较高 较低
调试难度 较低 较高
适用场景 固定审批、ETL、确定性流程 开放问题、多工具选择
成本控制 容易 需要步数和预算限制

系统需要判断是否检索、检索什么、是否二次检索，再组织答案，适合有限 Agent 或带模型路由的工作流。

不要把所有业务流程都 Agent 化。路径明确时，普通代码和状态机通常更稳定、更便宜。

二、Agent 的基本组成

可以把 Agent 理解为：

  • Model：理解任务、选择工具或生成答案
  • Tools：检索、数据库查询、业务 API
  • State：消息、用户身份、检索结果、步骤计数
  • Loop：调用模型、执行工具、继续判断
  • Guardrails：权限、预算、步数、超时和人工确认

ReAct 的核心思想是让模型在推理和行动之间循环：观察当前信息，决定一个动作，读取动作结果，再决定下一步。

四、LangGraph 的核心：状态、节点和边

五、为什么状态持久化很重要

普通内存循环一旦进程崩溃，执行进度就丢失。LangGraph 的持久化层可以通过 checkpointer 保存图状态，使流程具备：

  • 对话短期记忆
  • 中断后恢复
  • Human-in-the-loop
  • 从历史检查点重放或分叉
  • 故障排查

持久化不是把所有对话永久存储。应明确保留期限、租户隔离、加密和删除机制。

开发环境可以使用内存 checkpointer，生产环境应换成数据库等持久化实现，并为每次调用提供稳定的 thread_id：

thread_id 是状态定位键，不等于访问凭证。读取或恢复检查点前仍要校验租户和用户权限，不能允许客户端枚举其他线程。

5.1 中断与人工确认

LangGraph 的 interrupt() 会保存当前状态并把待确认信息返回调用方，之后使用同一 thread_id 和 Command(resume=...) 恢复：

恢复时节点可能从头重放，检查点也不会回滚已经发生的外部写入。因此，中断前的副作用必须幂等，支付、删除等动作应放在确认后的独立节点，并配合执行账本。

六、短期记忆和长期记忆不要混为一谈

6.1 短期记忆

服务于当前会话或任务，例如：

  • 当前文档 ID
  • 最近几轮消息
  • 工具调用结果
  • 当前执行步骤

通常与 thread/session 绑定。

6.2 长期记忆

跨会话保存的用户偏好或业务事实，例如：

  • 用户偏好的回答语言
  • 项目技术栈
  • 经用户确认需要长期保存的设置

长期记忆应有明确写入条件，不能把模型推断自动当成事实。敏感信息需要更严格的授权和治理。

七、错误恢复与终止条件

Agent 必须有边界：

工具错误可分为：

类型 处理方式
参数格式错误 允许模型修正一次
临时网络错误 后端有限重试
权限拒绝 直接返回，不让模型绕过
业务状态冲突 返回明确错误并结束或询问用户
未知工具 记录异常并终止当前步骤

模型不能决定忽略权限错误。

八、可观测性：不要只保存最终答案

一次 Agent 请求至少要能看到：

推荐指标：

  • 平均步骤数
  • 工具调用成功率
  • 每种工具使用频率
  • 超过最大步骤比例
  • 人工确认比例
  • 单请求 Token 和成本
  • 最终任务成功率
  • 无意义循环率

Trace 中的 Prompt、工具参数和结果可能包含敏感数据，应做脱敏和访问控制。

除了线上 Trace，还应对节点和图做自动化测试：节点测试输入状态与返回增量，路由测试边界阈值，整图测试正常、检索不足、工具超时和人工拒绝等路径。模型节点可以用固定响应替身，避免单元测试依赖随机输出。

九、单 Agent 还是多 Agent

多 Agent 并不天然更高级。它会增加：

  • 模型调用次数
  • 上下文传递成本
  • 调试难度
  • 角色间误解
  • 循环和责任不清

只有在子任务确实需要不同工具、不同权限、独立上下文或并行处理时，才考虑：

工具数量少、流程简单时，一个 Agent 加明确 Workflow 更合适。

十、面试高频问题

Q1：Workflow 和 Agent 的区别？

Workflow 的路径由代码预先定义；Agent 会让模型在运行时参与工具和步骤选择。前者可预测，后者更灵活但更难控制。

Q2：LangChain 和 LangGraph 的定位？

LangChain 提供模型、工具和高层 Agent 等组件；LangGraph 提供低层状态图编排，适合持久化、分支、恢复和人工介入的复杂流程。

Q3：Agent 为什么要限制最大步数？

模型可能反复调用工具或重试。最大步数可以限制延迟、成本和失控风险。

Q4：Agent 记忆有哪些类型？

短期记忆服务当前线程和任务；长期记忆跨会话保存经确认的偏好或事实。二者的生命周期、权限和存储方式不同。

Q5：什么时候不应该使用 Agent？

流程固定、规则清晰、强一致性要求高时，普通代码、工作流或状态机更可靠。

写在最后

Agent 的重点不是让模型“自由发挥”，而是在有限工具、有限状态和明确边界中完成任务。高质量 Agent 往往同时包含确定性 Workflow 与少量模型决策，而不是把所有控制权都交给模型。

从单工具调用开始，逐步增加状态、条件路由、持久化和人工确认，比一开始就设计复杂多 Agent 系统更容易获得可靠结果。

对于后端知识库项目，当前“判断是否检索并调用一个知识库工具”的实现属于有限路由，不必为了名称升级而强行改成多 Agent。等到确实需要查询改写、检索质量判断、人工审核和可恢复写操作时，再把这些步骤显式迁移为状态图。
