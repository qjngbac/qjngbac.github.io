Docker 实战：从 Dockerfile 到容器编排

发表于2026-05-05|更新于2026-07-19|后端工程
|浏览量:

前言

在没有容器化的年代，部署服务是件非常折腾的事：开发环境能跑，上线就报错；机器换一台，就要从头装一遍依赖、配一遍环境；多服务部署还要挨个处理端口冲突、依赖顺序，效率极低。

Docker 的出现改变了这一局面：把应用及其运行时依赖构建为镜像后，可以在兼容的容器运行时上以更一致的方式交付。它提升了部署效率和进程隔离能力，但容器仍然共享宿主机内核，并不等同于虚拟机级别的隔离。但很多人用了很久 Docker，依然只会抄 Dockerfile、敲 run 命令，说不清楚镜像为什么能复用、数据怎么持久化、多服务怎么编排。

本文就从底层原理到实战落地，完整覆盖镜像分层机制、Dockerfile 最佳实践、数据卷与网络、Docker Compose 多容器编排，带你从零掌握容器化的核心能力。

一、镜像底层：分层存储的核心逻辑

Docker 镜像最精妙的设计就是分层存储，理解了分层，你就能看懂所有镜像优化技巧的本质。

1.1 什么是镜像分层

Docker 镜像由一组只读文件系统层和镜像配置共同组成。RUN、COPY、ADD 等会改变文件系统的构建步骤通常会形成新的文件系统层；CMD、EXPOSE 等主要修改镜像元数据，不能简单理解为“每条指令都生成一个文件层”。

举个例子，一个 Java 应用的镜像，从下到上大致是：

  1. 基础操作系统层：比如 Alpine、Debian，提供最基础的系统文件
  2. JDK 环境层：在系统层之上安装 JDK
  3. 依赖层：应用依赖的 Jar 包、配置文件
  4. 应用代码层：业务代码本身

所有层堆叠在一起，对外表现为一个完整的文件系统。容器启动时，会在所有只读层之上增加可写的容器层。除 Volume、绑定挂载和 tmpfs 等挂载点外，针对容器文件系统的运行时修改都记录在可写层，不会改动底层镜像。

1.2 联合文件系统

分层之所以能工作，依靠的是联合文件系统（UnionFS）。它可以把多个目录层叠加在一起，对外呈现为一个统一的目录视图。

  • 读取文件时，从上往下逐层查找，找到第一个匹配的文件就返回
  • 修改文件时，只会在最上层的可写层操作（Copy-on-Write 机制）
  • 删除下层文件时，在上层创建 whiteout（白化标记）将其遮蔽，底层文件本身依然存在

具体存储实现取决于 Docker 版本和安装方式。传统 Docker Engine 在 Linux 上广泛使用 overlay2；Docker Engine 29.0 及之后的全新安装默认使用 containerd 镜像存储和 overlayfs snapshotter。无论后端如何变化，镜像只读层、容器可写层和 Copy-on-Write 的基本模型仍然成立，可用 docker info 查看当前后端。

1.3 分层设计的三大优势

  1. 镜像复用，节省空间
    多个镜像可以共享相同的底层。比如 10 个 Java 应用，都共用同一个基础系统层和 JDK 层，磁盘上只需要存一份。

  2. 构建加速，复用缓存
    构建镜像时，未变化的层会直接复用本地缓存。只要 Dockerfile 指令顺序合理，构建速度可以非常快。

  3. 传输高效，增量更新
    推送/拉取镜像时，只需要传输变化的层，不需要传整个镜像。

不要只为减少层数而机械合并所有步骤。更重要的是控制最终镜像内容、及时清理包管理器缓存，并让变化较少的依赖步骤优先执行，以提高构建缓存命中率。

二、Dockerfile 实战与最佳实践

Dockerfile 是构建镜像的蓝图。不同指令会修改文件系统或镜像配置，写好 Dockerfile 会直接影响镜像体积、构建速度和安全性。

2.1 核心指令速览

指令 作用 注意事项
FROM 指定基础镜像并开始构建阶段 通常是第一条构建指令；全局 ARG 可以写在它之前
RUN 构建时执行命令 会提交文件系统变更；相关安装与清理应放在同一步
COPY 复制本地文件到镜像 纯文件复制优先用 COPY
ADD 添加本地/远程文件 本地 tar 归档可自动解包；普通复制优先用 COPY
WORKDIR 指定工作目录 后续指令的执行路径
EXPOSE 声明暴露端口 只是声明，不代表实际映射
ENV 设置环境变量 构建时和运行时都生效
CMD 容器启动默认命令 可被 docker run 参数覆盖，只有最后一个生效
ENTRYPOINT 容器启动入口命令 普通位置参数不会替换它，但可用 --entrypoint 覆盖
HEALTHCHECK 容器健康检查 Docker 据此标记健康状态，不天然等同于“业务已就绪”

三、数据持久化与网络通信

容器的可写层会在停止、再次启动同一个容器时保留，但删除容器后会随之删除，因此不应承载需要独立保存的数据。要让数据生命周期脱离容器，应使用数据卷或绑定挂载；容器间通信则依赖容器网络。

3.1 数据卷：三种持久化方案

方式 特点 适用场景
命名卷 Docker 管理，有自定义名称，可跨容器共享 数据库数据、希望由 Docker 管理路径的数据
绑定挂载 直接挂载主机路径，便于宿主机访问，但与主机路径和权限耦合 开发环境、配置文件注入、已有主机目录
tmpfs 挂载 存在内存中，容器停止就丢失 临时敏感数据、高速缓存

最佳实践：命名卷便于由 Docker 管理，绑定挂载便于直接控制宿主机路径，两者都可用于生产，选择取决于备份、权限、迁移和运维方案。数据库等有状态服务应把数据放在容器可写层之外，并建立独立备份；挂载本身不等于备份。

3.2 容器网络：四种核心模式

模式 特点 适用场景
bridge（默认） 容器有独立 IP，通过虚拟网桥通信 单机多容器互访
host 共享宿主机网络命名空间，无独立容器 IP，隔离性较弱 需要直接使用宿主机网络栈的场景
none 无网络 安全计算场景
自定义 bridge 网络 自己创建，支持 DNS 名称解析和更清晰的隔离边界 单机应用的多容器互访

自定义网络的核心优势：同网络内容器可以直接用容器名/服务名互相访问，不需要记 IP，支持 DNS 解析。

实践中应按通信需要划分网络，只把确实需要互访的服务接入同一网络，并通过名称访问。旧式 --link 已被用户自定义网络和内置 DNS 取代，不应继续使用。

四、Docker Compose：多容器编排实战

单容器用 docker run 还能应付，但如果应用依赖 MySQL、Redis、Nginx 好几个服务，挨个启动、配网络、管依赖，会非常繁琐。

Docker Compose 就是用来解决单机多容器编排的工具：用一个 YAML 文件定义所有服务，一条命令启动/停止整套环境。

4.2 关键配置说明

  • restart: unless-stopped：容器异常退出或 Docker 重启后自动拉起，手动停止后保持停止
  • depends_on + condition: service_healthy：创建应用容器前等待 MySQL 健康检查通过；它只管理 Compose 启动流程，应用仍应实现连接重试和故障恢复
  • 服务名互访：应用连接数据库写 mysql:3306，Compose 自动做 DNS 解析
  • healthcheck：定义健康检查，depends_on 的 condition 依赖它
  • 端口最小暴露：示例只把应用端口发布到宿主机，MySQL 和 Redis 仅在 Compose 网络内可达
  • 敏感值外置：密码从宿主机环境或 .env 注入，生产环境可进一步使用 Compose secrets；不要把真实密码提交到仓库
  • 初始化脚本时机：MySQL 官方镜像只会在数据目录首次初始化时执行 /docker-entrypoint-initdb.d 中的脚本，已有数据卷时不会重复执行

七、面试高频问题

Q1：Docker 镜像的分层原理是什么？

镜像由多层只读文件系统和镜像配置组成，文件系统层通过联合挂载呈现为统一视图。容器启动时在其上增加可写层，并通过 Copy-on-Write 处理修改。传统 Engine 常使用 overlay2，新版本也可能使用 containerd 的 overlayfs snapshotter；多镜像可以共享相同内容层。

Q2：COPY 和 ADD 的区别？

COPY 用于从构建上下文或其他阶段复制文件。ADD 还支持远程来源，并可自动解包本地 tar 归档。普通文件复制优先用 COPY，明确需要 ADD 的附加语义时再使用它。

Q3：CMD 和 ENTRYPOINT 的区别？

CMD 提供默认命令或参数，可被 docker run IMAGE ... 的参数替换。ENTRYPOINT 定义默认可执行入口，普通位置参数会追加给它，但仍可通过 --entrypoint 覆盖。常见用法是 ENTRYPOINT 写入口、CMD 写默认参数。

Q4：Docker 网络模式有哪些？

常见模式包括默认 bridge、host、none，以及用户自定义网络。用户自定义 bridge 网络支持名称解析，适合单机多容器按通信关系组网；host 模式共享宿主机网络命名空间，隔离性更弱。

Q5：Docker Compose 中 depends_on 的问题？

短语法只保证依赖按顺序启动；长语法配合 condition: service_healthy 可以等待依赖通过健康检查。但它不能替代应用在运行期间的连接重试、超时和故障恢复。

Q6：什么是多阶段构建？

把编译和运行分开到两个阶段。第一阶段编译打包，第二阶段只复制运行产物到精简镜像。最终镜像不含编译工具和中间文件，体积大幅减小。

写在最后

Docker 之所以能成为容器化的事实标准，核心在于它用非常简洁的设计，解决了「环境一致性」这个行业痛点。分层镜像、数据卷、网络、编排，每一块设计都有明确的工程目的。

理解了底层原理，再去写 Dockerfile、做编排，就不是照搬模板了，而是知道每一行配置为什么这么写、能带来什么收益。从单容器打包到多容器编排，一步步掌握，足以覆盖绝大多数中小团队的容器化需求。
