RAG 评估实战：Recall@K、MRR 与回答忠实度

发表于2026-07-12|更新于2026-07-19|AI工程
|浏览量:

前言

很多 RAG 项目的优化方式是：改一个参数，随便问几个问题，感觉回答比以前好，就宣布“准确率提升”。这种方法无法复现，也无法判断提升来自检索、Prompt、模型还是偶然性。

RAG 是一条组合链路，至少包含文档处理、检索、排序和生成。评估也应拆开：检索阶段判断正确文档有没有被找回来、排在什么位置；生成阶段判断回答是否忠于上下文、是否真正回答问题；端到端阶段再看用户任务是否完成、延迟和成本是否可接受。

本文从构建评估集开始，实现 Recall@K、Hit Rate、MRR，并说明 Faithfulness、Answer Relevance、无答案识别和线上反馈应该如何组合。

一、RAG 评估要拆成三层

只看最终答案有两个问题：

  • 模型可能靠自身知识答对，掩盖检索失败
  • 检索正确但生成失败时，不知道应该改哪一层

2.1 标注粒度

优先标注 chunk_id，因为系统实际检索的是 Chunk。如果只标文件名，一个大文件中任何片段被召回都可能被误判为正确。

2.2 数据集要覆盖什么

  • 精确术语和错误码
  • 同义表达
  • 多跳问题
  • 指代问题
  • 需要多个片段的问题
  • 相似但答案不同的问题
  • 知识库外问题
  • 权限不可见问题
  • 文档版本冲突

第一版可以手工标注 30～50 条，重点是质量和代表性，而不是数量越多越好。

数据集最好分为开发集、回归测试集和最终验收集。开发集用于分析错误和调参，测试集用于持续回归，验收集应尽量少被日常调试看到；否则不断针对同一批问题优化，会得到看似很高但无法泛化的分数。问题也要记录来源文档版本，避免索引更新后标签已经失效。

MRR 是所有问题 Reciprocal Rank 的平均值。

MRR 对首个正确结果的位置敏感，但忽略后续其他相关文档。多证据问题应结合 Recall 或 nDCG。

五、Precision@K 与上下文噪声

RAG 中 Precision 低意味着上下文夹杂大量无关片段，可能：

  • 浪费 Token

  • 增加模型误读风险
  • 让引用变得混乱

六、无答案问题单独评估

对于 answerable=false 的问题，不能使用普通 Recall。应观察：

  • 正确拒答率
  • 无答案误答率
  • 有答案误拒率
  • 检索阈值分布

更完整的做法是把“系统判定可回答”作为二分类，统计混淆矩阵：可回答问题被拒绝是假阴性，无答案问题被回答是假阳性。分别报告无答案识别的 Precision、Recall 和有答案保留率，避免只提高拒答率却让系统什么都不敢回答。

这类指标对于企业知识库尤其重要，因为“诚实说不知道”通常比编造一个听起来合理的答案更安全。

七、生成评估：Faithfulness 与 Answer Relevance

7.1 Faithfulness

Faithfulness 衡量回答中的事实性陈述是否能由检索上下文支持。它不判断世界知识中的绝对真伪，而是判断“回答是否忠于给定资料”。

这就是不忠实，即使句子表达非常自信。

7.2 Answer Relevance

Answer Relevance 判断回答是否围绕用户问题。回答可能完全基于资料，却答非所问或包含大量无关信息。

7.3 自动评估的边界

可以使用 Ragas 或 LLM-as-a-Judge 辅助评估，但要注意：

  • 评估模型也会犯错
  • 不同 Prompt 和模型分数不可直接混用
  • 需要抽样人工复核
  • 关键安全场景不能只依赖模型裁判
  • 评估成本也要记录

最佳实践是“规则指标 + LLM 指标 + 人工抽检”。

Ragas 的指标和调用接口会随版本演进，项目应锁定版本。还要区分概念：Faithfulness 检查陈述是否受给定上下文支持，不等于世界事实正确；Response Relevancy 检查是否切题，也不验证事实。对业务结论仍需参考答案、规则校验或人工评审。

八、端到端评估

端到端指标可以包括：

维度 示例
正确性 关键点覆盖率、人工评分
可追溯性 引用存在率、引用支持率
安全 越权内容泄露率、注入成功率
体验 首 Token 延迟、总延迟
稳定性 错误率、超时率
成本 单问题 Token、估算费用

对比实验应固定：

  • 测试集
  • 文档版本
  • Embedding 模型
  • 生成模型
  • Prompt 版本
  • Top-K
  • 随机性参数

否则无法归因。

生成模型即使在低温度下也可能波动。重要实验可重复运行，报告均值、失败率以及 bootstrap 置信区间；当两个方案差异落在波动范围内时，不应宣称有稳定提升。

十、线上评估与用户反馈

离线评估覆盖有限，线上还应收集：

  • 点赞/点踩
  • 用户是否继续追问同一问题
  • 是否点击引用
  • 是否复制答案
  • 是否人工纠正
  • 工具任务是否成功完成

用户点击“没帮助”不一定代表答案错误，可能是太长、太慢或没有引用。反馈要结合具体原因选项，不能只保存一个布尔值。

点击、复制和继续追问都只是代理信号：用户可能复制错误答案，也可能不点击却已解决问题。线上数据用于发现趋势和异常，不能直接替代带标签的正确性评估。

线上实验要设置保护：

  • 小流量灰度
  • 成本上限
  • 错误率告警
  • 可随时回滚
  • 不对高风险用户操作直接 A/B 测试

十一、面试高频问题

Q1：Recall@K 和 Precision@K 的区别？

Recall 关注全部相关文档中找回了多少；Precision 关注返回结果中有多少是相关的。

Q2：MRR 衡量什么？

衡量第一个相关结果出现得有多靠前，对首个正确答案的排名敏感。

Q3：为什么检索和生成要分开评估？

最终回答失败可能来自没检索到资料，也可能来自模型没有正确使用资料。分开评估才能定位问题。

Q4：Faithfulness 是事实正确率吗？

不是完全等同。它主要判断回答中的陈述是否能由给定上下文支持。

Q5：为什么要加入无答案问题？

为了测试系统能否在资料不足时拒答，防止总是生成看似合理但没有依据的答案。

写在最后

RAG 优化的第一步不是换模型，而是建立评估。Recall@K 告诉你有没有找回正确资料，MRR 告诉你正确资料排得是否靠前，Faithfulness 告诉你答案是否忠于上下文。

有了固定数据集、实验配置和失败样本，才能把“感觉更好”变成可复现的工程结论，也才能在面试和简历中给出真实、可信的优化数据。

前言基础 RAG 通常使用一个 Embedding 模型，把问题转成向量，再做 Top-K 相似度检索。这条链路简单，但很快会遇到问题：搜索 SETNX 时找不到包含精确命令的片段；用户说“它为什么会失效”，单独拿这句话无法检索；向量结果语义相近，却没有真正回答问题；正确片段排在第 8 名，最终只取了前 4 名。 检索优化的核心不是盲目增加模型，而是分阶段解决： 12345让查询更适合检索→ 多路召回扩大候选集→ 融合不同排序→ 对候选做精细重排→ 只把高质量上下文交给生成模型 本文依次介绍 Query Rewrite、BM25 + 向量混合检索、RRF 融合、Reranker 与降级策略，并说明为什么所有优化都必须通过固定评估集验证。 一、先定位检索失败属于哪一类 失败类型 示例 常见方案 查询缺少上下文 “它为什么会失效？” 对话指代消解、查询改写 精确词召回失败 SETNX、错误码、类名 BM25/关键词检索 同义表达不一致 “缓存雪崩”与“批量过期” 向量检索 正确文档排序靠后 Top-10 有，Top-4 无 RRF、Reranke...
