Ollama 本地部署大模型：模型选择、API 调用与 Spring Boot 集成

发表于2026-07-22|更新于2026-07-19|AI工程
|浏览量:

前言

云端模型 API 上手快、能力强，但有持续费用、网络依赖和数据出网限制。Ollama 提供了一套简单的本地模型管理与 API 服务，可以在 Windows、macOS 和 Linux 上拉取并运行支持的模型，再通过本地 HTTP 接口接入 Python、Java 或 AI 框架。

本地部署不等于“零成本”。模型需要磁盘、内存或显存，CPU 推理速度可能较慢，升级、并发和监控都需要自己承担。它更准确的价值是：降低按次 API 费用、提高数据本地性，并为开发测试提供可控模型服务。

本文介绍 Ollama 的安装验证、模型选择、CPU/GPU 运行、REST API、Embedding、Spring Boot 接入、上下文和内存优化，以及本地部署的适用边界。

二、没有独立 GPU 能不能运行

可以。Ollama 能在 CPU 上运行支持的量化模型，但速度通常明显慢于合适的 GPU。是否可用主要取决于：

  • 模型文件大小
  • 系统内存
  • 上下文长度
  • CPU 性能与内存带宽
  • 并发数

没有独立 GPU 时建议：

  1. 从较小模型开始
  2. 选择合理量化版本
  3. 将上下文控制在实际需要范围
  4. 单用户、低并发验证
  5. 不在同一机器同时加载多个大模型

能加载不代表体验良好。模型权重、KV Cache 和运行时还会占用额外内存，应为操作系统和应用保留空间。

2.1 GPU 的作用

支持的 GPU 可以加速模型推理。模型无法完全放入显存时，可能发生部分卸载或回退，速度取决于硬件和运行时支持。

五、流式输出

Ollama 原生流通常是逐行 JSON 事件，不应直接假设它与浏览器 SSE 格式完全相同。对前端暴露时，可以由 Spring Boot 转换为自己的 SSE 协议。

最后一个 done=true 事件还包含加载、Prompt 处理、生成耗时以及 prompt_eval_count、eval_count 等统计。性能分析应读取这些字段，区分模型加载慢、首轮 Prompt 计算慢和生成速度慢，而不是只记录整个 HTTP 请求耗时。

六、生成 Embedding

Embedding 模型和聊天模型不是同一种用途。用于 RAG 时，应选择专门的 Embedding 模型，并保证文档与查询使用同一模型。

注意：

  • 记录模型名和版本
  • 更换模型后重建向量索引
  • 检查向量维度
  • 用真实检索集评估 Recall@K
  • 不要拿聊天模型输出的隐藏状态随意当 Embedding

Ollama 也支持用 JSON Schema 约束结构化输出。Schema 解决格式问题，不保证事实正确；本地模型对复杂 Schema 的遵循能力差异较大，应用仍需校验、限制重试，并为不支持的模型准备普通 JSON 或文本降级。

八、OpenAI 兼容接口的使用边界

Ollama 提供部分 OpenAI API 兼容能力，便于某些现有客户端切换 base_url。但“兼容”不代表所有端点、参数、工具行为和事件格式完全一致。

当前兼容层包含 /v1/chat/completions 和 /v1/responses 等端点；其中 Responses 只支持无状态调用，不支持用 previous_response_id 或 conversation 延续服务端状态。OpenAI 客户端通常要求填写 API Key，但本地 Ollama 会忽略该值，它不是业务鉴权机制。

生产接入前应逐项验证：

  • Chat/Responses 支持范围
  • Tool Calling
  • Structured Output
  • 流式事件格式
  • Usage 字段
  • 错误码
  • 多模态能力

需要 Ollama 完整能力时，优先使用原生 REST API 或官方客户端；统一模型网关则应在自己的适配层消化差异。

九、上下文长度、内存与性能

上下文长度是模型在一次请求中能够处理的 Token 范围。增加上下文会扩大 KV Cache 和计算量，通常会增加内存占用、首 Token 延迟和总耗时。

Ollama 当前会根据可用显存选择默认上下文：低于 24 GiB 通常为 4K，24～48 GiB 为 32K，48 GiB 及以上为 256K；该默认值可能随版本变化，可通过配置或模型参数调整。模型声明的最大上下文不代表当前机器应直接使用最大值。

不要因为模型“支持长上下文”就始终设为最大值。RAG 更应该通过检索选择少量相关片段，而不是把全部知识库塞进 Prompt。

优化顺序：

  1. 选择与硬件匹配的模型规模
  2. 控制上下文长度
  3. 控制输出长度
  4. 避免同时加载多个模型
  5. 使用 keep_alive 平衡冷启动与内存占用
  6. 限制并发和请求队列
  7. 记录加载时间、首 Token 和 tokens/s

Ollama API 的 keep_alive 可控制请求后模型在内存中保留多久。保留更久能减少再次加载延迟，但会持续占用内存。

并发还受 OLLAMA_MAX_LOADED_MODELS、OLLAMA_NUM_PARALLEL 和 OLLAMA_MAX_QUEUE 等配置影响。粗略理解，单个模型的内存需求会随“并行请求数 × 上下文长度”增加；队列超过上限时服务会返回 503。应先在目标硬件上测单并发基线，再逐步增加并发，并同时观察吞吐、P95 延迟和内存，而不是只调大队列。

十、本地部署的安全问题

本地运行不等于自动安全：

  • 默认监听范围是否只限本机
  • 是否通过反向代理暴露公网
  • 是否配置鉴权和 TLS
  • 哪些用户可以拉取或删除模型
  • Prompt 和文档是否记录日志
  • 模型文件和许可证如何管理

不要把未鉴权的 Ollama 端口直接暴露到公网。需要远程访问时，应放在受控网络后，通过 API Gateway、VPN 或带鉴权的后端代理访问。

本地 API 本身不提供面向业务用户的认证授权。即使只监听内网，也要由网关或应用层限制调用者、模型列表、最大上下文、并发和费用等价资源；模型文件还应记录来源、版本哈希和许可证，升级后重新运行质量与性能测试。

十一、Ollama 适合和不适合的场景

适合

  • 本地学习和原型
  • 隐私敏感的内部文档测试
  • 无网络或弱网络环境
  • 小规模、低并发应用
  • 云模型的开发降级方案

不一定适合

  • 对高并发和稳定 SLA 有严格要求
  • 单机资源不足但要运行大模型
  • 需要顶级模型质量
  • 不具备模型运维和安全能力
  • 需要频繁使用超长上下文

此时应评估云 API、专用推理平台或托管服务。

十二、面试高频问题

Q1：没有 GPU 能运行 Ollama 吗？

可以使用 CPU 运行支持的模型，但速度较慢，需要选择较小或合理量化的模型，并控制上下文和并发。

Q2：参数量和量化分别影响什么？

参数量影响模型容量和资源需求；量化降低权重精度以减少内存和提高可运行性，但可能带来质量损失。

Q3：为什么上下文越长越占内存？

除模型权重外，推理还要保存与上下文相关的 KV Cache；上下文增加会带来额外内存和计算开销。

Q4：Ollama 原生 API 和 OpenAI 兼容 API 如何选？

需要完整 Ollama 能力时使用原生 API；迁移已有客户端时可评估兼容接口，但要验证实际支持范围。

Q5：本地部署就没有成本吗？

不是。仍有硬件、电力、磁盘、维护和性能成本，只是减少了按调用付费和数据出网依赖。

写在最后

Ollama 让本地模型的安装、管理和 API 调用变得简单，但它不是把云端模型“免费搬到本机”。硬件能力、模型规模、上下文、并发和安全决定了最终体验。

对个人学习来说，选择一个能在现有机器稳定运行的小模型，跑通 Chat、Embedding 和 Spring Boot 接入，比追求最大参数量更有价值。再通过固定测试集比较本地模型与云 API，才能做出合理选型。
