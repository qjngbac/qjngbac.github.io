向量数据库选型实战：Chroma、Milvus 与 pgvector 对比

发表于2026-07-13|更新于2026-07-19|AI工程
|浏览量:

前言

做 RAG 时经常会看到三类建议：个人项目用 Chroma，海量向量用 Milvus，已有 PostgreSQL 就用 pgvector。这个结论方向上没有问题，但真正选型不能只看一句话。

向量数据库不仅负责“存一个 float 数组”。它还要处理向量索引、Top-K 搜索、metadata 过滤、增删改、并发、备份恢复、监控和客户端接入。不同产品的定位不同：Chroma 更偏 AI 应用快速检索，Milvus 是专用、可扩展的向量数据库，pgvector 则把向量能力放进 PostgreSQL 生态。

本文从架构、索引、过滤、运维和适用场景比较三者，并给出一套公平的性能测试方法。

一、向量检索在做什么

Embedding 模型把文本映射为固定维度向量：

常见度量：

度量 含义 注意点
Cosine 比较方向相似度 常用于文本向量
Inner Product 点积 是否需归一化取决于模型
L2 欧氏距离 距离越小通常越近

Embedding 模型、维度、是否归一化和索引操作符必须匹配。更换模型后，旧向量通常不能直接与新向量混合查询。

二、精确搜索与近似搜索

2.1 精确搜索

逐个计算查询向量与所有向量的距离，结果最准确，但数据量大时成本高。

2.2 ANN 近似最近邻

HNSW、IVF 等索引用额外数据结构减少搜索范围，以较小召回损失换取速度。

索引不是免费午餐：

  • 需要构建时间
  • 占用内存和磁盘
  • 参数影响速度与 Recall
  • 插入和更新成本可能增加

测试向量库时不能只看延迟，还要同时报告 Recall@K。

三、Chroma：快速搭建 AI 检索应用

Chroma 提供集合、Embedding、文档、metadata、过滤和查询能力。当前产品形态要区分本地嵌入式、单节点服务和分布式/云部署，不能再把 Chroma 简化成只有本地原型数据库。

3.1 优势

  • Python 接入简单
  • 与 LangChain 等框架集成方便
  • 本地开发成本低
  • 文档和 metadata 管理直接
  • 当前文档也覆盖稠密、稀疏、全文和混合检索能力，具体支持取决于部署形态与版本

3.2 边界

  • 单机原型的简单不等于天然满足大规模生产要求
  • 部署、备份和高可用仍需设计
  • 不同版本的索引与服务能力应以当前文档为准
  • 大规模压测前不要根据小数据量推断上限

具体容量仍取决于向量维度、硬件、并发和过滤复杂度，不能只按条数判断。

四、Milvus：专用、可扩展的向量数据库

Milvus 面向高性能向量搜索，支持多种向量索引、标量过滤、稠密/稀疏向量和分布式架构。分布式部署采用计算与存储分离等设计，可独立扩展不同组件。

4.1 优势

  • 专用向量检索能力丰富
  • 多种 ANN 和量化索引
  • 适合高数据量和高并发
  • 支持水平扩展和复杂部署
  • 对稠密、稀疏和混合搜索有更完整能力

4.2 代价

  • 组件与运维复杂度更高
  • 资源占用通常高于嵌入式方案
  • 需要监控、备份、容量规划
  • 小数据量下难以体现分布式价值

Milvus 有三种主要部署形态，选型时要明确比较对象，而不是把它们当成同一种成本：

形态 特点 常见用途
Milvus Lite 嵌入 Python 进程、接入简单 本地开发、小规模验证
Standalone 单机服务部署 中等规模、希望独立服务化
Distributed 多组件水平扩展 大规模、高吞吐和高可用

适合：

这里的数量级不是硬门槛。最终仍应以压测和运维能力决定。

五、pgvector：把向量放进 PostgreSQL

pgvector 是 PostgreSQL 扩展，提供 vector 类型、距离操作符、精确搜索以及 HNSW、IVFFlat 等近似索引。

5.1 优势

  • 与业务关系数据在同一事务和权限体系中
  • SQL、JOIN、JSONB 过滤能力成熟
  • 复用 PostgreSQL 备份、监控和高可用体系
  • Java 团队容易接入

5.2 边界

  • 向量负载会与 OLTP 业务争夺资源
  • 大规模高并发向量搜索需要细致调优
  • 需要考虑表膨胀、索引构建和维护
  • 不能因为“已经有 PostgreSQL”就忽略容量隔离

可以将向量表放在独立实例或资源组中，避免影响核心交易库。

还要特别关注 ANN 与过滤的交互。pgvector 的近似索引通常先扫描索引候选，再应用 WHERE 过滤；租户条件选择性很高时，返回数量可能不足。新版本提供 iterative scan，可逐步扩大扫描范围，但仍应为真实租户分布测试 Recall，并为过滤列建立普通索引或使用分区、部分索引等策略。

七、选型决策树

场景 1：个人项目和课程 Demo

优先 Chroma：

  • 快速实现
  • 本地持久化
  • 框架集成简单

场景 2：已有 PostgreSQL，向量与业务权限强关联

优先评估 pgvector：

  • 用户、租户、文档权限可通过 SQL 过滤
  • 复用事务和备份
  • 减少新基础设施

场景 3：大规模专用向量检索平台

评估 Milvus：

  • 数据规模和并发较高
  • 需要多索引和水平扩展
  • 团队有独立运维能力

九、索引参数的基本理解

HNSW

通常有较好的速度-召回折中，但构建较慢、内存占用较高。m 影响每个节点的连接数，ef_construction 影响构建质量与时间，查询时的 ef_search 影响搜索范围。参数越积极，Recall 通常越高，延迟和资源消耗也会上升。

IVFFlat

先把向量聚类到多个列表，查询时只探测部分列表。需要选择列表数量（lists）和探测数量（probes），通常应在已有数据后构建并用真实数据调参。探测列表越多，Recall 往往越高；探测全部列表时会逐渐失去近似搜索的速度优势。

精确索引/全表搜索

小数据量、离线评估或需要最高准确性时仍然有价值，也可用作 ANN Recall 的基准答案。

metadata 与租户隔离

向量相似度不能替代权限过滤。多租户系统应把 tenant_id、可见范围和文档状态作为结构化字段，在检索阶段过滤，并验证过滤后的 Recall 与延迟。无论使用哪种产品，都要设计认证、传输加密、备份恢复、密钥管理和删除传播，不能因为它是“AI 基础设施”就降低数据库安全标准。

十、面试高频问题

Q1：Chroma、Milvus、pgvector 如何选？

原型和中小知识库可优先 Chroma；大规模专用向量检索评估 Milvus；已有 PostgreSQL 且需要关系查询和事务时评估 pgvector。

Q2：HNSW 和 IVFFlat 有什么区别？

HNSW 基于多层图，通常查询性能和召回折中较好，但构建与内存成本较高；IVFFlat 基于聚类列表，查询探测部分分区，参数和数据分布影响明显。

Q3：为什么 ANN 要评估 Recall？

近似索引用精度换速度，只看延迟可能得到很快但漏掉正确邻居的系统。

Q4：pgvector 最大优势是什么？

向量和业务数据处于 PostgreSQL 体系中，可以复用 SQL、事务、权限、JSONB、备份和监控。

Q5：更换 Embedding 模型要做什么？

检查维度和向量空间，通常创建新索引或集合并重新生成全部文档向量，不能直接混用旧向量。

写在最后

向量数据库选型不是寻找一个永远最快的产品，而是在数据规模、过滤需求、团队技术栈、可用性和运维成本之间取舍。

个人 RAG 项目先用 Chroma 完成检索与评估完全合理；当业务需要 SQL 权限和事务时考虑 pgvector；只有规模和吞吐真正达到瓶颈，且团队能承担运维复杂度时，再引入 Milvus 等专用平台。

后端知识库项目当前选择 Chroma 是合理的：数据规模有限，首要目标是把切分、引用和评估链路做正确。代码仍应通过自有检索接口隔离 Chroma 返回类型，并保存稳定 ID 与 metadata，这样未来迁移到 pgvector 或 Milvus 时无需重写问答业务层。
