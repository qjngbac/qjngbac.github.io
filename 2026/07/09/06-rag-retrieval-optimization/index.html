前言

基础 RAG 通常使用一个 Embedding 模型，把问题转成向量，再做 Top-K 相似度检索。这条链路简单，但很快会遇到问题：搜索 SETNX 时找不到包含精确命令的片段；用户说“它为什么会失效”，单独拿这句话无法检索；向量结果语义相近，却没有真正回答问题；正确片段排在第 8 名，最终只取了前 4 名。

检索优化的核心不是盲目增加模型，而是分阶段解决：

本文依次介绍 Query Rewrite、BM25 + 向量混合检索、RRF 融合、Reranker 与降级策略，并说明为什么所有优化都必须通过固定评估集验证。

一、先定位检索失败属于哪一类

失败类型 示例 常见方案
查询缺少上下文 “它为什么会失效？” 对话指代消解、查询改写
精确词召回失败 SETNX、错误码、类名 BM25/关键词检索
同义表达不一致 “缓存雪崩”与“批量过期” 向量检索
正确文档排序靠后 Top-10 有，Top-4 无 RRF、Reranker
文档本身没覆盖 知识库没有 Redlock 拒答或补充知识库
Chunk 设计错误 关键结论被拆开 调整切分与 metadata

先看错误样本，再选技术。若知识库根本没有答案，增加 Reranker 也不能凭空创造正确文档。

三、向量检索与 BM25 为什么互补

3.1 向量检索

优点：

  • 捕捉语义相似和同义表达
  • 对自然语言问题友好
  • 不要求文档出现完全相同词语

不足：

  • 精确错误码、方法名、缩写可能不稳定
  • 相似主题容易混入
  • 分数跨模型和索引不一定可直接比较

3.2 BM25

BM25 是经典稀疏检索方法，基于词频、逆文档频率和文档长度进行相关性排序。

七、Reranker：对候选做更精细的相关性判断

Embedding 检索通常分别编码 query 和 document，速度快，适合大范围召回。Cross-Encoder Reranker 会把 query + document 一起输入模型，直接预测相关性，通常更准确但更慢。

八、完整检索管线

真实项目还要加入：

  • 权限和 metadata 过滤
  • 超时与降级
  • 查询缓存
  • 召回结果日志
  • 每阶段耗时
  • 重写失败时使用原查询
  • Reranker 不可用时回退 RRF 结果

权限、租户和时间范围等过滤应尽量在各检索器内部尽早执行。需要注意，有些 ANN 索引会先取近邻再做过滤：过滤条件越严格，最终结果越可能不足 Top-K。此时需要提高候选量、使用支持过滤的索引策略，或在离线评估中按不同过滤选择性单独测量召回。

还应给每个阶段分配延迟预算，例如改写、两路召回、融合和重排分别设置超时。改写超时保留原查询，BM25 故障退回向量检索，Reranker 超时返回 RRF 排名；降级必须记录指标，否则长期故障会被“仍能回答”掩盖。

九、检索阈值与无答案判断

单个固定相似度阈值容易失效，因为模型、距离度量、文档类型变化都会影响分数分布。

更可靠的方法：

  1. 在标注集上观察相关与不相关结果的分数分布
  2. 分不同主题或查询类型设策略
  3. 结合 Top-1、Top-2 差距和 Reranker 分数
  4. 用一个支持性判定器判断上下文是否能回答
  5. 对无答案问题单独评估误答率

如果没有足够依据，系统应拒答，而不是因为“总能检索出 Top-K”就强行生成。

还要查看错误样本：

  • 精确词问题是否改善
  • 同义问题是否退化
  • Reranker 是否过度偏爱长文本
  • Query Rewrite 是否改变原意
  • 延迟增加是否值得

没有评估数据时，不要在简历里写“准确率显著提升”。

对正在建设的后端知识库项目，建议按可归因顺序落地：先固定问题集并测纯向量基线，再加入 BM25 与 RRF，确认精确词和 Recall@K 改善后，最后评估 Reranker 的增益。这样每一步都能说明解决了哪类错误，也避免一次引入四个组件后无法定位退化来源。

十一、面试高频问题

Q1：为什么要做混合检索？

向量检索擅长语义匹配，BM25 擅长精确关键词，两者互补，可以提高候选召回覆盖。

Q2：RRF 为什么适合融合？

它只依赖各路排名，不要求不同检索器的分数处于同一量纲，简单且容易扩展多路结果。

Q3：Retriever 和 Reranker 的区别？

Retriever 从大规模语料快速召回候选；Reranker 对较小候选集进行更精细的 query-document 相关性排序。

Q4：为什么先扩大候选再重排？

Reranker 只能调整候选顺序，无法找回召回阶段完全漏掉的正确文档。

Q5：Query Rewrite 有什么风险？

可能改变原意、加入不存在的事实或丢失关键词，因此应保留原查询、限制改写数量并通过测试集验证。

写在最后

RAG 检索优化不是把 Query Rewrite、BM25、RRF 和 Reranker 全部堆上去，而是用错误样本找出瓶颈，再用最小方案解决。

合理的工程顺序是：先建立基线和评估集，再加混合召回，确认 Recall 提升后加入重排序，最后比较质量、延迟和成本。只有能量化地说明“为什么更好”，优化才真正有价值。
