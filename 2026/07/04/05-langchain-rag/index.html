LangChain RAG 实战：从文档入库到可追溯问答

发表于2026-07-04|更新于2026-07-19|AI工程
|浏览量:

前言

大模型能够回答大量通用问题，但它不知道你的私有文档，也可能使用过时知识或生成没有依据的内容。RAG（Retrieval-Augmented Generation，检索增强生成）通过“先检索、再生成”把外部知识放进模型上下文，使回答能够参考知识库并附带来源。

RAG 不能彻底消除幻觉。检索可能漏掉正确文档，切分可能破坏语义，生成模型也可能误读上下文。因此，一个合格的 RAG 系统不仅要能回答，还要保存稳定文档标识、展示引用，并在资料不足时明确降级。

本文使用 LangChain、Chroma 和 Markdown 文档搭建一条可运行的基础链路：文档加载、标题感知切分、Embedding、向量入库、Top-K 检索、带引用生成和无资料拒答。

一、RAG 的完整链路

两条链路要分开设计：文档更新不应阻塞每次问答；在线请求也不应临时重新切分整个知识库。

基础问答通常采用确定的“两步 RAG”：先检索，再生成。只有当任务确实需要模型决定是否检索、调用多个数据源或反复补充证据时，才需要 Agentic RAG。两步链路更容易评估、缓存和控制延迟，也是项目初期更稳妥的基线。

四、Markdown 标题感知切分

固定字符切分简单，但可能把标题和正文、代码块和解释拆开。对于技术博客，可以先按 Markdown 标题拆分，再对过长章节进行二次长度切分。

RecursiveCharacterTextSplitter 是按分隔符和长度递归切分，并不等同于真正的语义切分。标题感知切分负责保留文档结构，长度切分负责控制模型和 Embedding 输入。

4.1 Chunk 大小没有通用答案

过小：

  • 上下文不完整
  • 召回片段碎片化
  • 引用难以阅读

过大：

  • 一个块包含多个主题
  • 相似度被无关内容稀释
  • 输入 Token 和重排序成本增加

应基于自己的问题集比较不同大小的 Recall@K 和回答质量，而不是永远使用 1000/200。

技术文档还要特别保护代码围栏、表格和配置块。通用字符切分器可能在代码中间截断，严谨做法是先用 Markdown 解析器识别块级结构，再按 Token 长度切分；对于“标题提供主题、正文提供细节”的内容，可以保存父章节并在命中子块后补回父上下文。

7.1 引用不能只靠模型生成

模型可能引用不存在的编号。应用层应：

  • 限制可用编号范围
  • 检查答案中的引用是否存在
  • 返回引用对应的 source、section、chunk_id
  • 点击引用时展示原文

还应验证引用覆盖率：答案中的关键事实是否至少有一个来源支持，而不仅是编号能否解析。对外展示时最好从可信元数据生成链接，模型只负责选择编号，不能让它自由生成 URL 或文件路径。

八、资料不足时如何拒答

不能仅根据 Top-1 分数武断判断。可以组合：

  • 检索结果是否为空
  • Top-K 是否包含关键术语
  • 多路检索结果是否一致
  • Reranker 分数
  • 由独立判定步骤检查上下文能否支持回答

更新顺序也影响一致性。若先删除旧块、再写新块，中途失败会出现知识空窗；若只先写新块，失败后又可能新旧并存。常见做法是写入带新 index_version 的暂存集合，校验数量和抽样检索后原子切换别名；小型项目也至少应先成功写入新块、更新清单，再按清单删除旧 ID，并让整个过程可重试。

对于后端知识库项目，第一阶段可以保留“Markdown + 长度切分 + Chroma + 向量 Top-K”的基线，但清单、稳定 ID、标题路径和 chunk 级引用应尽早加入。它们是后续做增量更新、混合检索和 Recall 评估的前提，不属于可有可无的优化。

十一、面试高频问题

Q1：RAG 能彻底解决幻觉吗？

不能。RAG 可以提供外部依据并降低无依据生成，但仍可能检索错误、漏召回、上下文冲突或生成误读。

Q2：为什么文档要切分？

Embedding 和模型上下文都有长度与成本限制；合理切分还能让每个向量更聚焦。但切得过小会丢失上下文，过大则主题混杂。

Q3：chunk overlap 的作用？

减少边界处语义被截断，但重叠过大会造成重复召回和存储浪费。

Q4：为什么要保存 metadata？

支持来源引用、权限过滤、版本管理、删除同步和问题排查。

Q5：如何实现增量索引？

通过文档 ID 和内容哈希识别新增、修改与删除，只处理变化文档，并同步删除旧 chunk。

写在最后

一个基础 RAG 系统并不复杂，真正拉开差距的是细节：文档能否正确更新和删除、权限能否在检索前过滤、引用能否定位到具体片段、资料不足时能否诚实拒答。

先把这条基础链路做正确，再引入 Query Rewrite、混合检索、RRF 和 Reranker，才能知道每项优化究竟解决了什么问题。
