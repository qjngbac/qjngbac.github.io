JVM 运行时内存区域：你真的理解堆和栈吗？

发表于2026-07-06|更新于2026-07-08|Java 深度
|浏览量:

前言

写了多年 Java 代码，new 了无数个对象，调用了数不清的方法，但被问到”JVM 内存里堆和栈有什么区别”时，很多人依然只能说出”堆存对象，栈存变量”。面试被追问方法区、程序计数器、对象创建流程时，更是容易卡壳。

JVM 运行时内存区域是 Java 工程师的基本功，它不仅是面试高频考点，更是我们排查内存溢出、理解 GC 机制、进行性能调优的底层基石。今天我们就系统梳理一遍 JVM 运行时数据区，把堆、栈、方法区、程序计数器讲透，再完整走一遍对象从加载到创建的全过程。

一、JVM 运行时数据区总览

Java 虚拟机在执行 Java 程序的过程中，会把它管理的内存划分为若干个不同的数据区域。这些区域各有用途、创建销毁时间各不相同：有的随着虚拟机进程启动而存在，有的则依赖用户线程的启动和结束而建立和销毁。

按照线程共享和线程私有两大维度，可以划分为以下区域：

  • 线程私有：随线程生而生，随线程灭而灭
    • 程序计数器
    • Java 虚拟机栈
    • 本地方法栈
  • 线程共享：所有线程共享，随虚拟机启动而创建
    • Java 堆
    • 方法区（JDK 8 后演变为元空间）
    • 运行时常量池（方法区的一部分）

下面我们逐个拆解。

二、线程私有区域：每条线程都有一份

2.1 程序计数器：字节码的”行号指示器”

程序计数器（Program Counter Register）是一块很小的内存空间，它可以看作是当前线程所执行的字节码的行号指示器。

核心作用

字节码解释器工作时，就是通过改变这个计数器的值来选取下一条需要执行的字节码指令。分支、循环、跳转、异常处理、线程恢复等基础功能都依赖这个计数器完成。

为什么要线程私有？

因为 JVM 的多线程是通过线程轮流切换、分配处理器执行时间的方式实现的。在任何一个确定的时刻，一个处理器内核都只会执行一条线程中的指令。为了线程切换后能恢复到正确的执行位置，每条线程都需要有一个独立的程序计数器，互不影响，独立存储。

一个特殊之处

程序计数器是 JVM 规范中唯一一个没有规定任何 OutOfMemoryError 情况的内存区域。它只存一个很小的线程执行位置标记，通常不会成为内存溢出的来源。

2.2 Java 虚拟机栈：方法调用的”执行现场”

Java 虚拟机栈（Java Virtual Machine Stack）就是我们常说的”栈”，它描述的是 Java 方法执行的内存模型：每个方法在执行的同时，都会创建一个栈帧（Stack Frame）。

栈帧里装了什么？

栈帧是方法运行期的基础数据结构，里面包含：

  • 局部变量表：存放方法参数和方法内部定义的局部变量，包括基本数据类型和对象引用
  • 操作数栈：方法执行过程中各种字节码指令写入和提取数据的”工作台”
  • 动态链接：将符号引用转化为直接引用，支持多态
  • 方法返回地址：方法退出时，回到上层调用者的位置

方法从调用到执行完成的过程，就对应着一个栈帧在虚拟机栈中从入栈到出栈的过程。

常见异常

  • StackOverflowError：线程请求的栈深度大于虚拟机所允许的深度（比如无限递归）
  • OutOfMemoryError：如果虚拟机栈可以动态扩展，扩展时无法申请到足够内存

2.3 本地方法栈：Native 方法的后花园

本地方法栈（Native Method Stack）与虚拟机栈作用非常相似，区别在于：

  • 虚拟机栈为执行 Java 方法（字节码）服务
  • 本地方法栈为执行 Native 方法服务

在 HotSpot 虚拟机中，直接把本地方法栈和虚拟机栈合二为一了。它同样会抛出 StackOverflowError 和 OutOfMemoryError。

三、线程共享区域：所有线程共用的内存

3.1 Java 堆：对象内存的”大本营”

Java 堆（Java Heap）是 JVM 所管理的内存中最大的一块，也是所有线程共享的内存区域，在虚拟机启动时创建。

核心作用

唯一目的：存放对象实例。几乎所有的对象实例和数组都在这里分配内存。它也是垃圾收集器（GC）管理的主要区域，因此很多时候也被称作”GC 堆”。

堆的内部划分

从垃圾回收的角度，现代垃圾收集器基本都采用分代收集算法，堆通常会被划分为：

  • 新生代：刚创建的对象在这里，又细分为 Eden 区、Survivor0 区、Survivor1 区
  • 老年代：存活了很久、熬过多次 GC 的对象会进入老年代

注意：这种划分只是垃圾收集器的共性设计，不是 JVM 规范的强制要求。

常见异常

当堆中没有内存完成实例分配，并且堆也无法再扩展时，会抛出 OutOfMemoryError: Java heap space。

3.2 方法区/元空间：类元数据的”档案室”

方法区（Method Area）也是线程共享的内存区域，用于存储已被虚拟机加载的类信息、常量、静态变量、即时编译器编译后的代码缓存等数据。

JDK 7 与 JDK 8 的重要变化

很多人都听过”永久代”和”元空间”，它们本质上是方法区在不同版本的实现：

  • JDK 6 及之前：HotSpot 使用”永久代”实现方法区，字符串常量池也在永久代中，受 -XX:MaxPermSize 限制，容易 OOM
  • JDK 7：仍然使用永久代，但字符串常量池已经移到堆中，永久代中保留的内容逐步减少
  • JDK 8 及之后：移除了永久代，改用”元空间（Metaspace）”实现，使用本地操作系统内存，默认只受物理内存限制
对比项 永久代（JDK 6/7） 元空间（JDK 8+）
内存位置 JVM 管理的永久代 本地内存（OS）
大小限制 -XX:MaxPermSize 默认无上限，受物理内存限制
OOM 风险 较高 较低
字符串常量池 JDK 6 在永久代，JDK 7 移到堆中 堆中

运行时常量池

运行时常量池是方法区的一部分。Class 文件中的常量池表（存放编译期生成的各种字面量和符号引用），会在类加载后存放到运行时常量池中。

我们常说的字符串常量池，从 JDK 7 开始已经从方法区移到了堆中，这点需要特别注意。

四、堆 vs 栈：最经典的面试题

讲完了各个区域，我们用一张表把最核心的”堆和栈的区别”梳理清楚：

维度 Java 堆 Java 虚拟机栈
所属范围 线程共享，全局唯一一份 线程私有，每个线程一份
存储内容 对象实例、数组 基本数据类型、对象引用、栈帧
空间大小 通常很大，默认占物理内存 1/4 通常很小，默认 1M 左右
分配方式 运行期动态分配，大小不固定 单个栈帧大小可在编译期确定，线程栈大小由 JVM 参数和实现决定
异常类型 OutOfMemoryError StackOverflowError、OutOfMemoryError
GC 行为 GC 主要回收区域，频繁回收 方法结束自动出栈，不涉及 GC
内存连续性 可连续也可不连续 栈结构天然连续

一句话总结：栈管运行，堆管存储。栈负责程序的执行逻辑，堆负责对象的存储。

五、一个对象的诞生：对象创建的完整过程

当我们写下 Object obj = new Object(); 这行代码时，JVM 背后做了哪些事情？对象创建并不是简单地分配一块内存，而是一套完整的流程。

第一步：类加载检查

虚拟机遇到一条 new 指令时，首先检查这个指令的参数能否在常量池中定位到一个类的符号引用，并且检查这个符号引用代表的类是否已被加载、解析和初始化过。如果没有，就先执行相应的类加载过程。

第二步：分配内存

类加载检查通过后，虚拟机将为新生对象分配内存。对象所需内存大小在类加载完成后便可完全确定。

分配方式主要有两种：

  1. 指针碰撞：堆内存规整时，用过的和空闲的各放一边，中间放一个指针作为分界点，分配时把指针向空闲方向挪动一段与对象大小相等的距离
  2. 空闲列表：堆内存不规整时，虚拟机维护一个列表，记录哪些内存块可用，分配时找一块足够大的空间划分给对象

分配方式由堆是否规整决定，而堆是否规整又由所采用的垃圾收集器是否带有空间压缩整理功能决定。

并发安全问题：对象创建在高并发下非线程安全。解决方案有两种：一是对分配内存空间的动作进行同步处理（CAS + 失败重试）；二是把内存分配按线程划分在不同空间中进行，即本地线程分配缓冲（TLAB）。

第三步：初始化零值

内存分配完成后，虚拟机需要将分配到的内存空间都初始化为零值（不包括对象头）。这一步保证了对象的实例字段在 Java 代码中可以不赋初始值就直接使用，程序能访问到这些字段的数据类型对应的零值。

第四步：设置对象头

初始化零值之后，虚拟机要对对象进行必要的设置，例如这个对象是哪个类的实例、如何才能找到类的元数据信息、对象的哈希码、对象的 GC 分代年龄等。这些信息存放在对象的对象头之中。

第五步：执行 <init> 方法

上面的工作都完成后，从虚拟机视角看，一个新对象已经产生了。但从 Java 程序视角看，对象创建才刚刚开始——<init> 方法还没执行，所有字段都还是零值。

执行 new 指令之后会接着执行 <init> 方法，把对象按照程序员的意愿进行初始化，这样一个真正可用的对象才算完全构造出来。

六、对象的内存布局与访问定位

对象在内存里长什么样？

在 HotSpot 虚拟机中，对象在堆内存中的存储布局可以划分为三部分：

  1. 对象头：包括 Mark Word（哈希码、GC 分代年龄、锁状态标志等）和类型指针（指向它的类元数据的指针）
  2. 实例数据：对象真正存储的有效信息，也就是我们在类里定义的各种字段内容
  3. 对齐填充：非必需，起占位符作用。HotSpot 要求对象起始地址必须是 8 字节的整数倍，不够就对齐填充

对象是怎么被访问的？

我们的栈上存的是对象引用，那通过这个引用怎么定位到堆中的具体对象？主流访问方式有两种：

  • 句柄访问：堆中划分一块内存作为句柄池，引用存句柄地址，句柄里存对象实例数据和类型数据的地址。好处是引用稳定，对象移动时只改句柄
  • 直接指针访问：引用直接存对象地址，速度更快。HotSpot 主要使用这种方式

七、常见误区澄清

1. “对象一定都在堆上分配”

不对。随着逃逸分析技术的成熟，如果一个对象不会逃逸出方法外，虚拟机可能会对它进行栈上分配优化，直接在栈上分配内存，方法结束自动销毁，不需要 GC 介入。

2. “方法区就是元空间”

不准确。方法区是 JVM 规范中的概念，元空间是 HotSpot 在 JDK 8 对方法区的具体实现。就像”汽车”是概念，”特斯拉”是具体车型一样。

3. “字符串常量池在方法区”

只在 JDK 6 及之前是对的。JDK 7 开始，字符串常量池已经移到了堆中。

八、实战：用 JVM 参数观察内存区域

写在最后

JVM 运行时内存区域看起来概念很多，但核心逻辑其实很清晰：线程私有区域服务于方法执行，线程共享区域负责存储对象和类信息。堆和栈是其中最核心的两块，也是我们日常开发和面试中最常接触的部分。

理解这些不是为了死记硬背面试题，而是当你遇到内存溢出、GC 频繁、性能瓶颈时，能快速定位问题出在哪块内存区域，知道该从哪个方向去调优。这才是底层知识真正的价值。
