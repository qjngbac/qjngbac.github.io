GC 调优实战：一次线上 OOM 的排查经历

发表于2026-07-07|更新于2026-07-08|Java 深度
|浏览量:

前言

凌晨两点，运维告警群突然弹出一连串红色预警：核心订单服务多个实例响应超时，K8s Pod 频繁自动重启，错误日志中反复出现 java.lang.OutOfMemoryError: Java heap space。

作为后端团队的核心开发，我紧急介入排查。整个过程从 GC 现象观察到堆内存定位，从收集器原理到参数调优落地，前后三小时彻底解决问题。

很多人觉得 GC 调优是”玄学”，靠背参数就能应付。但真实线上场景里，懂 GC 算法原理、会用原生工具定位问题，才是解决 OOM 的核心能力。本文就以这次实战为线索，系统梳理 GC 核心算法、主流收集器选型，以及 jstat、jmap 的实战用法。

一、故障初现：线上 OOM 的现场信息

1.1 故障现象

当时正值业务高峰期，监控平台显示：

  • 接口平均响应时间从 50ms 飙升至 800ms，大量请求超时熔断
  • Pod 每 10 分钟左右触发一次自动重启，服务完全无法稳定承载流量
  • JVM 监控显示老年代内存占用持续走高，Full GC 频率远超正常水平

1.2 初步日志定位

登录容器查看服务错误日志，核心报错非常明确：

这说明堆内存已不足以分配新对象，JVM 即便触发 Full GC 也无法回收出足够空间。

当时的第一判断：要么是内存泄漏，对象被意外持有无法回收；要么是堆内存配置不足，或是 GC 收集器选型不合理，回收效率跟不上对象分配速度。

二、先补理论：GC 算法与主流收集器对比

排查之前，必须先理清 GC 的底层逻辑。不同收集器有完全不同的回收行为，选错收集器，再怎么调参数都是事倍功半。

2.1 三大经典 GC 算法

所有垃圾收集器的核心逻辑，都基于这三种基础算法：

算法 核心思路 优点 缺点 适用场景
标记-清除（Mark-Sweep） 先标记所有存活对象，再统一回收未标记的内存 实现简单，无需移动对象 产生大量内存碎片 老年代、低回收频率场景
标记-复制（Mark-Copy） 将存活对象复制到另一块内存，清空当前块 无内存碎片，分配效率高 空间利用率只有 50% 新生代（对象存活率极低）
标记-整理（Mark-Compact） 标记后将存活对象向一端移动，清理边界外空间 无内存碎片，空间利用率高 移动对象耗时较长 老年代、对象存活率高的场景

2.2 主流垃圾收集器全盘点

JDK 发展至今出现了多代收集器，核心差异在于分代设计、线程模型、停顿时间控制：

收集器 新生代算法 老年代算法 线程模型 核心特点 适用场景
Serial + Serial Old 标记-复制 标记-整理 单线程 实现简单，STW 时间长 单核 CPU、小内存
ParNew + CMS 标记-复制 标记-清除 多线程 低延迟目标，并发回收 JDK 7/8 互联网服务
Parallel Scavenge + Parallel Old 标记-复制 标记-整理 多线程 吞吐量优先，自适应调优 后台计算、批量处理
G1 分区 + 复制 整堆混合回收 多线程 可预测停顿时间 JDK 9+ 默认，大堆应用
ZGC 并发标记-整理 全并发 多线程 亚毫秒级停顿 超大内存、超低延迟
Shenandoah 并发标记-压缩 全并发 多线程 低停顿，与堆大小无关 低延迟要求场景

2.3 G1 收集器深入理解

G1（Garbage-First）是目前最主流的收集器，理解它的核心机制对调优至关重要：

G1 将堆划分为大小相等的 Region，JVM 会根据堆大小选择 Region 大小，目标通常是维持约 2048 个 Region。每个 Region 可以是 Eden、Survivor、Old 或 Humongous。核心优势：

  • 可预测停顿：通过 -XX:MaxGCPauseMillis 设置目标停顿时间，G1 优先回收垃圾最多的 Region
  • 混合回收：不只回收新生代，还会选择性回收部分老年代 Region
  • 大对象处理：超过 Region 50% 的对象直接分配到 Humongous Region

三、实战排查：jstat + jmap 精准定位

JDK 自带了一整套原生诊断工具，无需额外安装，是线上排查的首选。我们按照「先看 GC 整体趋势 → 再看对象分布排行 → 最后堆快照深查」的思路逐步推进。

3.1 第一步：jstat 实时监控 GC 全貌

jstat 是 JVM 统计监测工具，能实时查看堆内存各区域占用率、GC 次数与耗时，是排查 GC 问题的第一入口。

字段 含义

YGC/YGCT Young GC 总次数与总耗时

我们的异常现象

连续观察 1 分钟，确认了三个核心异常：

  1. Young GC 极其频繁：每分钟触发 30 次以上，Eden 区被快速填满
  2. Full GC 高度密集：每 5 分钟左右一次，每次回收后老年代占用仅从 98% 降到 92%，回收效果极差
  3. 单次 Full GC 耗时过高：平均 230ms，峰值甚至达到 500ms，严重阻塞业务线程

jstat 常用补充参数：

  • jstat -gc <pid>：查看各区域内存容量的绝对值（KB）
  • jstat -gccause <pid>：查看最近一次 GC 的触发原因
  • jstat -gcold <pid>：单独查看老年代的 GC 详情

⚠️ 线上操作注意事项：

  1. jmap -dump:live 会强制触发一次 Full GC，高峰期可能导致服务卡顿
  2. 大堆内存导出的文件体积很大（可能几个 GB），提前确认磁盘空间
  3. 生产环境优先用 jmap -histo 快速排查，尽量避免直接 dump 全量快照
  4. 如果必须 dump，建议在低峰期操作，或先摘掉流量

3.3 第三步：MAT 深度定位泄漏点

将 heap.hprof 文件下载到本地，用 Eclipse MAT（Memory Analyzer Tool） 打开，重点分析两个维度：

  1. 支配树（Dominator Tree）：找到占用内存最大的对象，以及持有它的上层引用链
  2. 泄漏嫌疑报告（Leak Suspects）：MAT 自动分析并标记高概率内存泄漏点

最终定位到根因：

业务代码中用了一个静态 HashMap 做本地缓存，存储订单详情对象，但没有设置过期时间和容量上限。高峰期订单量暴涨，对象不断被放入 Map，永远无法被 GC 回收，最终占满老年代触发 OOM。

四、调优落地：代码 + JVM 双管齐下

找到原因后，我们从代码和 JVM 两个维度同时优化，彻底解决问题。

4.1 代码层面：修复内存泄漏

这是最根本的解决手段：

1. 替换为 Caffeine 缓存

2. 优化对象创建逻辑

减少订单详情 VO 中不必要的字段和大数组引用，避免缓存对象过重；如果真实场景中存在超过 Region 50% 的大对象，G1 会将其分配到 Humongous Region，回收效率通常更低。

3. 移除不必要的静态集合引用

确保方法执行结束后对象可被正常回收。

六、避坑总结：GC 调优的通用方法论

6.1 排查工具使用心得

工具 定位 使用场景
jstat 先锋 先看 GC 频率、耗时、内存变化趋势，快速判断是否异常
jmap -histo 手术刀 快速定位异常对象类型，线上首选
jmap -dump 深度检查 精确定位泄漏引用链，低峰期使用
MAT 分析师 离线分析堆快照，找出根因
Arthas 全能 阿里开源诊断工具，支持在线诊断，功能更丰富

6.2 GC 调优的核心原则

  1. 先优化代码，再调参数：大量 OOM 都是代码问题（内存泄漏、大对象滥用、无界集合），不要上来就堆内存、换收集器

  2. 没有万能参数：吞吐量优先选 Parallel，低延迟选 G1/ZGC，根据业务场景选型，不要盲目照搬网上的”最优配置”

  3. 调优要有量化目标：比如”Full GC 每天少于 5 次””平均停顿 <100ms”，凭感觉调优只会越调越乱

  4. 一次只改一个参数：同时改多个参数，你不知道哪个起了作用

6.3 常见 OOM 排查路径

OOM 类型 可能原因 排查方向
Java heap space 内存泄漏或堆太小 jmap -histo 查对象，MAT 分析引用链
GC overhead limit exceeded GC 耗时占比过高 基本是内存泄漏，同上
Metaspace 动态代理/类加载器滥用 检查是否有大量动态类生成
Direct buffer memory NIO 直接内存泄漏 检查 ByteBuffer 是否正确释放
Unable to create new native thread 线程数超限 检查线程池配置，OS 线程限制

写在最后

GC 调优从来不是背参数的玄学，而是「观察现象 → 定位原因 → 验证优化」的科学过程。jstat、jmap 这些原生工具看似朴素，却是线上排查最可靠的武器；而 GC 算法和收集器原理，则是我们读懂数据、做出正确决策的底层支撑。

记住一个原则：先治代码的病，再调 JVM 的药。大多数 OOM 的根因都在代码里，不在 JVM 参数里。
