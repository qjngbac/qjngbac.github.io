Git、Docker、Linux 面试高频问答：后端开发必会的三大基本功

前言

面试中，Git、Docker、Linux 这三个工具几乎每场都会被问到。很多人觉得”会用就行”，但面试官问的不是”你会不会敲命令”，而是”你理不理解背后的原理”。

本文整理了这三个方向的高频面试题，每个问题都给出简洁但有深度的回答。不是命令手册，而是帮你建立知识框架。

一、Git 篇

Q1：Git 的工作区域有哪些？

Git 本地有三个核心区域，配合一个远程仓库：

  • 工作区：你正在编辑的文件目录
  • 暂存区：git add 后文件的临时存储区，准备提交
  • 本地仓库：git commit 后文件存储的地方，完整历史记录
  • 远程仓库：GitHub/GitLab 等托管平台

Q2：git merge 和 git rebase 有什么区别？

对比项 merge rebase
提交历史 保留分支结构，产生合并提交 变成一条直线，历史更干净
是否修改历史 不修改 修改（重放 commit）
冲突处理 一次性解决 逐个 commit 解决
适用场景 多人协作、公共分支 个人分支整理提交历史
安全性 安全 不要在公共分支上 rebase

面试加分回答：merge 适合合并公共分支，保留完整协作历史；rebase 适合整理个人分支的提交记录，让 log 更清晰。永远不要对公共分支执行 rebase，因为会修改提交历史，导致其他人的代码冲突。

Q3：git stash 是什么？什么场景用？

git stash 可以把当前工作区的修改临时保存起来，让工作区恢复干净。

Q6：git cherry-pick 是什么？

从另一个分支单独提取某一个 commit 应用到当前分支。

Q7：git reset 的三种模式区别？

模式 HEAD 暂存区 工作区 用途
--soft 移动 不变 不变 撤销 commit，保留修改在暂存区
--mixed（默认） 移动 重置 不变 撤销 commit 和 add，保留修改在工作区
--hard 移动 重置 重置 彻底回退，丢弃所有修改

二、Docker 篇

Q1：镜像和容器的区别？

对比项 镜像（Image） 容器（Container）
本质 只读模板，类似类 运行实例，类似对象
状态 不可变 可读写
存储 分层存储，共享基础层 在镜像层上加一层可写层
类比 Java 的 Class Java 的 Object

Q2：Dockerfile 常用指令有哪些？

ENTRYPOINT vs CMD 区别：

对比项 ENTRYPOINT CMD
作用 容器启动时执行的主命令 默认参数，可被 docker run 参数覆盖
组合 通常配合 CMD 使用 可被覆盖
示例 ENTRYPOINT ["java"] CMD ["-jar", "app.jar"]

Q3：docker compose 是什么？怎么用？

docker compose 用于定义和运行多个容器的编排配置。旧版本常写作 docker-compose，新版本推荐使用 docker compose 子命令。

Q4：数据卷（Volume）是什么？为什么需要它？

容器删除后数据会丢失。数据卷用于持久化存储，数据保存在宿主机上，容器删除也不影响。

Q5：Docker 网络模式有哪些？

模式 说明 用途
bridge 默认模式，容器有独立 IP 大多数场景
host 容器共享宿主机网络 需要高性能网络
none 无网络 安全隔离场景
自定义 bridge 容器间可通过服务名访问 Compose 默认

Q6：如何优化 Docker 镜像大小？

优化技巧：

  • 使用多阶段构建，最终镜像只包含运行时文件
  • 合并 RUN 指令减少层数
  • 把不常变的层放前面（利用缓存）
  • 使用 -slim 或 -alpine 基础镜像
  • .dockerignore 排除不需要的文件
