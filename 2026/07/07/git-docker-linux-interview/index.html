Git、Docker、Linux 面试高频问答：后端开发必会的三大基本功

前言

面试中，Git、Docker、Linux 这三个工具几乎每场都会被问到。很多人觉得”会用就行”，但面试官问的不是”你会不会敲命令”，而是”你理不理解背后的原理”。

本文整理了这三个方向的高频面试题，每个问题都给出简洁但有深度的回答。不是命令手册，而是帮你建立知识框架。

一、Git 篇

Q1：Git 的工作区域有哪些？

Git 本地有三个核心区域，配合一个远程仓库：

  • 工作区：你正在编辑的文件目录
  • 暂存区：git add 后文件的临时存储区，准备提交
  • 本地仓库：git commit 后文件存储的地方，完整历史记录
  • 远程仓库：GitHub/GitLab 等托管平台

Q2：git merge 和 git rebase 有什么区别？

对比项 merge rebase
提交历史 保留已有提交；必要时产生合并提交 把提交重放到新基点，历史更线性
是否修改历史 不修改 修改（重放 commit）
冲突处理 通常在一次合并中解决 可能在逐个重放提交时多次解决
适用场景 多人协作、公共分支 个人分支整理提交历史
安全性 不改写已有提交 ID 避免擅自改写他人已基于其开发的共享历史

面试加分回答：merge 在可快进时默认可能只移动分支指针，并不一定生成 merge commit；需要显式保留合并节点可用 --no-ff。rebase 适合整理尚未共享的个人提交。已经推送且被他人基于其开发的历史，不应擅自改写；团队明确协调的维护操作则另当别论。

二、Docker 篇

Q1：镜像和容器的区别？

对比项 镜像（Image） 容器（Container）
本质 只读模板，类似类 运行实例，类似对象
状态 不可变 可读写
存储 分层存储，共享基础层 在镜像层上加一层可写层
类比 Java 的 Class Java 的 Object

Q2：Dockerfile 常用指令有哪些？

ENTRYPOINT vs CMD 区别：

对比项 ENTRYPOINT CMD
作用 容器启动时执行的主命令 默认参数，可被 docker run 参数覆盖
组合 通常配合 CMD 使用 可被覆盖
示例 ENTRYPOINT ["java"] CMD ["-jar", "app.jar"]

Q3：docker compose 是什么？怎么用？

docker compose 用于定义和运行多个容器的编排配置。旧版本常写作 docker-compose，新版本推荐使用 docker compose 子命令。

depends_on 主要控制容器的创建和启动顺序，不等同于 MySQL、Redis 已经可以接收请求。应用仍应具备连接重试；需要 Compose 等待健康状态时，还要为依赖服务配置 healthcheck，并使用支持的 condition: service_healthy 写法。

Q4：数据卷（Volume）是什么？为什么需要它？

容器可写层会随容器删除而丢失。数据卷用于持久化存储，由 Docker 在宿主机管理，删除容器通常不会删除命名卷；但显式执行 docker volume rm 或 docker compose down -v 仍会删除卷数据。

Q5：Docker 网络模式有哪些？

模式 说明 用途
bridge 默认模式，容器有独立 IP 大多数场景
host 容器共享宿主机网络 需要高性能网络
none 无网络 安全隔离场景
自定义 bridge 容器间可通过服务名访问 Compose 默认

Q6：如何优化 Docker 镜像大小？

优化技巧：

  • 使用多阶段构建，最终镜像只包含运行时文件
  • 把安装与清理放在同一个 RUN 中，避免已删除文件仍留在旧层；层数本身不是镜像大小的唯一决定因素
  • 把不常变的层放前面（利用缓存）
  • 选择满足运行依赖的精简基础镜像；使用 Alpine 前要验证 musl libc、字体、时区和原生库兼容性
  • .dockerignore 排除不需要的文件

Q3：Git 分支管理策略？

常见的分支模型：

分支 用途 从哪来 合并到哪
main 生产环境代码 - -
develop 开发主干 main main
feature/* 功能开发 develop develop
release/* 版本发布 develop main + develop
hotfix/* 紧急修复 main main + develop

Git Flow 适合有明确发布周期和多版本维护的团队，但不是唯一标准。持续交付团队常采用 trunk-based development：短生命周期分支尽快合入主干，用自动化测试和 feature flag 控制未完成能力。分支模型应匹配发布节奏，不能仅因为面试常见就默认采用 Git Flow。

写在最后

Git、Docker、Linux 不是”背命令”的东西，而是用出来的技能。

建议：

  • Git：在你的博客项目里练习分支操作、解决冲突
  • Docker：用 Docker 部署你的 Java 项目
  • Linux：买台云服务器，把项目部署上去

面试时，能说出”我在自己项目中实际用过”，比背 100 个命令更有说服力。
