Spring AI 实战：Java 应用接入大模型、RAG 与 Tool Calling

发表于2026-07-17|更新于2026-07-19|AI工程
|浏览量:

前言

Python 生态在大模型应用中非常丰富，但对于已经使用 Spring Boot 的团队，鉴权、配置、数据库、日志、监控和部署体系都在 Java 侧。为了接入模型而重新搭建一套独立 Python 服务，并不总是最合适的选择。

Spring AI 为 Java 应用提供统一的模型、Embedding、向量库、Tool Calling、Chat Memory、RAG Advisor 和可观测性抽象。它的价值不是“用 Java 重新实现模型”，而是让 AI 能力进入熟悉的 Spring 配置和依赖注入体系。

本文基于 Spring AI 当前的核心抽象，搭建同步/流式聊天、结构化输出、Tool Calling 和简单 RAG，并说明生产项目中的配置、安全和可观测性问题。

一、Spring AI 的核心抽象

抽象 作用
ChatModel 与聊天模型交互的底层统一接口

八、Chat Memory 的边界

大模型本身无状态。Chat Memory 通过应用保存历史消息，再在后续请求中重新提供。

不要把 Chat Memory 当作数据库：

  • 短期消息有长度上限
  • 需要租户和会话隔离
  • 敏感内容需要过期和删除
  • 重要业务状态应结构化保存
  • 工具结果不应无限回放

例如当前文档 ID、版本号等状态，更适合保存在业务会话对象，而不是靠模型从大量历史中猜。

使用 Memory Advisor 时，每次请求都应显式传入可信的会话标识，并在服务端校验其所属用户和租户。不要直接接受客户端任意 conversation_id，否则可能读取其他人的历史；消息保留窗口和持久化周期也要分别配置。

九、可观测性与生产配置

Spring AI 可基于 Spring 生态提供 ChatClient、ChatModel、EmbeddingModel、VectorStore 等组件的 metrics 和 tracing。

建议关注：

同时注意 Prompt 和响应可能含敏感信息。观测配置不应默认把完整正文暴露到普通监控平台。

Spring AI 默认不会为了观测而导出完整 Prompt、Completion 和工具参数等高敏感内容；若主动开启相关属性，要先完成脱敏、采样、访问控制和保留期设计。Trace ID 应贯穿 Controller、ChatClient、VectorStore 与工具调用，方便定位一次回答的完整链路。

生产配置还应包括：

  • API Key 外置
  • 超时与有限重试
  • 用户/租户限流
  • 输出长度限制
  • 工具最小权限
  • 请求追踪 ID
  • 费用预算
  • 模型和 Prompt 版本记录

十、Spring AI 与 LangChain4j 如何选

维度 Spring AI LangChain4j
Spring 集成 原生、紧密 良好，但框架独立性更强
核心风格 Spring 抽象、Advisor、Boot 自动配置 AI Services、模型/工具/RAG 抽象
适合团队 已深度使用 Spring 希望更独立的 Java LLM 框架
选型原则 看所需模型、向量库、版本稳定性和团队习惯

不要只根据“哪个更火”决定，先用同一业务场景做小型验证。

十一、面试高频问题

Q1：ChatModel 和 ChatClient 的区别？

ChatModel 是较底层的模型接口；ChatClient 提供更面向应用的同步和流式调用方式，并可组合 Prompt、Advisor、Memory 和 Tools。

Q2：Spring AI 中 Embedding 接口叫什么？

核心接口是 EmbeddingModel，负责把文本转换为向量。

Q3：Advisor 的作用？

在 ChatClient 请求/响应链路中插入可复用逻辑，例如 RAG 检索、Chat Memory 和日志增强。

Q4：Tool Calling 如何保证权限？

工具方法内部从可信登录上下文获取用户身份，并调用业务权限服务校验，不能相信模型传入的 userId。

Q5：Spring AI 和 LangChain4j 如何选？

Spring 项目希望使用 Boot 自动配置和 Advisor 体系时优先评估 Spring AI；需要更框架独立的 Java AI 抽象时评估 LangChain4j，最终以 PoC 和版本能力决定。

写在最后

Spring AI 的意义，是让模型、Embedding、向量库和工具能力进入 Spring 应用熟悉的配置、依赖注入与可观测体系。但框架不能替你解决权限、幂等、评估和数据治理。

最合适的落地方式不是一开始构建复杂 Agent，而是先用 ChatClient 完成稳定调用，再加入结构化输出、一个安全工具和一条可评估的 RAG 链路。

如果现有后端知识库项目以 Python、LangChain 和 Chroma 为主，不需要为了使用 Spring AI 生硬重写。更自然的演进是保留 Python 检索服务，Java 负责统一鉴权和业务接口；只有当团队确认 Java 侧能覆盖切分、评估和运维需求时，再逐步迁移具体组件。
